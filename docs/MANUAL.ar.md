## ما هو wall-vault؟

**wall-vault = وكيل (بروكسي) ذكاء اصطناعي + خزنة مفاتيح API لـ OpenClaw**

لاستخدام خدمات الذكاء الاصطناعي، تحتاج إلى **مفاتيح API**. مفتاح API يشبه **بطاقة دخول رقمية** تثبت أن "هذا الشخص لديه صلاحية استخدام هذه الخدمة". لكن بطاقات الدخول هذه لها عدد محدود من الاستخدامات يومياً، وقد تتعرض للكشف إذا لم تتم إدارتها بشكل صحيح.

يقوم wall-vault بتخزين بطاقات الدخول هذه في خزنة آمنة، ويعمل كـ **وكيل (بروكسي)** بين OpenClaw وخدمات الذكاء الاصطناعي. ببساطة، كل ما يحتاج OpenClaw فعله هو الاتصال بـ wall-vault، وسيتولى wall-vault بقية الأمور المعقدة.

المشكلات التي يحلها wall-vault:

- **التدوير التلقائي لمفاتيح API**: عندما يصل مفتاح إلى حد استخدامه أو يُحظر مؤقتاً (cooldown)، ينتقل بهدوء إلى المفتاح التالي. يستمر OpenClaw في العمل دون انقطاع.
- **التبديل التلقائي للخدمة (fallback)**: إذا لم تستجب Google، يتحول تلقائياً إلى OpenRouter؛ وإذا فشل ذلك أيضاً، يتحول إلى Ollama/LM Studio/vLLM (ذكاء اصطناعي محلي) المثبت على جهازك. لا تنقطع الجلسة. عندما تتعافى الخدمة الأصلية، يعود إليها تلقائياً من الطلب التالي (v0.1.18+، LM Studio/vLLM: v0.1.21+).
- **المزامنة الفورية (SSE)**: عند تغيير النموذج في لوحة تحكم الخزنة، ينعكس التغيير على شاشة OpenClaw خلال 1-3 ثوانٍ. SSE (Server-Sent Events) هي تقنية يدفع بها الخادم التحديثات فورياً إلى العميل.
- **الإشعارات الفورية**: أحداث مثل نفاد المفاتيح أو أعطال الخدمة تُعرض فوراً في أسفل واجهة TUI (شاشة الطرفية) الخاصة بـ OpenClaw.

> 💡 يمكن أيضاً توصيل **Claude Code وCursor وVS Code**، لكن الغرض الأساسي من wall-vault هو الاستخدام مع OpenClaw.

```
OpenClaw (شاشة TUI للطرفية)
        │
        ▼
  بروكسي wall-vault (:56244)   ← إدارة المفاتيح، التوجيه، الاحتياطي، الأحداث
        │
        ├─ Google Gemini API
        ├─ OpenRouter API (أكثر من 340 نموذجاً)
        ├─ Ollama / LM Studio / vLLM (جهازك المحلي، الملاذ الأخير)
        └─ OpenAI / Anthropic API
```

## التثبيت

### Linux / macOS

افتح الطرفية والصق الأوامر أدناه.

```bash
# Linux (كمبيوتر عادي، خادم — amd64)
curl -L https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-linux-amd64 \
  -o ~/.local/bin/wall-vault && chmod +x ~/.local/bin/wall-vault

# macOS Apple Silicon (M1/M2/M3 Mac)
curl -L https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-darwin-arm64 \
  -o /usr/local/bin/wall-vault && chmod +x /usr/local/bin/wall-vault
```

- `curl -L ...` — يقوم بتنزيل الملف من الإنترنت.
- `chmod +x` — يجعل الملف المُنزَّل "قابلاً للتنفيذ". تخطي هذه الخطوة سيؤدي إلى خطأ "رفض الإذن".

### Windows

افتح PowerShell (بصلاحيات المسؤول) ونفّذ الأوامر أدناه.

```powershell
# التنزيل
Invoke-WebRequest -Uri `
  "https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-windows-amd64.exe" `
  -OutFile "$env:LOCALAPPDATA\Programs\wall-vault\wall-vault.exe"

# إضافة إلى PATH (يُطبَّق بعد إعادة تشغيل PowerShell)
$env:PATH += ";$env:LOCALAPPDATA\Programs\wall-vault"
```

> 💡 **ما هو PATH؟** هو قائمة المجلدات التي يبحث فيها الكمبيوتر عن الأوامر. بإضافته إلى PATH، يمكنك تشغيل `wall-vault` من أي مجلد.

### البناء من المصدر (للمطورين)

ينطبق فقط إذا كانت بيئة تطوير لغة Go مثبتة لديك.

```bash
git clone https://github.com/sookmook/wall-vault
cd wall-vault
make build       # bin/wall-vault (الإصدار: v0.1.25.YYYYMMDD.HHmmss)
make install     # ~/.local/bin/wall-vault
```

> 💡 **إصدار بطابع زمني للبناء**: عند البناء بـ `make build`، يُنشأ الإصدار تلقائياً بصيغة مثل `v0.1.25.20260408.022325` تتضمن التاريخ والوقت. أما البناء المباشر بـ `go build ./...` فيعرض الإصدار كـ `"dev"` فقط.

## البداية الأولى

### تشغيل معالج الإعداد

بعد التثبيت، يجب تشغيل **معالج الإعداد** بالأمر أدناه في المرة الأولى. سيرشدك المعالج ويسألك عن العناصر المطلوبة واحداً تلو الآخر.

```bash
wall-vault setup
```

الخطوات التي يمر بها المعالج:

```
1. اختيار اللغة (10 لغات بما فيها الكورية)
2. اختيار السمة (light / dark / gold / cherry / ocean)
3. وضع التشغيل — الاستخدام الفردي (standalone) أو المشترك على عدة أجهزة (distributed)
4. اسم البوت — الاسم الذي سيظهر في لوحة التحكم
5. إعداد المنافذ — الافتراضي: بروكسي 56244، خزنة 56243 (اضغط Enter إذا لم تحتج للتغيير)
6. اختيار خدمات الذكاء الاصطناعي — Google / OpenRouter / Ollama / LM Studio / vLLM
7. إعداد مرشح أدوات الأمان
8. إعداد رمز المسؤول — كلمة مرور لقفل وظائف إدارة لوحة التحكم. يمكن إنشاؤه تلقائياً أيضاً
9. إعداد كلمة مرور تشفير مفاتيح API — لتخزين المفاتيح بأمان أكبر (اختياري)
10. مسار حفظ ملف الإعدادات
```

> ⚠️ **تذكَّر رمز المسؤول.** ستحتاجه لاحقاً لإضافة مفاتيح أو تغيير الإعدادات في لوحة التحكم. إذا فقدته، سيتعين عليك تعديل ملف الإعدادات يدوياً.

عند اكتمال المعالج، يُنشأ ملف الإعدادات `wall-vault.yaml` تلقائياً.

### التشغيل

```bash
wall-vault start
```

يبدأ خادمان في وقت واحد:

- **البروكسي** (`http://localhost:56244`) — الوسيط الذي يربط OpenClaw بخدمات الذكاء الاصطناعي
- **خزنة المفاتيح** (`http://localhost:56243`) — إدارة مفاتيح API ولوحة التحكم الإلكترونية

افتح `http://localhost:56243` في المتصفح لرؤية لوحة التحكم فوراً.

## تسجيل مفاتيح API

هناك أربع طرق لتسجيل مفاتيح API. **للمبتدئين، نوصي بالطريقة 1 (متغيرات البيئة)**.

### الطريقة 1: متغيرات البيئة (موصى بها — الأبسط)

متغيرات البيئة هي **قيم مُعدَّة مسبقاً** يقرأها البرنامج عند بدء التشغيل. اكتبها في الطرفية كما يلي.

```bash
# تسجيل مفتاح Google Gemini
export WV_KEY_GOOGLE=AIzaSy...

# تسجيل مفتاح OpenRouter
export WV_KEY_OPENROUTER=sk-or-v1-...

# التشغيل بعد التسجيل
wall-vault start
```

إذا كان لديك عدة مفاتيح، اربطها بفواصل(,). سيستخدمها wall-vault تلقائياً بالتناوب (round robin):

```bash
export WV_KEY_GOOGLE=AIzaSy...,AIzaSy...,AIzaSy...
```

> 💡 **نصيحة**: أمر `export` ينطبق فقط على جلسة الطرفية الحالية. لجعله يستمر بعد إعادة تشغيل الكمبيوتر، أضف الأسطر أعلاه إلى ملف `~/.bashrc` أو `~/.zshrc`.

### الطريقة 2: واجهة لوحة التحكم (بالنقر بالماوس)

1. ادخل إلى `http://localhost:56243` في المتصفح
2. انقر على زر `[+ إضافة]` في بطاقة **🔑 مفاتيح API** في الأعلى
3. أدخل نوع الخدمة وقيمة المفتاح والتسمية (اسم مرجعي) والحد اليومي، ثم احفظ

### الطريقة 3: واجهة REST API (للأتمتة/السكربتات)

REST API هي طريقة لتبادل البيانات بين البرامج عبر HTTP. مفيدة للتسجيل التلقائي عبر السكربتات.

```bash
curl -X POST http://localhost:56243/admin/keys \
  -H "Authorization: Bearer رمز_المسؤول" \
  -H "Content-Type: application/json" \
  -d '{
    "service": "google",
    "key": "AIzaSy...",
    "label": "المفتاح الرئيسي",
    "daily_limit": 1000
  }'
```

### الطريقة 4: علامة البروكسي (للاختبار السريع)

تُستخدم لاختبار مؤقت بإدخال مفتاح دون تسجيل رسمي. يختفي عند إيقاف البرنامج.

```bash
wall-vault proxy --key-google=AIzaSy... --key-openrouter=sk-or-...
```

## استخدام البروكسي

### الاستخدام مع OpenClaw (الغرض الرئيسي)

إليك كيفية إعداد OpenClaw للاتصال بخدمات الذكاء الاصطناعي عبر wall-vault.

افتح ملف `~/.openclaw/openclaw.json` وأضف المحتوى أدناه:

```json5
// ~/.openclaw/openclaw.json
{
  models: {
    providers: {
      "wall-vault": {
        baseUrl: "http://localhost:56244/v1",
        apiKey: "your-agent-token",   // رمز وكيل الخزنة
        api: "openai-completions",
        models: [
          { id: "wall-vault/gemini-2.5-flash" },
          { id: "wall-vault/gemini-2.5-pro" },
          { id: "wall-vault/hunter-alpha" },    // سياق 1M مجاني
          { id: "wall-vault/claude-opus-4-6" }
        ]
      }
    }
  }
}
```

> 💡 **طريقة أسهل**: انقر على زر **🦞 نسخ إعدادات OpenClaw** في بطاقة الوكيل في لوحة التحكم. سيُنسخ مقتطف يحتوي على الرمز والعنوان مملوءين مسبقاً إلى الحافظة. كل ما عليك هو اللصق.

**إلى أين يشير `wall-vault/` في بداية اسم النموذج؟**

يحدد wall-vault تلقائياً خدمة الذكاء الاصطناعي التي سيُرسل إليها الطلب بناءً على اسم النموذج:

| صيغة النموذج | الخدمة المتصلة |
|----------|--------------|
| `wall-vault/gemini-*` | اتصال مباشر بـ Google Gemini |
| `wall-vault/gpt-*`, `wall-vault/o3`, `wall-vault/o4*` | اتصال مباشر بـ OpenAI |
| `wall-vault/claude-*` | اتصال بـ Anthropic عبر OpenRouter |
| `wall-vault/hunter-alpha`, `wall-vault/healer-alpha` | OpenRouter (مليون توكن سياق مجاني) |
| `wall-vault/kimi-*`, `wall-vault/glm-*`, `wall-vault/deepseek-*` | اتصال بـ OpenRouter |
| `google/اسم_النموذج`, `openai/اسم_النموذج`, `anthropic/اسم_النموذج` إلخ. | اتصال مباشر بالخدمة المعنية |
| `custom/google/اسم_النموذج`, `custom/openai/اسم_النموذج` إلخ. | يزيل جزء `custom/` ويعيد التوجيه |
| `اسم_النموذج:cloud` | يزيل جزء `:cloud` ويتصل عبر OpenRouter |

> 💡 **ما هو السياق (context)؟** هو كمية المحادثة التي يمكن للذكاء الاصطناعي تذكرها دفعة واحدة. 1M (مليون توكن) يعني القدرة على معالجة محادثات طويلة جداً أو مستندات ضخمة دفعة واحدة.

### الاتصال المباشر بصيغة Gemini API (التوافق مع الأدوات الموجودة)

إذا كانت لديك أدوات تستخدم Google Gemini API مباشرة، فقط غيّر العنوان إلى wall-vault:

```bash
export ANTHROPIC_BASE_URL=http://localhost:56244/google
```

أو إذا كانت الأداة تسمح بتحديد URL مباشرة:

```
http://localhost:56244/google/v1beta/models/gemini-2.5-flash:generateContent
```

### الاستخدام مع OpenAI SDK (بايثون)

يمكنك أيضاً توصيل wall-vault في كود بايثون يستخدم الذكاء الاصطناعي. فقط غيّر `base_url`:

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:56244/v1",
    api_key="not-needed"  # مفاتيح API يديرها wall-vault
)

response = client.chat.completions.create(
    model="google/gemini-2.5-flash",   # صيغة provider/model
    messages=[{"role": "user", "content": "مرحباً"}]
)
```

### تغيير النموذج أثناء التشغيل

لتغيير نموذج الذكاء الاصطناعي أثناء تشغيل wall-vault:

```bash
# تغيير النموذج بالطلب مباشرة من البروكسي
curl -X PUT http://localhost:56244/api/config/model \
  -H "Content-Type: application/json" \
  -d '{"service": "openrouter", "model": "anthropic/claude-3.5-sonnet"}'

# في الوضع الموزع (متعدد البوتات)، غيّر من خادم الخزنة → ينعكس فوراً عبر SSE
curl -X PUT http://localhost:56243/admin/clients/معرف-البوت \
  -H "Authorization: Bearer رمز_المسؤول" \
  -H "Content-Type: application/json" \
  -d '{"default_service": "google", "default_model": "gemini-2.5-pro"}'
```

### التحقق من قائمة النماذج المتاحة

```bash
# عرض القائمة الكاملة
curl http://localhost:56244/api/models | python3 -m json.tool

# عرض نماذج Google فقط
curl "http://localhost:56244/api/models?service=google"

# البحث بالاسم (مثال: النماذج التي تحتوي على "claude")
curl "http://localhost:56244/api/models?q=claude"
```

**ملخص النماذج الرئيسية حسب الخدمة:**

| الخدمة | النماذج الرئيسية |
|--------|----------|
| Google | gemini-2.5-pro, gemini-2.5-flash, gemini-2.5-flash-8b, gemini-2.0-flash |
| OpenAI | gpt-4o, gpt-4o-mini, o3, o1, o1-mini |
| OpenRouter | أكثر من 346 (Hunter Alpha مع سياق 1M مجاني، DeepSeek R1/V3، Qwen 2.5 إلخ) |
| Ollama | اكتشاف تلقائي للخادم المحلي المثبت على الكمبيوتر |
| LM Studio | خادم محلي على الكمبيوتر (المنفذ 1234) |
| vLLM | خادم محلي على الكمبيوتر (المنفذ 8000) |

## لوحة تحكم خزنة المفاتيح

ادخل إلى `http://localhost:56243` في المتصفح لرؤية لوحة التحكم.

**تخطيط الشاشة:**
- **الشريط العلوي الثابت (topbar)**: الشعار، محددات اللغة/السمة، مؤشر حالة اتصال SSE
- **شبكة البطاقات**: بطاقات الوكلاء والخدمات ومفاتيح API مرتبة في شكل بلاطات

### بطاقة مفاتيح API

بطاقة لإدارة مفاتيح API المسجلة بنظرة واحدة.

- تعرض قائمة المفاتيح مصنفة حسب الخدمة.
- `today_usage`: عدد التوكنات (الأحرف التي قرأها وكتبها الذكاء الاصطناعي) المُعالجة بنجاح اليوم
- `today_attempts`: إجمالي عدد الاستدعاءات اليوم (تشمل النجاح + الفشل)
- سجّل مفاتيح جديدة بزر `[+ إضافة]` واحذفها بـ `✕`.

> 💡 **ما هو التوكن (token)؟** هو الوحدة التي يستخدمها الذكاء الاصطناعي لمعالجة النصوص. يعادل تقريباً كلمة واحدة بالإنجليزية، أو 1-2 حرف بالكورية. عادةً تُحسب رسوم API بناءً على عدد التوكنات.

### بطاقة الوكيل

بطاقة تعرض حالة البوتات (الوكلاء) المتصلة ببروكسي wall-vault.

**تُعرض حالة الاتصال في 4 مستويات:**

| المؤشر | الحالة | المعنى |
|------|------|------|
| 🟢 | قيد التشغيل | البروكسي يعمل بشكل طبيعي |
| 🟡 | متأخر | يستجيب لكن ببطء |
| 🔴 | غير متصل | البروكسي لا يستجيب |
| ⚫ | غير متصل/معطل | البروكسي لم يتصل بالخزنة أبداً أو معطل |

**دليل الأزرار في أسفل بطاقة الوكيل:**

عند تسجيل وكيل، حدد **نوع الوكيل** وستظهر أزرار الراحة المناسبة تلقائياً.

---

#### 🔘 زر نسخ الإعدادات — ينشئ إعدادات الاتصال تلقائياً

عند النقر على الزر، يُنسخ مقتطف إعدادات يحتوي على رمز الوكيل وعنوان البروكسي ومعلومات النموذج مملوءة مسبقاً إلى الحافظة. كل ما عليك هو لصق المحتوى المنسوخ في المكان المحدد في الجدول أدناه لإكمال إعداد الاتصال.

| الزر | نوع الوكيل | مكان اللصق |
|------|-------------|-------------|
| 🦞 نسخ إعدادات OpenClaw | `openclaw` | `~/.openclaw/openclaw.json` |
| 🦀 نسخ إعدادات NanoClaw | `nanoclaw` | `~/.openclaw/openclaw.json` |
| 🟠 نسخ إعدادات Claude Code | `claude-code` | `~/.claude/settings.json` |
| ⌨ نسخ إعدادات Cursor | `cursor` | Cursor → Settings → AI |
| 💻 نسخ إعدادات VSCode | `vscode` | `~/.continue/config.json` |

**مثال — لنوع Claude Code سيُنسخ هذا المحتوى:**

```json
// ~/.claude/settings.json
{
  "apiProvider": "openai",
  "baseUrl": "http://192.168.1.20:56244/v1",
  "apiKey": "رمز-هذا-الوكيل"
}
```

**مثال — لنوع VSCode (Continue):**

```yaml
# ~/.continue/config.yaml  ← الصق في config.yaml وليس config.json
name: My Config
version: 0.0.1
schema: v1

models:
  - name: wall-vault proxy
    provider: openai
    model: gemini-2.5-flash
    apiBase: http://192.168.1.20:56244/v1
    apiKey: رمز-هذا-الوكيل
    roles:
      - chat
      - edit
      - apply
```

> ⚠️ **أحدث إصدار من Continue يستخدم `config.yaml`.** إذا كان `config.yaml` موجوداً، يتم تجاهل `config.json` تماماً. تأكد من اللصق في `config.yaml`.

**مثال — لنوع Cursor:**

OPENAI_API_KEY=رمز-هذا-الوكيل

> ⚠️ **عندما لا تعمل النسخ إلى الحافظة**: قد تمنع سياسة أمان المتصفح النسخ. إذا ظهر مربع نص منبثق، حدد الكل بـ Ctrl+A وانسخ بـ Ctrl+C.

#### ⚡ زر التطبيق التلقائي — نقرة واحدة والإعداد جاهز

للوكلاء من نوع `cline` أو `claude-code` أو `openclaw` أو `nanoclaw`، يظهر زر **⚡ تطبيق الإعدادات** في بطاقة الوكيل. النقر على هذا الزر يحدّث ملف الإعدادات المحلي للوكيل تلقائياً.

| الزر | نوع الوكيل | الملف المستهدف |
|------|-------------|-------------|
| ⚡ تطبيق إعدادات Cline | `cline` | `~/.cline/data/globalState.json` + `secrets.json` |
| ⚡ تطبيق إعدادات Claude Code | `claude-code` | `~/.claude/settings.json` |
| ⚡ تطبيق إعدادات OpenClaw | `openclaw` | `~/.openclaw/openclaw.json` |
| ⚡ تطبيق إعدادات NanoClaw | `nanoclaw` | `~/.openclaw/openclaw.json` |

> ⚠️ هذا الزر يرسل طلباً إلى **localhost:56244** (البروكسي المحلي). يجب أن يكون البروكسي قيد التشغيل على تلك الآلة ليعمل.

#### 🔀 سحب وإفلات البطاقات لإعادة الترتيب (v0.1.17، محسّن v0.1.25)

يمكنك **سحب** بطاقات الوكلاء في لوحة التحكم لإعادة ترتيبها حسب رغبتك.

1. امسك منطقة **إشارة المرور (●)** في الزاوية العلوية اليسرى للبطاقة واسحب
2. أفلت فوق البطاقة في الموضع المرغوب وسيتغير الترتيب

> 💡 جسم البطاقة (حقول الإدخال، الأزرار، إلخ) لا يمكن سحبه. يمكن السحب فقط من منطقة إشارة المرور.

#### 🟠 اكتشاف عملية الوكيل (v0.1.25)

عندما يعمل البروكسي بشكل طبيعي لكن عملية الوكيل المحلي (NanoClaw، OpenClaw) توقفت، تتحول إشارة مرور البطاقة إلى **برتقالي (وامض)** وتظهر رسالة "توقف عملية الوكيل".

- 🟢 أخضر: البروكسي + الوكيل طبيعيان
- 🟠 برتقالي (وامض): البروكسي طبيعي، الوكيل متوقف
- 🔴 أحمر: البروكسي غير متصل
3. الترتيب المُعدَّل **يُحفظ فوراً على الخادم** ويبقى حتى بعد تحديث الصفحة

> 💡 الأجهزة اللمسية (الهاتف/التابلت) غير مدعومة بعد. استخدم متصفح سطح المكتب.

#### 🔄 مزامنة النماذج ثنائية الاتجاه (v0.1.16)

عند تغيير نموذج وكيل في لوحة تحكم الخزنة، يتم تحديث الإعدادات المحلية للوكيل تلقائياً.

**لـ Cline:**
- تغيير النموذج في الخزنة → حدث SSE → البروكسي يحدّث حقل النموذج في `globalState.json`
- الحقول المحدّثة: `actModeOpenAiModelId`, `planModeOpenAiModelId`, `openAiModelId`
- لا يتم تغيير `openAiBaseUrl` ومفتاح API
- **يلزم إعادة تحميل VS Code (`Ctrl+Alt+R` أو `Ctrl+Shift+P` → `Developer: Reload Window`)**
  - Cline لا يعيد قراءة ملف الإعدادات أثناء التشغيل

**لـ Claude Code:**
- تغيير النموذج في الخزنة → حدث SSE → البروكسي يحدّث حقل `model` في `settings.json`
- بحث تلقائي في مسارات WSL وWindows (`~/.claude/`, `/mnt/c/Users/*/.claude/`)

**الاتجاه المعاكس (الوكيل → الخزنة):**
- عندما يرسل وكيل (Cline, Claude Code, إلخ) طلباً إلى البروكسي، يضمّن البروكسي معلومات الخدمة/النموذج لذلك العميل في نبضة القلب
- تُعرض الخدمة/النموذج المستخدمة حالياً في الوقت الفعلي على بطاقة الوكيل في لوحة تحكم الخزنة

> 💡 **النقطة الأساسية**: يحدد البروكسي الوكيل برمز Authorization في الطلب ويوجّهه تلقائياً إلى الخدمة/النموذج المُعدَّة في الخزنة. حتى لو أرسل Cline أو Claude Code اسم نموذج مختلف، يتجاوزه البروكسي بإعدادات الخزنة.

### استخدام Cline في VS Code — دليل مفصل

#### الخطوة 1: تثبيت Cline

ثبّت **Cline** (المعرف: `saoudrizwan.claude-dev`) من سوق إضافات VS Code.

#### الخطوة 2: تسجيل الوكيل في الخزنة

1. افتح لوحة تحكم الخزنة (`http://عنوان_الخزنة:56243`)
2. في قسم **الوكلاء**، انقر **+ إضافة**
3. أدخل ما يلي:

| الحقل | القيمة | الوصف |
|------|----|------|
| المعرف | `my_cline` | معرف فريد (بالإنجليزية، بدون مسافات) |
| الاسم | `My Cline` | الاسم المعروض في لوحة التحكم |
| نوع الوكيل | `cline` | ← يجب اختيار `cline` |
| الخدمة | اختر الخدمة المطلوبة (مثال: `google`) | |
| النموذج | أدخل النموذج المطلوب (مثال: `gemini-2.5-flash`) | |

4. عند النقر على **حفظ** يُنشأ الرمز تلقائياً

#### الخطوة 3: الاتصال بـ Cline

**الطريقة أ — التطبيق التلقائي (موصى به)**

1. تأكد من تشغيل **بروكسي** wall-vault على تلك الآلة (`localhost:56244`)
2. انقر على زر **⚡ تطبيق إعدادات Cline** في بطاقة الوكيل في لوحة التحكم
3. إذا ظهر إشعار "تم تطبيق الإعدادات بنجاح!" فقد نجح
4. أعد تحميل VS Code (`Ctrl+Alt+R`)

**الطريقة ب — الإعداد اليدوي**

افتح الإعدادات (⚙️) في الشريط الجانبي لـ Cline:
- **API Provider**: `OpenAI Compatible`
- **Base URL**: `http://عنوان_البروكسي:56244/v1`
  - إذا كان على نفس الآلة: `http://localhost:56244/v1`
  - إذا كان على آلة أخرى (مثل خادم Mini): `http://192.168.1.20:56244/v1`
- **API Key**: الرمز الصادر من الخزنة (انسخه من بطاقة الوكيل)
- **Model ID**: النموذج المُعدَّ في الخزنة (مثال: `gemini-2.5-flash`)

#### الخطوة 4: التحقق

أرسل أي رسالة في محادثة Cline. إذا كان كل شيء طبيعياً:
- ستظهر **نقطة خضراء (● قيد التشغيل)** في بطاقة الوكيل المعني في لوحة تحكم الخزنة
- ستُعرض الخدمة/النموذج الحالية على البطاقة (مثال: `google / gemini-2.5-flash`)

#### تغيير النموذج

عند الرغبة في تغيير نموذج Cline، غيّره من **لوحة تحكم الخزنة**:

1. غيّر القائمة المنسدلة للخدمة/النموذج في بطاقة الوكيل
2. انقر **تطبيق**
3. أعد تحميل VS Code (`Ctrl+Alt+R`) — سيتم تحديث اسم النموذج في تذييل Cline
4. سيُستخدم النموذج الجديد من الطلب التالي

> 💡 فعلياً، يحدد البروكسي طلب Cline بالرمز ويوجّهه إلى النموذج المُعدَّ في الخزنة. حتى بدون إعادة تحميل VS Code، **النموذج المستخدم فعلياً يتغير فوراً** — إعادة التحميل فقط لتحديث عرض النموذج في واجهة Cline.

#### اكتشاف انقطاع الاتصال

عند إغلاق VS Code، تتحول بطاقة الوكيل في لوحة تحكم الخزنة إلى أصفر (متأخر) بعد حوالي **90 ثانية** وإلى أحمر (غير متصل) بعد **3 دقائق**. (ابتداءً من v0.1.18، أصبح اكتشاف عدم الاتصال أسرع مع فحص الحالة كل 15 ثانية.)

#### حل المشكلات

| العرض | السبب | الحل |
|------|------|------|
| خطأ "فشل الاتصال" في Cline | البروكسي غير قيد التشغيل أو العنوان خاطئ | تحقق من البروكسي بـ `curl http://localhost:56244/health` |
| النقطة الخضراء لا تظهر في الخزنة | لم يتم تعيين مفتاح API (الرمز) | انقر على زر **⚡ تطبيق إعدادات Cline** مرة أخرى |
| اسم النموذج في تذييل Cline لا يتغير | Cline يخزن الإعدادات مؤقتاً | أعد تحميل VS Code (`Ctrl+Alt+R`) |
| اسم نموذج خاطئ يُعرض | خطأ قديم (تم إصلاحه في v0.1.16) | حدّث البروكسي إلى v0.1.16 أو أحدث |

#### 🟣 زر نسخ أمر النشر — يُستخدم عند التثبيت على آلات جديدة

يُستخدم عند تثبيت بروكسي wall-vault لأول مرة على كمبيوتر جديد وتوصيله بالخزنة. عند النقر على الزر، يُنسخ سكربت التثبيت الكامل. الصقه في طرفية الكمبيوتر الجديد ونفّذه ليتم معالجة ما يلي دفعة واحدة:

1. تثبيت ملف wall-vault التنفيذي (يُتخطى إذا كان مثبتاً مسبقاً)
2. تسجيل تلقائي لخدمة systemd للمستخدم
3. بدء الخدمة والاتصال التلقائي بالخزنة

> 💡 السكربت يحتوي مسبقاً على رمز هذا الوكيل وعنوان خادم الخزنة، لذا يمكن تنفيذه فوراً بعد اللصق دون تعديلات.

### بطاقة الخدمات

بطاقة لتفعيل وتعطيل أو إعداد خدمات الذكاء الاصطناعي المراد استخدامها.

- مفتاح تبديل لتفعيل/تعطيل كل خدمة
- أدخل عنوان خادم الذكاء الاصطناعي المحلي (Ollama, LM Studio, vLLM إلخ الذي يعمل على جهازك) ليتم اكتشاف النماذج المتاحة تلقائياً.
- **مؤشر حالة اتصال الخدمة المحلية**: النقطة ● بجانب اسم الخدمة تكون **خضراء** إذا متصلة و**رمادية** إذا غير متصلة
- **إشارة مرور تلقائية للخدمات المحلية** (v0.1.23+): الخدمات المحلية (Ollama, LM Studio, vLLM) تُفعَّل/تُعطَّل تلقائياً حسب إمكانية الاتصال. عند تفعيل خدمة، خلال 15 ثانية تتحول النقطة ● إلى أخضر ويُفعَّل مربع الاختيار، وعند إيقافها تُعطَّل تلقائياً. تعمل بنفس طريقة التبديل التلقائي لخدمات السحابة (Google, OpenRouter, إلخ) بناءً على وجود مفاتيح API.

> 💡 **إذا كانت الخدمة المحلية تعمل على كمبيوتر آخر**: أدخل عنوان IP لذلك الكمبيوتر في حقل URL الخدمة. مثال: `http://192.168.1.20:11434` (Ollama), `http://192.168.1.20:1234` (LM Studio). إذا كانت الخدمة مرتبطة بـ `127.0.0.1` فقط بدلاً من `0.0.0.0`، لن يعمل الوصول عبر IP خارجي، لذا تحقق من عنوان الربط في إعدادات الخدمة.

### إدخال رمز المسؤول

عند محاولة استخدام وظائف مهمة في لوحة التحكم مثل إضافة/حذف المفاتيح، ستظهر نافذة لإدخال رمز المسؤول. أدخل الرمز الذي أعددته في معالج الإعداد. بمجرد إدخاله، يبقى صالحاً حتى إغلاق المتصفح.

> ⚠️ **إذا تجاوزت محاولات المصادقة الفاشلة 10 مرات خلال 15 دقيقة، سيتم حظر عنوان IP مؤقتاً.** إذا نسيت الرمز، تحقق من عنصر `admin_token` في ملف `wall-vault.yaml`.

## الوضع الموزع (متعدد البوتات)

هو تكوين يتم فيه **مشاركة خزنة مفاتيح واحدة** عند تشغيل OpenClaw على عدة أجهزة كمبيوتر في وقت واحد. إنه مريح لأن إدارة المفاتيح تتم في مكان واحد.

### مثال التكوين

```
[خادم خزنة المفاتيح]
  wall-vault vault    (خزنة المفاتيح :56243، لوحة التحكم)

[WSL ألفا]            [Raspberry Pi غاما]    [Mac Mini محلي]
  wall-vault proxy      wall-vault proxy        wall-vault proxy
  openclaw TUI          openclaw TUI            openclaw TUI
  ↕ مزامنة SSE          ↕ مزامنة SSE            ↕ مزامنة SSE
```

جميع البوتات تشير إلى خادم الخزنة المركزي، لذا عند تغيير النموذج أو إضافة مفاتيح في الخزنة، ينعكس التغيير فوراً على جميع البوتات.

### الخطوة 1: بدء خادم خزنة المفاتيح

نفّذ على الكمبيوتر الذي سيُستخدم كخادم خزنة:

```bash
wall-vault vault
```

### الخطوة 2: تسجيل كل بوت (عميل)

سجّل مسبقاً معلومات كل بوت سيتصل بخادم الخزنة:

```bash
curl -X POST http://localhost:56243/admin/clients \
  -H "Authorization: Bearer رمز_المسؤول" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "botA",
    "name": "البوت أ",
    "token": "bota-secret",
    "default_service": "google",
    "default_model": "gemini-2.5-flash"
  }'
```

### الخطوة 3: بدء البروكسي على كل كمبيوتر بوت

نفّذ البروكسي على كل كمبيوتر مثبت عليه البوت مع تحديد عنوان خادم الخزنة والرمز:

```bash
WV_VAULT_URL=http://192.168.x.x:56243 \
WV_VAULT_TOKEN=bota-secret \
WV_VAULT_CLIENT_ID=botA \
wall-vault proxy
```

> 💡 **استبدل `192.168.x.x`** بعنوان IP الداخلي الفعلي لكمبيوتر خادم الخزنة. يمكن التحقق منه في إعدادات الراوتر أو بأمر `ip addr`.

## إعداد التشغيل التلقائي

إذا كان من المزعج تشغيل wall-vault يدوياً في كل مرة يتم إعادة تشغيل الكمبيوتر، سجّله كخدمة نظام. بمجرد التسجيل، سيبدأ تلقائياً عند الإقلاع.

### Linux — systemd (معظم توزيعات Linux)

systemd هو نظام بدء وإدارة البرامج تلقائياً في Linux:

```bash
wall-vault doctor deploy
systemctl --user daemon-reload
systemctl --user enable --now wall-vault
```

عرض السجلات:

```bash
journalctl --user -u wall-vault -f
```

### macOS — launchd

هو النظام المسؤول عن التشغيل التلقائي للبرامج في macOS:

```bash
wall-vault doctor deploy launchd
launchctl load ~/Library/LaunchAgents/com.wall-vault.plist
```

### Windows — NSSM

1. حمّل NSSM من [nssm.cc](https://nssm.cc/download) وأضفه إلى PATH.
2. في PowerShell بصلاحيات المسؤول:

```powershell
wall-vault doctor deploy windows
```

## RTK توفير التوكنات

*(v0.1.24+)*

**RTK (أداة توفير التوكنات)** يضغط تلقائياً مخرجات أوامر الشل التي ينفذها وكلاء الذكاء الاصطناعي (مثل Claude Code)، مما يقلل استهلاك التوكنات. مثلاً، مخرجات `git status` من 15 سطراً تُختصر إلى سطرين.

### الاستخدام الأساسي

```bash
# غلّف الأوامر بـ wall-vault rtk للتصفية التلقائية للمخرجات
wall-vault rtk git status          # يعرض فقط قائمة الملفات المعدلة
wall-vault rtk git diff HEAD~1     # السطور المعدلة + سياق أدنى فقط
wall-vault rtk git log -10         # سطر واحد لكل commit: الهاش + الرسالة
wall-vault rtk go test ./...       # يعرض فقط الاختبارات الفاشلة
wall-vault rtk ls -la              # الأوامر غير المدعومة تُقطع تلقائياً
```

### الأوامر المدعومة والتوفير

| الأمر | طريقة التصفية | التوفير |
|------|----------|--------|
| `git status` | ملخص الملفات المعدلة فقط | ~87% |
| `git diff` | السطور المعدلة + 3 سطور سياق | ~60-94% |
| `git log` | الهاش + أول سطر من الرسالة | ~90% |
| `git push/pull/fetch` | إزالة التقدم، ملخص فقط | ~80% |
| `go test` | عرض الفاشل فقط، عد الناجح | ~88-99% |
| `go build/vet` | عرض الأخطاء فقط | ~90% |
| جميع الأوامر الأخرى | أول 50 + آخر 50 سطراً، بحد أقصى 32KB | متغير |

### خط أنابيب تصفية من 3 مراحل

1. **مرشح هيكلي حسب الأمر** — يفهم تنسيق مخرجات git, go إلخ ويستخلص الأجزاء ذات المعنى فقط
2. **معالجة لاحقة بالتعبيرات النظامية** — إزالة رموز ألوان ANSI، تقليص الأسطر الفارغة، تجميع الأسطر المكررة
3. **مرور + قطع** — الأوامر غير المدعومة تحتفظ فقط بأول/آخر 50 سطراً

### التكامل مع Claude Code

يمكن إعداد جميع أوامر الشل لتمر تلقائياً عبر RTK باستخدام خطاف `PreToolUse` الخاص بـ Claude Code.

```bash
# تثبيت الخطاف (يُضاف تلقائياً إلى settings.json الخاص بـ Claude Code)
wall-vault rtk hook install
```

أو أضف يدوياً إلى `~/.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "command": "wall-vault rtk rewrite"
    }]
  }
}
```

> 💡 **الحفاظ على كود الخروج**: يُرجع RTK كود خروج الأمر الأصلي كما هو. إذا فشل الأمر (كود الخروج ≠ 0)، يكتشف الذكاء الاصطناعي الفشل بدقة.

> 💡 **إخراج إنجليزي إجباري**: ينفذ RTK الأوامر بـ `LC_ALL=C` لإنتاج مخرجات إنجليزية دائماً بغض النظر عن إعدادات لغة النظام. هذا يضمن عمل المرشحات بشكل صحيح.

## مرجع متغيرات البيئة

متغيرات البيئة هي طريقة لتمرير قيم الإعدادات للبرنامج. أدخلها في الطرفية بصيغة `export اسم_المتغير=القيمة` أو ضعها في ملف خدمة التشغيل التلقائي للتطبيق الدائم.

| المتغير | الوصف | قيمة مثال |
|------|------|---------|
| `WV_LANG` | لغة لوحة التحكم | `ko`, `en`, `ja` |
| `WV_THEME` | سمة لوحة التحكم | `light`, `dark`, `gold` |
| `WV_KEY_GOOGLE` | مفتاح/مفاتيح Google API (بفاصلة) | `AIza...,AIza...` |
| `WV_KEY_OPENROUTER` | مفتاح OpenRouter API | `sk-or-v1-...` |
| `WV_VAULT_URL` | عنوان خادم الخزنة في الوضع الموزع | `http://192.168.x.x:56243` |
| `WV_VAULT_TOKEN` | رمز مصادقة العميل (البوت) | `my-secret-token` |
| `WV_ADMIN_TOKEN` | رمز المسؤول | `admin-token-here` |
| `WV_MASTER_PASS` | كلمة مرور تشفير مفاتيح API | `my-password` |
| `WV_AVATAR` | مسار ملف الأفاتار (نسبي إلى `~/.openclaw/`) | `workspace/avatars/avatar.png` |
| `OLLAMA_URL` | عنوان خادم Ollama المحلي | `http://192.168.x.x:11434` |

## حل المشكلات

### عندما لا يبدأ البروكسي

غالباً ما يكون المنفذ مستخدماً بالفعل من قبل برنامج آخر.

```bash
ss -tlnp | grep 56244   # التحقق من يستخدم المنفذ 56244
wall-vault proxy --port 8080   # البدء برقم منفذ آخر
```

### أخطاء مفاتيح API (429, 402, 401, 403, 582)

| كود الخطأ | المعنى | الإجراء |
|----------|------|----------|
| **429** | طلبات كثيرة جداً (تجاوز حد الاستخدام) | انتظر أو أضف مفتاحاً آخر |
| **402** | يلزم الدفع أو الرصيد غير كافٍ | اشحن الرصيد في الخدمة المعنية |
| **401 / 403** | مفتاح خاطئ أو بدون صلاحية | تحقق من قيمة المفتاح وأعد التسجيل |
| **582** | حمل زائد على البوابة (cooldown 5 دقائق) | سيُحرر تلقائياً بعد 5 دقائق |

```bash
# التحقق من قائمة المفاتيح المسجلة وحالتها
curl -H "Authorization: Bearer رمز_المسؤول" http://localhost:56243/admin/keys

# إعادة تعيين عدادات استخدام المفاتيح
curl -X POST -H "Authorization: Bearer رمز_المسؤول" http://localhost:56243/admin/keys/reset
```

### عندما يظهر الوكيل بحالة "غير متصل"

"غير متصل" يعني أن عملية البروكسي لا ترسل إشارات (heartbeat) إلى الخزنة. **لا يعني أن الإعدادات لم تُحفظ.** يجب أن يكون البروكسي قيد التشغيل مع عنوان خادم الخزنة والرمز ليتغير حالة الاتصال.

```bash
# بدء البروكسي مع تحديد عنوان خادم الخزنة والرمز ومعرف العميل
WV_VAULT_URL=http://عنوان_خادم_الخزنة:56243 \
WV_VAULT_TOKEN=رمز_العميل \
WV_VAULT_CLIENT_ID=معرف_العميل \
wall-vault proxy
```

إذا نجح الاتصال، ستتحول الحالة إلى 🟢 قيد التشغيل في لوحة التحكم خلال حوالي 20 ثانية.

### عندما لا يعمل الاتصال بـ Ollama

Ollama هو برنامج يشغّل الذكاء الاصطناعي مباشرة على جهازك. أولاً، تحقق مما إذا كان Ollama مشغّلاً.

```bash
curl http://localhost:11434/api/tags   # إذا ظهرت قائمة النماذج فهو يعمل
export OLLAMA_URL=http://192.168.x.x:11434   # إذا كان يعمل على كمبيوتر آخر
```

> ⚠️ إذا لم يستجب Ollama، ابدأه أولاً بأمر `ollama serve`.

> ⚠️ **النماذج الكبيرة بطيئة**: نماذج كبيرة مثل `qwen3.5:35b` و `deepseek-r1` قد تستغرق عدة دقائق لتوليد الردود. حتى لو بدا أنه لا توجد استجابة، قد يكون المعالجة طبيعية، لذا انتظر.

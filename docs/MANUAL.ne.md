## wall-vault के हो?

**wall-vault = OpenClaw का लागि AI मध्यस्थ (प्रोक्सी) + API कुञ्जी तिजोरी**

AI सेवाहरू प्रयोग गर्न **API कुञ्जी** आवश्यक पर्दछ। API कुञ्जी भनेको **डिजिटल प्रवेश पास** हो — "यो व्यक्तिलाई यो सेवा प्रयोग गर्ने अधिकार छ" भनेर प्रमाणित गर्ने विशेष कोड। यो प्रवेश पासमा दैनिक प्रयोगको सीमा हुन्छ र राम्रोसँग सुरक्षित नगरे चोरी हुने जोखिम पनि हुन्छ।

wall-vault यी प्रवेश पासहरूलाई सुरक्षित तिजोरीमा राख्छ र OpenClaw तथा AI सेवाहरूबीच **मध्यस्थ (प्रोक्सी)** को काम गर्दछ। सजिलो भाषामा भन्दा, OpenClaw लाई केवल wall-vault सँग जोड्नुपर्छ, बाँकी जटिल काम wall-vault ले आफैँ सम्हाल्छ।

wall-vault ले समाधान गर्ने समस्याहरू:

- **API कुञ्जी स्वचालित घुमाउने**: एउटा कुञ्जीको सीमा सकियो वा अस्थायी रूपमा अवरुद्ध भयो (कुलडाउन) भने, चुपचाप अर्को कुञ्जीमा सर्छ। OpenClaw बिना अवरोध चलिरहन्छ।
- **सेवा स्वचालित बदल्ने (फलब्याक)**: Google ले जवाफ नदिए OpenRouter मा, त्यो पनि नभए मेरो कम्प्युटरमा स्थापित Ollama·LM Studio·vLLM (स्थानीय AI) मा स्वचालित रूपमा सर्छ। सत्र टुटँदैन। मूल सेवा पुनर्स्थापित भएपछि अर्को अनुरोधबाट स्वचालित रूपमा फिर्ता हुन्छ (v0.1.18+, LM Studio/vLLM: v0.1.21+)।
- **रियल-टाइम समन्वय (SSE)**: तिजोरी ड्यासबोर्डमा मोडेल बदल्दा १–३ सेकेन्डभित्र OpenClaw स्क्रिनमा देखिन्छ। SSE (Server-Sent Events) भनेको सर्भरले परिवर्तनहरू तुरुन्तै क्लाइन्टलाई पठाउने प्रविधि हो।
- **रियल-टाइम सूचना**: कुञ्जी सकिनु वा सेवा बन्द हुनु जस्ता घटनाहरू OpenClaw TUI (टर्मिनल स्क्रिन) को तल्लो भागमा तुरुन्तै देखिन्छन्।

> 💡 **Claude Code, Cursor, VS Code** पनि जोड्न सकिन्छ, तर wall-vault को मूल उद्देश्य OpenClaw सँग मिलेर प्रयोग गर्नु हो।

```
OpenClaw (TUI टर्मिनल स्क्रिन)
        │
        ▼
  wall-vault प्रोक्सी (:56244)   ← कुञ्जी व्यवस्थापन, राउटिङ, फलब्याक, घटना
        │
        ├─ Google Gemini API
        ├─ OpenRouter API (३४०+ मोडेल)
        ├─ Ollama / LM Studio / vLLM (मेरो कम्प्युटर, अन्तिम विकल्प)
        └─ OpenAI / Anthropic API
```

## स्थापना

### Linux / macOS

टर्मिनल खोल्नुहोस् र तलका आदेशहरू जस्ताको तस्तै टाँस्नुहोस्।

```bash
# Linux (सामान्य PC, सर्भर — amd64)
curl -L https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-linux-amd64 \
  -o ~/.local/bin/wall-vault && chmod +x ~/.local/bin/wall-vault

# macOS Apple Silicon (M1/M2/M3 म्याक)
curl -L https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-darwin-arm64 \
  -o /usr/local/bin/wall-vault && chmod +x /usr/local/bin/wall-vault
```

- `curl -L ...` — इन्टरनेटबाट फाइल डाउनलोड गर्दछ।
- `chmod +x` — डाउनलोड गरिएको फाइललाई "चलाउन योग्य" बनाउँछ। यो चरण छुटाउनुभयो भने "अनुमति छैन" त्रुटि आउँछ।

### Windows

PowerShell (प्रशासक अधिकारमा) खोल्नुहोस् र तलका आदेशहरू चलाउनुहोस्।

```powershell
# डाउनलोड
Invoke-WebRequest -Uri `
  "https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-windows-amd64.exe" `
  -OutFile "$env:LOCALAPPDATA\Programs\wall-vault\wall-vault.exe"

# PATH थप्नुहोस् (PowerShell पुन: सुरु गरेपछि लागू हुन्छ)
$env:PATH += ";$env:LOCALAPPDATA\Programs\wall-vault"
```

> 💡 **PATH भनेको के हो?** कम्प्युटरले आदेशहरू खोज्ने फोल्डरहरूको सूची हो। PATH मा थप्नुभयो भने कुनै पनि फोल्डरबाट `wall-vault` टाइप गरेर चलाउन सकिन्छ।

### स्रोतबाट सिधै निर्माण (विकासकर्ताका लागि)

Go भाषा विकास वातावरण स्थापित भएको अवस्थामा मात्र लागू हुन्छ।

```bash
git clone https://github.com/sookmook/wall-vault
cd wall-vault
make build       # bin/wall-vault (संस्करण: v0.1.24.YYYYMMDD.HHmmss)
make install     # ~/.local/bin/wall-vault
```

> 💡 **निर्माण टाइमस्ट्याम्प संस्करण**: `make build` ले निर्माण गर्दा संस्करण `v0.1.24.20260406.211004` जस्तो मिति·समय समावेश भएको ढाँचामा स्वचालित रूपमा उत्पन्न हुन्छ। `go build ./...` ले सिधै निर्माण गर्दा संस्करणमा `"dev"` मात्र देखिन्छ।

## पहिलो पटक सुरु गर्नुहोस्

### setup जादुगर चलाउनुहोस्

स्थापनापछि, पहिले तलको आदेशले **सेटअप जादुगर** अवश्य चलाउनुहोस्। जादुगरले आवश्यक विषयहरू एक-एक गरी सोध्दै मार्गदर्शन गरिदिन्छ।

```bash
wall-vault setup
```

जादुगरले गर्ने चरणहरू यस्ता छन्:

```
1. भाषा चयन (नेपाली सहित १० भाषा)
2. थिम चयन (light / dark / gold / cherry / ocean)
3. सञ्चालन मोड — एक्लै प्रयोग (standalone), वा धेरै मेसिनमा सँगै (distributed)
4. बट नाम — ड्यासबोर्डमा देखिने नाम
5. पोर्ट सेटिङ — पूर्वनिर्धारित: प्रोक्सी 56244, तिजोरी 56243 (बदल्नु पर्दैन भने Enter थिच्नुहोस्)
6. AI सेवा चयन — Google / OpenRouter / Ollama / LM Studio / vLLM मध्ये प्रयोग गर्ने सेवा
7. उपकरण सुरक्षा फिल्टर सेटिङ
8. प्रशासक टोकन सेटिङ — ड्यासबोर्ड प्रशासन सुविधा लक गर्ने पासवर्ड। स्वचालित उत्पन्न पनि गर्न सकिन्छ
9. API कुञ्जी इन्क्रिप्सन पासवर्ड सेटिङ — कुञ्जीलाई अझ सुरक्षित रूपमा भण्डारण गर्न चाहँदा (ऐच्छिक)
10. कन्फिगरेसन फाइल बचत स्थान
```

> ⚠️ **प्रशासक टोकन अवश्य सम्झनुहोस्।** पछि ड्यासबोर्डमा कुञ्जी थप्न वा सेटिङ बदल्न आवश्यक पर्दछ। बिर्सनुभयो भने कन्फिगरेसन फाइल सिधै सम्पादन गर्नुपर्छ।

जादुगर पूरा भएपछि `wall-vault.yaml` कन्फिगरेसन फाइल स्वचालित रूपमा उत्पन्न हुन्छ।

### चलाउनुहोस्

```bash
wall-vault start
```

तलका दुई सर्भर एकैसाथ सुरु हुन्छन्:

- **प्रोक्सी** (`http://localhost:56244`) — OpenClaw र AI सेवाहरूबीच जोड्ने मध्यस्थ
- **कुञ्जी तिजोरी** (`http://localhost:56243`) — API कुञ्जी व्यवस्थापन तथा वेब ड्यासबोर्ड

ब्राउजरमा `http://localhost:56243` खोल्नुहोस् र ड्यासबोर्ड तुरुन्तै हेर्न सकिन्छ।

## API कुञ्जी दर्ता

API कुञ्जी दर्ता गर्ने चार विधि छन्। **पहिलो पटक सुरु गर्ने व्यक्तिका लागि विधि १ (वातावरण चर) सिफारिस गरिन्छ**।

### विधि १: वातावरण चर (सिफारिस — सबैभन्दा सजिलो)

वातावरण चर भनेको **पहिले नै सेट गरिएका मान** हुन् जुन प्रोग्राम सुरु हुँदा पढ्छ। टर्मिनलमा यसरी टाइप गर्नुहोस्:

```bash
# Google Gemini कुञ्जी दर्ता
export WV_KEY_GOOGLE=AIzaSy...

# OpenRouter कुञ्जी दर्ता
export WV_KEY_OPENROUTER=sk-or-v1-...

# दर्तापछि चलाउनुहोस्
wall-vault start
```

यदि तपाईंसँग धेरै कुञ्जीहरू छन् भने अल्पविराम (,) ले जोड्नुहोस्। wall-vault ले कुञ्जीहरू पालैपालो स्वचालित रूपमा प्रयोग गर्दछ (राउन्ड रोबिन):

```bash
export WV_KEY_GOOGLE=AIzaSy...,AIzaSy...,AIzaSy...
```

> 💡 **सुझाव**: `export` आदेश हालको टर्मिनल सत्रमा मात्र लागू हुन्छ। कम्प्युटर पुन: सुरु गरेपछि पनि कायम राख्न `~/.bashrc` वा `~/.zshrc` फाइलमा माथिको लाइन थप्नुहोस्।

### विधि २: ड्यासबोर्ड UI (माउसले क्लिक गर्ने)

1. ब्राउजरमा `http://localhost:56243` खोल्नुहोस्
2. माथिको **🔑 API कुञ्जी** कार्डमा `[+ थप्नुहोस्]` बटन क्लिक गर्नुहोस्
3. सेवाको प्रकार, कुञ्जीको मान, लेबल (सम्झनाको नाम), र दैनिक सीमा भर्नुहोस् र बचत गर्नुहोस्

### विधि ३: REST API (स्वचालन र स्क्रिप्टका लागि)

REST API भनेको प्रोग्रामहरूले HTTP मार्फत डाटा आदानप्रदान गर्ने तरिका हो। स्क्रिप्टले स्वचालित दर्ता गर्दा उपयोगी हुन्छ।

```bash
curl -X POST http://localhost:56243/admin/keys \
  -H "Authorization: Bearer प्रशासक-टोकन" \
  -H "Content-Type: application/json" \
  -d '{
    "service": "google",
    "key": "AIzaSy...",
    "label": "मुख्य कुञ्जी",
    "daily_limit": 1000
  }'
```

### विधि ४: proxy फ्ल्याग (छोटो परीक्षणको लागि)

औपचारिक दर्ता बिना अस्थायी रूपमा कुञ्जी राखेर परीक्षण गर्न प्रयोग गरिन्छ। प्रोग्राम बन्द गर्दा हराउँछ।

```bash
wall-vault proxy --key-google=AIzaSy... --key-openrouter=sk-or-...
```

## प्रोक्सी प्रयोग विधि

### OpenClaw मा प्रयोग (मुख्य उद्देश्य)

OpenClaw लाई wall-vault मार्फत AI सेवाहरूमा जोड्ने तरिका।

`~/.openclaw/openclaw.json` फाइल खोल्नुहोस् र तलको सामग्री थप्नुहोस्:

```json5
// ~/.openclaw/openclaw.json
{
  models: {
    providers: {
      "wall-vault": {
        baseUrl: "http://localhost:56244/v1",
        apiKey: "your-agent-token",   // vault एजेन्ट टोकन
        api: "openai-completions",
        models: [
          { id: "wall-vault/gemini-2.5-flash" },
          { id: "wall-vault/gemini-2.5-pro" },
          { id: "wall-vault/hunter-alpha" },    // निःशुल्क 1M context
          { id: "wall-vault/claude-opus-4-6" }
        ]
      }
    }
  }
}
```

> 💡 **अझ सजिलो तरिका**: ड्यासबोर्ड एजेन्ट कार्डको **🦞 OpenClaw सेटिङ कपी** बटन थिच्नुहोस्। टोकन र ठेगाना पहिले नै भरिएको स्निपेट क्लिपबोर्डमा कपी हुन्छ। टाँस्नु मात्र पर्छ।

**मोडेल नामको अगाडिको `wall-vault/` कहाँ जोडिन्छ?**

मोडेल नाम हेरेर wall-vault ले कुन AI सेवामा अनुरोध पठाउने स्वचालित रूपमा निर्णय गर्दछ:

| मोडेल ढाँचा | जोडिने सेवा |
|----------|--------------|
| `wall-vault/gemini-*` | Google Gemini सिधै जडान |
| `wall-vault/gpt-*`, `wall-vault/o3`, `wall-vault/o4*` | OpenAI सिधै जडान |
| `wall-vault/claude-*` | OpenRouter मार्फत Anthropic जडान |
| `wall-vault/hunter-alpha`, `wall-vault/healer-alpha` | OpenRouter (निःशुल्क १० लाख टोकन context) |
| `wall-vault/kimi-*`, `wall-vault/glm-*`, `wall-vault/deepseek-*` | OpenRouter जडान |
| `google/मोडेल-नाम`, `openai/मोडेल-नाम`, `anthropic/मोडेल-नाम` आदि | सम्बन्धित सेवा सिधै जडान |
| `custom/google/मोडेल-नाम`, `custom/openai/मोडेल-नाम` आदि | `custom/` भाग हटाएर पुन: राउट |
| `मोडेल-नाम:cloud` | `:cloud` भाग हटाएर OpenRouter मार्फत जडान |

> 💡 **context भनेको के हो?** AI ले एकैपटक सम्झन सक्ने कुराकानीको मात्रा हो। 1M (१० लाख टोकन) भनेको धेरै लामो कुराकानी वा लामो कागजात एकैचोटि प्रशोधन गर्न सकिन्छ भन्ने अर्थ हो।

### Gemini API ढाँचामा सिधै जडान (विद्यमान उपकरण अनुकूलता)

Google Gemini API सिधै प्रयोग गर्ने उपकरणहरू छन् भने, ठेगाना मात्र wall-vault मा बदल्नुहोस्:

```bash
export ANTHROPIC_BASE_URL=http://localhost:56244/google
```

वा URL सिधै निर्दिष्ट गर्ने उपकरणमा:

```
http://localhost:56244/google/v1beta/models/gemini-2.5-flash:generateContent
```

### OpenAI SDK (Python) मा प्रयोग

Python मा AI प्रयोग गर्ने कोडमा पनि wall-vault जोड्न सकिन्छ। `base_url` मात्र बदल्नुहोस्:

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:56244/v1",
    api_key="not-needed"  # API कुञ्जी wall-vault ले आफैँ व्यवस्थापन गर्दछ
)

response = client.chat.completions.create(
    model="google/gemini-2.5-flash",   # provider/model ढाँचामा इनपुट
    messages=[{"role": "user", "content": "नमस्कार"}]
)
```

### चलिरहेको अवस्थामा मोडेल बदल्ने

wall-vault पहिले नै चलिरहेको अवस्थामा प्रयोग गर्ने AI मोडेल बदल्न:

```bash
# प्रोक्सीमा सिधै अनुरोध गरेर मोडेल परिवर्तन
```

ब्राउजरमा `http://localhost:56243` खोल्नुहोस् र ड्यासबोर्ड हेर्न सकिन्छ।

**स्क्रिन संरचना:**

- **कार्ड ग्रिड**: एजेन्ट·सेवा·API कुञ्जी कार्डहरू टाइल आकारमा व्यवस्थित

### API कुञ्जी कार्ड

दर्ता गरिएका API कुञ्जीहरू एकै दृष्टिमा व्यवस्थापन गर्न सकिने कार्ड।

- सेवा अनुसार छुट्याएर कुञ्जी सूची देखाउँछ।
- `today_usage`: आज सफलतापूर्वक प्रशोधन भएका टोकन (AI ले पढेको र लेखेको अक्षर) संख्या
- `today_attempts`: आजको कुल अनुरोध संख्या (सफल + असफल समावेश)
- `[+ थप्नुहोस्]` बटनले नयाँ कुञ्जी दर्ता गर्ने, र `✕` ले कुञ्जी मेटाउने।

> 💡 **टोकन (token) भनेको के हो?** AI ले टेक्स्ट प्रशोधन गर्दा प्रयोग गर्ने एकाइ हो। अंग्रेजीमा लगभग एउटा शब्द, वा अन्य भाषामा १–२ अक्षर बराबर हुन्छ। API शुल्क सामान्यतया यो टोकन संख्या अनुसार गणना गरिन्छ।

#### 🔄 दुवै तर्फी मोडेल सिङ्क्रोनाइजेसन (v0.1.16)

तिजोरी ड्यासबोर्डमा एजेन्टको मोडेल बदल्दा, सम्बन्धित एजेन्टको स्थानीय सेटिङ स्वचालित रूपमा अद्यावधिक हुन्छ।

**Cline को हकमा:**
- तिजोरीमा मोडेल परिवर्तन → SSE घटना → प्रोक्सीले `globalState.json` को मोडेल फिल्ड अद्यावधिक
- अद्यावधिक लक्ष्य: `actModeOpenAiModelId`, `planModeOpenAiModelId`, `openAiModelId`
- `openAiBaseUrl` र API कुञ्जी छुइँदैन
- **VS Code रिलोड (`Ctrl+Alt+R` वा `Ctrl+Shift+P` → `Developer: Reload Window`) आवश्यक छ**
  - Cline ले चलिरहेको अवस्थामा कन्फिगरेसन फाइल पुन: पढ्दैन

**Claude Code को हकमा:**
- तिजोरीमा मोडेल परिवर्तन → SSE घटना → प्रोक्सीले `settings.json` को `model` फिल्ड अद्यावधिक
- WSL र Windows दुवै मार्ग स्वचालित रूपमा खोज्छ (`~/.claude/`, `/mnt/c/Users/*/.claude/`)

**विपरीत दिशा (एजेन्ट → तिजोरी):**
- एजेन्ट (Cline, Claude Code आदि) ले प्रोक्सीमा अनुरोध पठाउँदा, प्रोक्सीले हृदयस्पन्दनमा सम्बन्धित क्लाइन्टको सेवा·मोडेल जानकारी समावेश गर्दछ
- तिजोरी ड्यासबोर्डको एजेन्ट कार्डमा हाल प्रयोग भइरहेको सेवा/मोडेल रियल-टाइममा देखिन्छ

> 💡 **मुख्य कुरा**: प्रोक्सीले अनुरोधको Authorization टोकनबाट एजेन्ट पहिचान गर्दछ र तिजोरीमा सेट गरिएको सेवा/मोडेलमा स्वचालित राउटिङ गर्दछ। Cline वा Claude Code ले अर्को मोडेल नाम पठाए पनि, प्रोक्सीले तिजोरी सेटिङले ओभरराइड गर्दछ।

### VS Code मा Cline प्रयोग गर्ने — विस्तृत मार्गदर्शन

#### चरण १: Cline स्थापना

VS Code एक्सटेन्सन मार्केटप्लेसबाट **Cline** (ID: `saoudrizwan.claude-dev`) स्थापना गर्नुहोस्।

#### चरण २: तिजोरीमा एजेन्ट दर्ता

1. तिजोरी ड्यासबोर्ड (`http://तिजोरी-IP:56243`) खोल्नुहोस्
2. **एजेन्ट** खण्डमा **+ थप्नुहोस्** क्लिक गर्नुहोस्
3. यसरी भर्नुहोस्:

| फिल्ड | मान | विवरण |
|------|----|------|
| ID | `mero_cline` | अद्वितीय पहिचानकर्ता (अंग्रेजी अक्षर, खाली ठाउँ बिना) |

| मोडेल | प्रयोग गर्ने मोडेल भर्नुहोस् (उदा: `gemini-2.5-flash`) | |

4. **बचत** थिच्नुहोस् र टोकन स्वचालित रूपमा उत्पन्न हुन्छ

#### चरण ३: Cline जोड्नुहोस्

**विधि A — स्वचालित लागू (सिफारिस)**

1. सम्बन्धित मेसिनमा wall-vault **प्रोक्सी** चलिरहेको सुनिश्चित गर्नुहोस् (`localhost:56244`)
2. ड्यासबोर्डको एजेन्ट कार्डमा **⚡ Cline सेटिङ लागू** बटन क्लिक गर्नुहोस्
3. "सेटिङ लागू भयो!" सूचना देखियो भने सफल
4. VS Code रिलोड गर्नुहोस् (`Ctrl+Alt+R`)

**विधि B — म्यानुअल सेटिङ**

Cline साइडबारमा सेटिङ (⚙️) खोल्नुहोस्:
- **API Provider**: `OpenAI Compatible`
- **Base URL**: `http://प्रोक्सी-ठेगाना:56244/v1`
  - उही मेसिनमा: `http://localhost:56244/v1`
  - अर्को मेसिन जस्तै Mini सर्भर: `http://192.168.0.6:56244/v1`
- **API Key**: तिजोरीबाट जारी गरिएको टोकन (एजेन्ट कार्डबाट कपी)
- **Model ID**: तिजोरीमा सेट गरिएको मोडेल (उदा: `gemini-2.5-flash`)

#### चरण ४: पुष्टि

Cline च्याट बक्समा कुनै पनि सन्देश पठाउनुहोस्। सामान्य भएमा:
- तिजोरी ड्यासबोर्डको सम्बन्धित एजेन्ट कार्डमा **हरियो थोप्लो (● चलिरहेको)** देखिन्छ
- कार्डमा हालको सेवा/मोडेल देखिन्छ (उदा: `google / gemini-2.5-flash`)

#### मोडेल परिवर्तन

Cline को मोडेल बदल्न चाहनुहुँदा **तिजोरी ड्यासबोर्ड** बाट बदल्नुहोस्:

1. एजेन्ट कार्डको सेवा/मोडेल ड्रपडाउन बदल्नुहोस्
2. **लागू** क्लिक गर्नुहोस्
3. VS Code रिलोड (`Ctrl+Alt+R`) — Cline फुटरको मोडेल नाम अद्यावधिक हुन्छ
4. अर्को अनुरोधबाट नयाँ मोडेल प्रयोग हुन्छ

| Cline फुटर मोडेल बदलिँदैन | Cline ले सेटिङ क्यास गरेको | VS Code रिलोड (`Ctrl+Alt+R`) |
| गलत मोडेल नाम देखिन्छ | पुरानो बग (v0.1.16 मा सुधारिएको) | प्रोक्सी v0.1.16 वा माथि अद्यावधिक गर्नुहोस् |

नयाँ कम्प्युटरमा wall-vault प्रोक्सी पहिलो पटक स्थापना गरेर तिजोरीसँग जोड्दा प्रयोग गरिन्छ। बटन क्लिक गर्दा सम्पूर्ण स्थापना स्क्रिप्ट कपी हुन्छ। नयाँ कम्प्युटरको टर्मिनलमा टाँसेर चलाउनुहोस् र तलका कामहरू एकैचोटि सम्पन्न हुन्छन्:

1. wall-vault बाइनरी स्थापना (पहिले नै स्थापित भए छोड्छ)
2. systemd प्रयोगकर्ता सेवा स्वचालित दर्ता
3. सेवा सुरु र तिजोरी स्वचालित जडान

> 💡 स्क्रिप्टभित्र यो एजेन्टको टोकन र तिजोरी सर्भर ठेगाना पहिले नै भरिएको हुन्छ, त्यसैले टाँसेपछि कुनै परिवर्तन बिना तुरुन्तै चलाउन सकिन्छ।

### सेवा कार्ड

प्रयोग गर्ने AI सेवाहरू खोल्ने/बन्द गर्ने वा सेटिङ गर्ने कार्ड।

- सेवा अनुसार सक्रिय·निष्क्रिय टगल स्विच
- स्थानीय AI सर्भर (मेरो कम्प्युटरमा चलिरहेको Ollama, LM Studio, vLLM आदि) को ठेगाना भर्दा उपलब्ध मोडेलहरू स्वचालित रूपमा पत्ता लगाउँछ।
- **स्थानीय सेवा जडान स्थिति प्रदर्शन**: सेवा नामको छेउमा ● थोप्लो **हरियो** भए जोडिएको, **खैरो** भए जोडिएको छैन
- **स्थानीय सेवा स्वचालित सिग्नल बत्ती** (v0.1.23+): स्थानीय सेवाहरू (Ollama, LM Studio, vLLM) जडान सम्भव भएमा स्वचालित रूपमा सक्रिय र विच्छेद भएमा स्वचालित रूपमा निष्क्रिय हुन्छन्। सेवा खोल्दा १५ सेकेन्डभित्र ● हरियो हुन्छ र चेकबक्स खुल्छ, सेवा बन्द गर्दा स्वचालित रूपमा बन्द हुन्छ। क्लाउड सेवा (Google, OpenRouter आदि) ले API कुञ्जी छ/छैन अनुसार स्वचालित टगल हुने जस्तै हो।

> 💡 **स्थानीय सेवा अर्को कम्प्युटरमा चलिरहेको भए**: सेवा URL इनपुट बक्समा त्यो कम्प्युटरको IP भर्नुहोस्। उदा: `http://192.168.0.6:11434` (Ollama), `http://192.168.0.6:1234` (LM Studio)। सेवा `0.0.0.0` मा नभई `127.0.0.1` मा मात्र बाइन्ड भएको भए बाहिरी IP बाट पहुँच हुँदैन, त्यसैले सेवा सेटिङमा बाइन्डिङ ठेगाना जाँच्नुहोस्।

## वितरित मोड (बहु-बट)

धेरै कम्प्युटरमा एकैसाथ OpenClaw सञ्चालन गर्दा, **एउटा कुञ्जी तिजोरी साझा** गर्ने संरचना। कुञ्जी व्यवस्थापन एकै ठाउँबाट गर्न सकिन्छ।

### संरचना उदाहरण

```
[कुञ्जी तिजोरी सर्भर]
  wall-vault vault    (कुञ्जी तिजोरी :56243, ड्यासबोर्ड)

[WSL अल्फा]            [Raspberry Pi गामा]    [Mac Mini स्थानीय]
  wall-vault proxy      wall-vault proxy        wall-vault proxy
  openclaw TUI          openclaw TUI            openclaw TUI
  ↕ SSE समन्वय          ↕ SSE समन्वय            ↕ SSE समन्वय
```

सबै बटहरू बीचमा रहेको तिजोरी सर्भरतिर हेरिरहेका हुन्छन्, त्यसैले तिजोरीमा मोडेल बदल्दा वा कुञ्जी थप्दा सबै बटमा तुरुन्तै प्रतिबिम्बित हुन्छ।

### चरण १: कुञ्जी तिजोरी सर्भर सुरु

तिजोरी सर्भरको रूपमा प्रयोग गर्ने कम्प्युटरमा चलाउनुहोस्:

```bash
wall-vault vault
```

### चरण २: प्रत्येक बट (क्लाइन्ट) दर्ता

तिजोरी सर्भरमा जोडिने प्रत्येक बटको जानकारी पहिले नै दर्ता गर्नुहोस्:

```bash
curl -X POST http://localhost:56243/admin/clients \
  -H "Authorization: Bearer प्रशासक-टोकन" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "botA",
    "name": "बटA",
    "token": "bota-secret",
    "default_service": "google",
    "default_model": "gemini-2.5-flash"
  }'
```

### चरण ३: प्रत्येक बट कम्प्युटरमा प्रोक्सी सुरु

बट स्थापित भएको प्रत्येक कम्प्युटरमा तिजोरी सर्भर ठेगाना र टोकन निर्दिष्ट गरेर प्रोक्सी चलाउनुहोस्:

```bash
WV_VAULT_URL=http://192.168.x.x:56243 \
WV_VAULT_TOKEN=bota-secret \
WV_VAULT_CLIENT_ID=botA \
wall-vault proxy
```

> 💡 **`192.168.x.x`** भाग तिजोरी सर्भर कम्प्युटरको वास्तविक आन्तरिक IP ठेगानामा बदल्नुहोस्। राउटर सेटिङ वा `ip addr` आदेशले जाँच्न सकिन्छ।

## स्वचालित सुरुआत सेटअप

कम्प्युटर पुन: सुरु गर्दा हरेकपटक म्यानुअल रूपमा wall-vault खोल्नु झन्झटिलो लाग्दा, सिस्टम सेवाको रूपमा दर्ता गर्नुहोस्। एकचोटि दर्ता गरे बुटको बेला स्वचालित रूपमा सुरु हुन्छ।

### Linux — systemd (अधिकांश Linux)

systemd भनेको Linux मा प्रोग्रामहरू स्वचालित रूपमा सुरु र व्यवस्थापन गर्ने प्रणाली हो:

```bash
wall-vault doctor deploy
systemctl --user daemon-reload
systemctl --user enable --now wall-vault
```

लग हेर्ने:

```bash
journalctl --user -u wall-vault -f
```

### macOS — launchd

macOS मा प्रोग्राम स्वचालित चलाउने प्रणाली:

```bash
wall-vault doctor deploy launchd
launchctl load ~/Library/LaunchAgents/com.wall-vault.plist
```

### Windows — NSSM

1. [nssm.cc](https://nssm.cc/download) बाट NSSM डाउनलोड गरेर PATH मा थप्नुहोस्।
2. प्रशासक अधिकारको PowerShell मा:

```powershell
wall-vault doctor deploy windows
```

## RTK टोकन बचत

*(v0.1.24+)*

**RTK (टोकन बचत उपकरण)** ले AI कोडिङ एजेन्ट (Claude Code आदि) ले चलाउने शेल आदेशको आउटपुट स्वचालित रूपमा संकुचित गरेर टोकन प्रयोग घटाउँछ। उदाहरणका लागि, `git status` को १५ लाइन आउटपुट २ लाइन सारांशमा घट्छ।

### आधारभूत प्रयोग

```bash
# आदेशलाई wall-vault rtk ले बेर्नुहोस् र आउटपुट स्वचालित रूपमा फिल्टर हुन्छ
wall-vault rtk git status          # परिवर्तन भएका फाइल सूची मात्र
wall-vault rtk git diff HEAD~1     # परिवर्तन भएका लाइन + न्यूनतम सन्दर्भ मात्र
wall-vault rtk git log -10         # ह्यास + प्रत्येक प्रविष्टिको एक लाइन सन्देश
wall-vault rtk go test ./...       # असफल भएका परीक्षण मात्र
wall-vault rtk ls -la              # समर्थित नभएका आदेश स्वचालित काटिन्छ
```

### समर्थित आदेश र बचत प्रभाव

| आदेश | फिल्टर विधि | बचत दर |
|------|----------|--------|
| `git status` | परिवर्तन भएका फाइल सारांश मात्र | ~८७% |
| `git diff` | परिवर्तन भएका लाइन + ३ लाइन सन्दर्भ | ~६०-९४% |
| `git log` | ह्यास + पहिलो लाइन सन्देश | ~९०% |
| `git push/pull/fetch` | प्रगति हटाउने, सारांश मात्र | ~८०% |
| `go test` | असफल मात्र देखाउने, सफल गन्ने | ~८८-९९% |
| `go build/vet` | त्रुटि मात्र देखाउने | ~९०% |
| अन्य सबै आदेश | अगाडिको ५० लाइन + पछाडिको ५० लाइन, अधिकतम 32KB | परिवर्तनशील |

### ३ चरण फिल्टर पाइपलाइन

1. **आदेशअनुसार संरचना फिल्टर** — git, go आदिको आउटपुट ढाँचा बुझेर अर्थपूर्ण भाग मात्र निकाल्छ
2. **regex पश्च-प्रशोधन** — ANSI रङ कोड हटाउने, खाली लाइन घटाउने, दोहोरिएका लाइन गणना गर्ने
3. **पास-थ्रु + काट्ने** — समर्थित नभएका आदेशमा अगाडि/पछाडि ५० लाइन मात्र राख्ने

### Claude Code एकीकरण

Claude Code को `PreToolUse` हुकले सबै शेल आदेश स्वचालित रूपमा RTK बाट गुजार्न सेट गर्न सकिन्छ।

```bash
# हुक स्थापना (Claude Code settings.json मा स्वचालित थपिन्छ)
wall-vault rtk hook install
```

वा म्यानुअल रूपमा `~/.claude/settings.json` मा थप्नुहोस्:

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "command": "wall-vault rtk rewrite"
    }]
  }
}
```

> 💡 **Exit code संरक्षण**: RTK ले मूल आदेशको समाप्ति कोड जस्ताको तस्तै फिर्ता गर्दछ। आदेश असफल भयो भने (exit code ≠ 0) AI ले पनि असफलता सही रूपमा पत्ता लगाउँछ।

> 💡 **अंग्रेजी बाध्यता**: RTK ले `LC_ALL=C` मा आदेश चलाएर प्रणाली भाषा सेटिङसँग सम्बन्ध नराखी सधैँ अंग्रेजी आउटपुट उत्पन्न गर्दछ। यसले फिल्टर सही रूपमा काम गर्छ।

## वातावरण चर सन्दर्भ

वातावरण चर भनेको प्रोग्राममा सेटिङ मान पठाउने तरिका हो। टर्मिनलमा `export चर-नाम=मान` ढाँचामा इनपुट गर्नुहोस्, वा स्वचालित सुरु सेवा फाइलमा राख्नुहोस् र सधैँ लागू हुन्छ।

| चर | विवरण | उदाहरण मान |
|------|------|---------|
| `WV_LANG` | ड्यासबोर्ड भाषा | `ko`, `en`, `ja` |
| `WV_THEME` | ड्यासबोर्ड थिम | `light`, `dark`, `gold` |
| `WV_KEY_GOOGLE` | Google API कुञ्जी (अल्पविरामले धेरै) | `AIza...,AIza...` |
| `WV_KEY_OPENROUTER` | OpenRouter API कुञ्जी | `sk-or-v1-...` |
| `WV_VAULT_URL` | वितरित मोडमा तिजोरी सर्भर ठेगाना | `http://192.168.x.x:56243` |
| `WV_VAULT_TOKEN` | क्लाइन्ट (बट) प्रमाणीकरण टोकन | `my-secret-token` |
| `WV_ADMIN_TOKEN` | प्रशासक टोकन | `admin-token-here` |
| `WV_MASTER_PASS` | API कुञ्जी इन्क्रिप्सन पासवर्ड | `my-password` |
| `WV_AVATAR` | अवतार छवि फाइल मार्ग (`~/.openclaw/` सापेक्ष मार्ग) | `workspace/avatars/avatar.png` |
| `OLLAMA_URL` | Ollama स्थानीय सर्भर ठेगाना | `http://192.168.x.x:11434` |

## समस्या समाधान

### प्रोक्सी सुरु हुँदैन

पोर्ट पहिले नै अर्को प्रोग्रामले प्रयोग गरिरहेको हुन सक्छ।

```bash
ss -tlnp | grep 56244   # 56244 पोर्ट कसले प्रयोग गरिरहेको छ जाँच्ने
wall-vault proxy --port 8080   # अर्को पोर्ट नम्बरमा सुरु
```

### API कुञ्जी त्रुटि (429, 402, 401, 403, 582)

| त्रुटि कोड | अर्थ | समाधान |
|----------|------|----------|
| **429** | अनुरोध अत्यधिक (प्रयोग सीमा नाघ्यो) | केहीबेर पर्खनुहोस् वा अर्को कुञ्जी थप्नुहोस् |
| **402** | भुक्तानी आवश्यक वा क्रेडिट अपर्याप्त | सम्बन्धित सेवामा क्रेडिट रिचार्ज |
| **401 / 403** | कुञ्जी गलत वा अनुमति छैन | कुञ्जीको मान पुन: जाँच गरेर पुन: दर्ता |
| **582** | गेटवे ओभरलोड (कुलडाउन ५ मिनेट) | ५ मिनेटपछि स्वचालित रूपमा खुल्छ |

```bash
# दर्ता गरिएका कुञ्जी सूची र स्थिति जाँच
curl -H "Authorization: Bearer प्रशासक-टोकन" http://localhost:56243/admin/keys

# कुञ्जी प्रयोग काउन्टर रिसेट
curl -X POST -H "Authorization: Bearer प्रशासक-टोकन" http://localhost:56243/admin/keys/reset
```

### एजेन्ट "नजोडिएको" भनेर देखिन्छ

"नजोडिएको" भनेको प्रोक्सी प्रक्रियाले तिजोरीमा संकेत (heartbeat) पठाइरहेको छैन भन्ने अर्थ हो। **सेटिङ बचत भएको छैन भन्ने अर्थ होइन।** प्रोक्सीले तिजोरी सर्भर ठेगाना र टोकन जानेर चलिरहेको हुनुपर्छ त्यसपछि मात्र जडान स्थितिमा बदलिन्छ।

```bash
# तिजोरी सर्भर ठेगाना, टोकन, र क्लाइन्ट ID निर्दिष्ट गरेर प्रोक्सी सुरु
WV_VAULT_URL=http://तिजोरी-सर्भर-ठेगाना:56243 \
WV_VAULT_TOKEN=क्लाइन्ट-टोकन \
WV_VAULT_CLIENT_ID=क्लाइन्ट-ID \
wall-vault proxy
```

जडान सफल भएमा लगभग २० सेकेन्डमा ड्यासबोर्डमा 🟢 चलिरहेको मा बदलिन्छ।

### Ollama जोडिँदैन

Ollama भनेको मेरो कम्प्युटरमा सिधै AI चलाउने प्रोग्राम हो। पहिले Ollama खुलेको छ कि छैन जाँच्नुहोस्।

```bash
curl http://localhost:11434/api/tags   # मोडेल सूची देखियो भने सामान्य
export OLLAMA_URL=http://192.168.x.x:11434   # अर्को कम्प्युटरमा चलिरहेको भए
```

> ⚠️ Ollama ले प्रतिक्रिया नदिएमा `ollama serve` आदेशले पहिले Ollama सुरु गर्नुहोस्।

> ⚠️ **ठूला मोडेलमा प्रतिक्रिया ढिलो हुन्छ**: `qwen3.5:35b`, `deepseek-r1` जस्ता ठूला मोडेलमा प्रतिक्रिया उत्पन्न गर्न केही मिनेट लाग्न सक्छ। प्रतिक्रिया नआएजस्तो लागे पनि सामान्य रूपमा प्रशोधन भइरहेको हुन सक्छ, त्यसैले पर्खनुहोस्।

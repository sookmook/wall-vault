## ما هو wall-vault؟

**wall-vault = وكيل ذكاء اصطناعي (بروكسي) + خزنة مفاتيح API لـ OpenClaw**

لاستخدام خدمات الذكاء الاصطناعي، تحتاج إلى **مفتاح API**. مفتاح API يشبه **بطاقة دخول رقمية** تثبت أن "هذا الشخص مخول لاستخدام هذه الخدمة". لكن بطاقة الدخول هذه لها عدد محدود من الاستخدامات يوميًا، وقد تتعرض للكشف إذا لم تتم إدارتها بشكل صحيح.

يحفظ wall-vault بطاقات الدخول هذه في خزنة آمنة ويعمل كـ **وكيل (بروكسي)** بين OpenClaw وخدمات الذكاء الاصطناعي. ببساطة، يحتاج OpenClaw فقط للاتصال بـ wall-vault، وسيتولى wall-vault بقية الأمور المعقدة.

المشكلات التي يحلها wall-vault:

- **التدوير التلقائي لمفاتيح API**: عندما يصل مفتاح إلى حد استخدامه أو يتم حظره مؤقتًا (فترة تهدئة)، ينتقل بصمت إلى المفتاح التالي. يستمر OpenClaw في العمل دون انقطاع.
- **الاستبدال التلقائي للخدمة (fallback)**: إذا لم يستجب Google، ينتقل إلى OpenRouter؛ وإذا فشل ذلك أيضًا، ينتقل تلقائيًا إلى الذكاء الاصطناعي المحلي المثبت على جهازك (Ollama، LM Studio، vLLM). لا تنقطع الجلسة. عندما تتعافى الخدمة الأصلية، يعود تلقائيًا من الطلب التالي (v0.1.18+، LM Studio/vLLM: v0.1.21+).
- **المزامنة في الوقت الفعلي (SSE)**: عندما تغير النموذج في لوحة تحكم الخزنة، ينعكس التغيير على شاشة OpenClaw خلال 1-3 ثوانٍ. SSE (Server-Sent Events) هي تقنية تسمح للخادم بإرسال التحديثات للعميل في الوقت الفعلي.
- **الإشعارات في الوقت الفعلي**: تظهر الأحداث مثل نفاد المفاتيح أو أعطال الخدمة فورًا في أسفل واجهة TUI (شاشة الطرفية) لـ OpenClaw.

> 💡 يمكن أيضًا توصيل **Claude Code وCursor وVS Code**، لكن الغرض الأصلي لـ wall-vault هو الاستخدام مع OpenClaw.

```
OpenClaw (واجهة TUI في الطرفية)
        │
        ▼
  wall-vault proxy (:56244)   ← إدارة المفاتيح، التوجيه، الاحتياطي، الأحداث
        │
        ├─ Google Gemini API
        ├─ OpenRouter API (أكثر من 340 نموذج)
        ├─ Ollama / LM Studio / vLLM (جهازك، الملاذ الأخير)
        └─ OpenAI / Anthropic API
```

## التثبيت

### Linux / macOS

افتح الطرفية والصق الأوامر التالية:

```bash
# Linux (كمبيوتر عادي، خادم — amd64)
curl -L https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-linux-amd64 \
  -o ~/.local/bin/wall-vault && chmod +x ~/.local/bin/wall-vault

# macOS Apple Silicon (M1/M2/M3 Mac)
curl -L https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-darwin-arm64 \
  -o /usr/local/bin/wall-vault && chmod +x /usr/local/bin/wall-vault
```

- `curl -L ...` — يقوم بتنزيل الملف من الإنترنت.
- `chmod +x` — يجعل الملف المُنزَّل "قابلاً للتنفيذ". إذا تخطيت هذه الخطوة، سيظهر خطأ "رفض الإذن".

### Windows

افتح PowerShell (كمسؤول) ونفذ الأوامر التالية:

```powershell
# التنزيل
Invoke-WebRequest -Uri `
  "https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-windows-amd64.exe" `
  -OutFile "$env:LOCALAPPDATA\Programs\wall-vault\wall-vault.exe"

# إضافة إلى PATH (يُطبق بعد إعادة تشغيل PowerShell)
$env:PATH += ";$env:LOCALAPPDATA\Programs\wall-vault"
```

> 💡 **ما هو PATH؟** هو قائمة المجلدات التي يبحث فيها الكمبيوتر عن الأوامر. بإضافته إلى PATH، يمكنك تشغيل `wall-vault` من أي مجلد بمجرد كتابة الاسم.

### البناء من الكود المصدري (للمطورين)

ينطبق فقط إذا كانت بيئة تطوير لغة Go مثبتة.

```bash
git clone https://github.com/sookmook/wall-vault
cd wall-vault
make build       # bin/wall-vault (الإصدار: v0.1.24.YYYYMMDD.HHmmss)
make install     # ~/.local/bin/wall-vault
```

> 💡 **إصدار بطابع زمني للبناء**: عند البناء بـ `make build`، يتم إنشاء الإصدار تلقائيًا بتنسيق يتضمن التاريخ والوقت، مثل `v0.1.24.20260406.225957`. إذا بنيت مباشرة بـ `go build ./...`، سيظهر الإصدار فقط كـ `"dev"`.

## البداية الأولى

### تشغيل معالج الإعداد

بعد التثبيت، يجب تشغيل **معالج الإعداد** بالأمر التالي أولاً. سيرشدك المعالج بسؤالك عن كل عنصر مطلوب خطوة بخطوة.

```bash
wall-vault setup
```

الخطوات التي يمر بها المعالج:

```
1. اختيار اللغة (10 لغات بما في ذلك الكورية)
2. اختيار السمة (light / dark / gold / cherry / ocean)
3. وضع التشغيل — للاستخدام الفردي (standalone) أو المشترك على عدة أجهزة (distributed)
4. اسم البوت — الاسم المعروض في لوحة التحكم
5. إعداد المنافذ — الافتراضي: البروكسي 56244، الخزنة 56243 (اضغط Enter إذا لم تحتج للتغيير)
6. اختيار خدمات الذكاء الاصطناعي — Google / OpenRouter / Ollama / LM Studio / vLLM
7. إعداد مرشح أمان الأدوات
8. إعداد رمز المسؤول — كلمة مرور تقفل وظائف إدارة لوحة التحكم. التوليد التلقائي متاح
9. إعداد كلمة مرور تشفير مفاتيح API — لتخزين أكثر أمانًا (اختياري)
10. مسار حفظ ملف الإعدادات
```

> ⚠️ **تأكد من تذكر رمز المسؤول.** ستحتاجه لاحقًا لإضافة المفاتيح أو تغيير الإعدادات في لوحة التحكم. إذا فقدته، ستحتاج لتعديل ملف الإعدادات يدويًا.

بعد إكمال المعالج، سيتم إنشاء ملف الإعدادات `wall-vault.yaml` تلقائيًا.

### التشغيل

```bash
wall-vault start
```

يبدأ الخادمان التاليان في وقت واحد:

- **البروكسي** (`http://localhost:56244`) — الوكيل الذي يربط OpenClaw بخدمات الذكاء الاصطناعي
- **خزنة المفاتيح** (`http://localhost:56243`) — إدارة مفاتيح API ولوحة التحكم عبر الويب

افتح `http://localhost:56243` في المتصفح للوصول إلى لوحة التحكم فورًا.

## تسجيل مفاتيح API

هناك أربع طرق لتسجيل مفاتيح API. **للمبتدئين، نوصي بالطريقة 1 (متغيرات البيئة).**

### الطريقة 1: متغيرات البيئة (موصى بها — الأبسط)

متغيرات البيئة هي **قيم مُعدة مسبقًا** يقرأها البرنامج عند بدء التشغيل. أدخلها في الطرفية كالتالي:

```bash
# تسجيل مفتاح Google Gemini
export WV_KEY_GOOGLE=AIzaSy...

# تسجيل مفتاح OpenRouter
export WV_KEY_OPENROUTER=sk-or-v1-...

# التشغيل بعد التسجيل
wall-vault start
```

إذا كان لديك عدة مفاتيح، اربطها بفاصلة (,). سيستخدمها wall-vault تلقائيًا بالتناوب (round robin):

```bash
export WV_KEY_GOOGLE=AIzaSy...,AIzaSy...,AIzaSy...
```

> 💡 **نصيحة**: أمر `export` يُطبق فقط على جلسة الطرفية الحالية. لجعله يستمر بعد إعادة تشغيل الكمبيوتر، أضف السطر أعلاه إلى ملف `~/.bashrc` أو `~/.zshrc`.

### الطريقة 2: واجهة لوحة التحكم (بالنقر بالماوس)

1. ادخل إلى `http://localhost:56243` في المتصفح
2. انقر على زر `[+ إضافة]` في بطاقة **🔑 مفاتيح API** في الأعلى
3. أدخل نوع الخدمة، قيمة المفتاح، التسمية (اسم مرجعي)، والحد اليومي ثم احفظ

### الطريقة 3: REST API (للأتمتة/البرمجة النصية)

REST API هي طريقة لتبادل البيانات بين البرامج عبر HTTP. مفيدة للتسجيل التلقائي عبر البرامج النصية.

```bash
curl -X POST http://localhost:56243/admin/keys \
  -H "Authorization: Bearer رمز-المسؤول" \
  -H "Content-Type: application/json" \
  -d '{
    "service": "google",
    "key": "AIzaSy...",
    "label": "المفتاح الرئيسي",
    "daily_limit": 1000
  }'
```

### الطريقة 4: علم البروكسي (للاختبار السريع)

استخدمها لاختبار مفتاح مؤقتًا بدون تسجيل رسمي. يختفي المفتاح عند إغلاق البرنامج.

```bash
wall-vault proxy --key-google=AIzaSy... --key-openrouter=sk-or-...
```

## كيفية استخدام البروكسي

### الاستخدام مع OpenClaw (الغرض الرئيسي)

إليك كيفية إعداد OpenClaw للاتصال بخدمات الذكاء الاصطناعي عبر wall-vault.

افتح ملف `~/.openclaw/openclaw.json` وأضف المحتوى التالي:

```json5
// ~/.openclaw/openclaw.json
{
  models: {
    providers: {
      "wall-vault": {
        baseUrl: "http://localhost:56244/v1",
        apiKey: "your-agent-token",   // رمز وكيل الخزنة
        api: "openai-completions",
        models: [
          { id: "wall-vault/gemini-2.5-flash" },
          { id: "wall-vault/gemini-2.5-pro" },
          { id: "wall-vault/hunter-alpha" },    // سياق 1M مجاني
          { id: "wall-vault/claude-opus-4-6" }
        ]
      }
    }
  }
}
```

> 💡 **طريقة أسهل**: اضغط على زر **🦞 نسخ إعدادات OpenClaw** في بطاقة الوكيل في لوحة التحكم وسيتم نسخ مقتطف بالرمز والعنوان مملوءين مسبقًا إلى الحافظة. ما عليك سوى اللصق.

**إلى أين يوجه `wall-vault/` في بداية اسم النموذج؟**

يحدد wall-vault تلقائيًا أي خدمة ذكاء اصطناعي يرسل إليها الطلب بناءً على اسم النموذج:

| صيغة النموذج | الخدمة المتصلة |
|----------|--------------|
| `wall-vault/gemini-*` | اتصال مباشر بـ Google Gemini |
| `wall-vault/gpt-*`, `wall-vault/o3`, `wall-vault/o4*` | اتصال مباشر بـ OpenAI |
| `wall-vault/claude-*` | اتصال بـ Anthropic عبر OpenRouter |
| `wall-vault/hunter-alpha`, `wall-vault/healer-alpha` | OpenRouter (مليون رمز سياق مجاني) |
| `wall-vault/kimi-*`, `wall-vault/glm-*`, `wall-vault/deepseek-*` | اتصال عبر OpenRouter |
| `google/اسم-النموذج`, `openai/اسم-النموذج`, `anthropic/اسم-النموذج` إلخ. | اتصال مباشر بالخدمة المعنية |
| `custom/google/اسم-النموذج`, `custom/openai/اسم-النموذج` إلخ. | يزيل جزء `custom/` ويعيد التوجيه |
| `اسم-النموذج:cloud` | يزيل جزء `:cloud` ويتصل عبر OpenRouter |

> 💡 **ما هو السياق (context)؟** هو حجم المحادثة الذي يمكن للذكاء الاصطناعي "تذكره" في المرة الواحدة. 1M (مليون رمز) يعني القدرة على معالجة محادثات طويلة جدًا أو مستندات كبيرة دفعة واحدة.

### الاتصال المباشر بتنسيق Gemini API (التوافق مع الأدوات الحالية)

إذا كانت لديك أدوات تستخدم Google Gemini API مباشرة، فقط غيّر العنوان إلى wall-vault:

```bash
export ANTHROPIC_BASE_URL=http://localhost:56244/google
```

أو للأدوات التي تحدد URL مباشرة:

```
http://localhost:56244/google/v1beta/models/gemini-2.5-flash:generateContent
```

### الاستخدام مع OpenAI SDK (بايثون)

يمكنك أيضًا توصيل wall-vault في كود بايثون يستخدم الذكاء الاصطناعي. فقط غيّر `base_url`:

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:56244/v1",
    api_key="not-needed"  # يدير wall-vault مفاتيح API تلقائيًا
)

response = client.chat.completions.create(
    model="google/gemini-2.5-flash",   # أدخل بتنسيق provider/model
    messages=[{"role": "user", "content": "مرحبًا"}]
)
```

### تغيير النموذج أثناء التشغيل

لتغيير نموذج الذكاء الاصطناعي بينما wall-vault قيد التشغيل:

```bash
# تغيير النموذج بطلب مباشر للبروكسي
curl -X PUT http://localhost:56244/api/config/model \
  -H "Content-Type: application/json" \
  -d '{"service": "openrouter", "model": "anthropic/claude-3.5-sonnet"}'

# في الوضع الموزع (بوتات متعددة)، غيّر من خادم الخزنة → ينعكس فورًا عبر SSE
curl -X PUT http://localhost:56243/admin/clients/معرف-البوت \
  -H "Authorization: Bearer رمز-المسؤول" \
  -H "Content-Type: application/json" \
  -d '{"default_service": "google", "default_model": "gemini-2.5-pro"}'
```

### التحقق من قائمة النماذج المتاحة

```bash
# عرض القائمة الكاملة
curl http://localhost:56244/api/models | python3 -m json.tool

# عرض نماذج Google فقط
curl "http://localhost:56244/api/models?service=google"

# البحث بالاسم (مثال: النماذج التي تحتوي على "claude")
curl "http://localhost:56244/api/models?q=claude"
```

**ملخص النماذج الرئيسية حسب الخدمة:**

| الخدمة | النماذج الرئيسية |
|--------|----------|
| Google | gemini-2.5-pro, gemini-2.5-flash, gemini-2.5-flash-8b, gemini-2.0-flash |
| OpenAI | gpt-4o, gpt-4o-mini, o3, o1, o1-mini |
| OpenRouter | أكثر من 346 (Hunter Alpha سياق 1M مجاني، DeepSeek R1/V3، Qwen 2.5 إلخ.) |
| Ollama | كشف تلقائي للخادم المحلي المثبت على الكمبيوتر |
| LM Studio | خادم محلي على الكمبيوتر (المنفذ 1234) |
| vLLM | خادم محلي على الكمبيوتر (المنفذ 8000) |

## لوحة تحكم خزنة المفاتيح

ادخل إلى `http://localhost:56243` في المتصفح لعرض لوحة التحكم.

**هيكل الشاشة:**
- **الشريط العلوي الثابت (topbar)**: الشعار، محددات اللغة والسمة، مؤشر حالة اتصال SSE
- **شبكة البطاقات**: بطاقات الوكلاء والخدمات ومفاتيح API مرتبة على شكل بلاطات

### بطاقة مفاتيح API

بطاقة تتيح إدارة جميع مفاتيح API المسجلة بنظرة واحدة.

- تعرض قائمة المفاتيح مصنفة حسب الخدمة.
- `today_usage`: عدد الرموز (عدد الأحرف التي قرأها وكتبها الذكاء الاصطناعي) المعالجة بنجاح اليوم
- `today_attempts`: إجمالي عدد الاستدعاءات اليوم (بما في ذلك النجاح + الفشل)
- زر `[+ إضافة]` لتسجيل مفاتيح جديدة و`✕` لحذفها.

> 💡 **ما هو الرمز (token)؟** هو الوحدة التي يستخدمها الذكاء الاصطناعي لمعالجة النص. يعادل تقريبًا كلمة واحدة بالإنجليزية أو 1-2 حرف بالكورية. عادة ما يتم حساب رسوم API بناءً على عدد الرموز.

### بطاقة الوكيل

بطاقة تعرض حالة البوتات (الوكلاء) المتصلة ببروكسي wall-vault.

**يتم عرض حالة الاتصال في 4 مستويات:**

| المؤشر | الحالة | المعنى |
|------|------|------|
| 🟢 | قيد التشغيل | البروكسي يعمل بشكل طبيعي |
| 🟡 | متأخر | يستجيب لكن ببطء |
| 🔴 | غير متصل | البروكسي لا يستجيب |
| ⚫ | غير متصل/معطل | البروكسي لم يتصل بالخزنة من قبل أو معطل |

**دليل الأزرار في أسفل بطاقة الوكيل:**

عند تسجيل وكيل، إذا حددت **نوع الوكيل**، تظهر أزرار مناسبة لهذا النوع تلقائيًا.

---

#### 🔘 زر نسخ الإعدادات — ينشئ إعدادات الاتصال تلقائيًا

عند النقر على الزر، يتم نسخ مقتطف إعدادات مع الرمز وعنوان البروكسي ومعلومات النموذج مملوءة مسبقًا إلى الحافظة. ما عليك سوى لصق المحتوى المنسوخ في الموقع المشار إليه في الجدول أدناه لإكمال إعداد الاتصال.

| الزر | نوع الوكيل | مكان اللصق |
|------|-------------|-------------|
| 🦞 نسخ إعدادات OpenClaw | `openclaw` | `~/.openclaw/openclaw.json` |
| 🦀 نسخ إعدادات NanoClaw | `nanoclaw` | `~/.openclaw/openclaw.json` |
| 🟠 نسخ إعدادات Claude Code | `claude-code` | `~/.claude/settings.json` |
| ⌨ نسخ إعدادات Cursor | `cursor` | Cursor → Settings → AI |
| 💻 نسخ إعدادات VSCode | `vscode` | `~/.continue/config.json` |

**مثال — إذا كان النوع Claude Code، يتم نسخ المحتوى التالي:**

```json
// ~/.claude/settings.json
{
  "apiProvider": "openai",
  "baseUrl": "http://192.168.1.20:56244/v1",
  "apiKey": "رمز-هذا-الوكيل"
}
```

**مثال — إذا كان النوع VSCode (Continue):**

```yaml
# ~/.continue/config.yaml  ← الصق في config.yaml وليس config.json
name: My Config
version: 0.0.1
```

#### ⚡ زر التطبيق التلقائي — اضغط مرة واحدة وينتهي الإعداد

للوكلاء من نوع `cline` أو `claude-code` أو `openclaw` أو `nanoclaw`، يظهر زر **⚡ تطبيق الإعدادات** في بطاقة الوكيل. عند الضغط على هذا الزر، يتم تحديث ملف الإعدادات المحلي للوكيل تلقائيًا.

| الزر | نوع الوكيل | الملف المستهدف |
|------|-------------|-------------|
| ⚡ تطبيق إعدادات Cline | `cline` | `~/.cline/data/globalState.json` + `secrets.json` |
| ⚡ تطبيق إعدادات Claude Code | `claude-code` | `~/.claude/settings.json` |
| ⚡ تطبيق إعدادات OpenClaw | `openclaw` | `~/.openclaw/openclaw.json` |
| ⚡ تطبيق إعدادات NanoClaw | `nanoclaw` | `~/.openclaw/openclaw.json` |

> ⚠️ يرسل هذا الزر طلبات إلى **localhost:56244** (البروكسي المحلي). يجب أن يكون البروكسي قيد التشغيل على هذا الجهاز ليعمل.

#### 🔄 مزامنة النموذج ثنائية الاتجاه (v0.1.16)

عندما تغير نموذج وكيل في لوحة تحكم الخزنة، يتم تحديث الإعدادات المحلية للوكيل تلقائيًا.

**في حالة Cline:**
- تغيير النموذج في الخزنة → حدث SSE → البروكسي يحدث حقل النموذج في `globalState.json`
- الحقول المحدثة: `actModeOpenAiModelId`, `planModeOpenAiModelId`, `openAiModelId`
- لا يتم تغيير `openAiBaseUrl` ومفتاح API
- **يلزم إعادة تحميل VS Code (`Ctrl+Alt+R` أو `Ctrl+Shift+P` → `Developer: Reload Window`)**
  - لأن Cline لا يعيد قراءة ملف الإعدادات أثناء التشغيل

**في حالة Claude Code:**
- تغيير النموذج في الخزنة → حدث SSE → البروكسي يحدث حقل `model` في `settings.json`
- بحث تلقائي في مسارات WSL وWindows (`~/.claude/`, `/mnt/c/Users/*/.claude/`)

**الاتجاه المعاكس (الوكيل → الخزنة):**
- عندما يرسل وكيل (Cline، Claude Code إلخ.) طلبًا للبروكسي، يضمن البروكسي معلومات الخدمة/النموذج للعميل في نبض القلب
- يتم عرض الخدمة/النموذج المستخدم حاليًا في الوقت الفعلي في بطاقة الوكيل في لوحة تحكم الخزنة

> 💡 **النقطة الأساسية**: يحدد البروكسي الوكيل من خلال رمز Authorization في الطلب ويوجهه تلقائيًا إلى الخدمة/النموذج المُعد في الخزنة. حتى لو أرسل Cline أو Claude Code اسم نموذج مختلف، يستبدله البروكسي بإعدادات الخزنة.

### استخدام Cline في VS Code — دليل مفصل

#### الخطوة 1: تثبيت Cline

ثبت **Cline** (المعرف: `saoudrizwan.claude-dev`) من سوق إضافات VS Code.

#### الخطوة 2: تسجيل الوكيل في الخزنة

1. افتح لوحة تحكم الخزنة (`http://عنوان-الخزنة:56243`)
2. في قسم **الوكلاء**، انقر **+ إضافة**
3. أدخل ما يلي:

| الحقل | القيمة | الوصف |
|------|----|------|
| المعرف | `my_cline` | معرف فريد (بالإنجليزية، بدون مسافات) |
| الاسم | `My Cline` | الاسم المعروض في لوحة التحكم |
| نوع الوكيل | `cline` | ← يجب اختيار `cline` |
| الخدمة | اختر الخدمة المطلوبة (مثال: `google`) | |
| النموذج | أدخل النموذج المطلوب (مثال: `gemini-2.5-flash`) | |

4. عند النقر على **حفظ**، يتم إنشاء الرمز تلقائيًا

#### الخطوة 3: الاتصال بـ Cline

**الطريقة أ — التطبيق التلقائي (موصى بها)**

1. تأكد من أن **بروكسي** wall-vault قيد التشغيل على الجهاز (`localhost:56244`)
2. انقر على زر **⚡ تطبيق إعدادات Cline** في بطاقة الوكيل في لوحة التحكم
3. إذا ظهر إشعار "تم تطبيق الإعدادات بنجاح!"، فقد نجحت العملية
4. أعد تحميل VS Code (`Ctrl+Alt+R`)

**الطريقة ب — الإعداد اليدوي**

افتح الإعدادات (⚙️) في الشريط الجانبي لـ Cline:
- **API Provider**: `OpenAI Compatible`
- **Base URL**: `http://عنوان-البروكسي:56244/v1`
  - على نفس الجهاز: `http://localhost:56244/v1`
  - على جهاز آخر مثل خادم ميني: `http://192.168.1.20:56244/v1`
- **API Key**: الرمز الصادر من الخزنة (انسخه من بطاقة الوكيل)
- **Model ID**: النموذج المُعد في الخزنة (مثال: `gemini-2.5-flash`)

#### الخطوة 4: التحقق

أرسل أي رسالة في نافذة دردشة Cline. إذا كان كل شيء طبيعيًا:
- ستظهر **نقطة خضراء (● قيد التشغيل)** في بطاقة الوكيل المعني في لوحة تحكم الخزنة
- سيظهر الخدمة/النموذج الحالي في البطاقة (مثال: `google / gemini-2.5-flash`)

#### تغيير النموذج

عندما تريد تغيير نموذج Cline، قم بالتغيير من **لوحة تحكم الخزنة**:

1. غيّر قائمة الخدمة/النموذج المنسدلة في بطاقة الوكيل
2. انقر **تطبيق**
3. أعد تحميل VS Code (`Ctrl+Alt+R`) — سيتم تحديث اسم النموذج في تذييل Cline
4. سيتم استخدام النموذج الجديد من الطلب التالي

> 💡 في الواقع، يحدد البروكسي طلب Cline بالرمز ويوجهه إلى النموذج المُعد في الخزنة. حتى بدون إعادة تحميل VS Code، **يتغير النموذج المستخدم فعليًا فورًا** — إعادة التحميل فقط لتحديث عرض النموذج في واجهة Cline.

#### كشف انقطاع الاتصال

عند إغلاق VS Code، تتحول بطاقة الوكيل في لوحة تحكم الخزنة إلى اللون الأصفر (متأخر) بعد حوالي **90 ثانية** وإلى اللون الأحمر (غير متصل) بعد **3 دقائق**. (اعتبارًا من v0.1.18، أصبح كشف عدم الاتصال أسرع مع فحص الحالة كل 15 ثانية.)

#### استكشاف الأخطاء وإصلاحها

| العرض | السبب | الحل |
|------|------|------|
| خطأ "فشل الاتصال" في Cline | البروكسي لا يعمل أو العنوان خاطئ | تحقق من البروكسي بـ `curl http://localhost:56244/health` |
| النقطة الخضراء لا تظهر في الخزنة | مفتاح API (الرمز) غير مُعد | انقر على زر **⚡ تطبيق إعدادات Cline** مرة أخرى |
| نموذج تذييل Cline لا يتغير | Cline يخزن الإعدادات مؤقتًا | أعد تحميل VS Code (`Ctrl+Alt+R`) |
| يظهر اسم نموذج خاطئ | خطأ قديم (تم إصلاحه في v0.1.16) | حدّث البروكسي إلى v0.1.16 أو أحدث |

#### 🟣 زر نسخ أمر النشر — استخدمه عند التثبيت على جهاز جديد

استخدمه عند تثبيت بروكسي wall-vault لأول مرة على كمبيوتر جديد وتوصيله بالخزنة. عند النقر على الزر، يتم نسخ سكريبت التثبيت بالكامل. الصقه في طرفية الكمبيوتر الجديد ونفذه لمعالجة كل شيء دفعة واحدة:

1. تثبيت ملف wall-vault الثنائي (يتخطى إذا كان مثبتًا بالفعل)
2. تسجيل تلقائي لخدمة مستخدم systemd
3. بدء الخدمة والاتصال التلقائي بالخزنة

> 💡 يحتوي السكريبت بالفعل على رمز هذا الوكيل وعنوان خادم الخزنة مملوءين، لذا يمكن تنفيذه فورًا بعد اللصق بدون تعديلات إضافية.

### بطاقة الخدمة

بطاقة لتفعيل/تعطيل وإعداد خدمات الذكاء الاصطناعي.

- مفاتيح تبديل التفعيل/التعطيل لكل خدمة
- عند إدخال عنوان خوادم الذكاء الاصطناعي المحلية (Ollama، LM Studio، vLLM إلخ. العاملة على جهازك)، يتم اكتشاف النماذج المتاحة تلقائيًا.
- **مؤشر حالة اتصال الخدمة المحلية**: النقطة ● بجانب اسم الخدمة تكون **خضراء** عند الاتصال و**رمادية** عند عدم الاتصال
- **إشارة مرور تلقائية للخدمة المحلية** (v0.1.23+): يتم تفعيل/تعطيل الخدمات المحلية (Ollama، LM Studio، vLLM) تلقائيًا حسب إمكانية الاتصال. عند تفعيل الخدمة، تتحول إلى ● خضراء خلال 15 ثانية ويتم تفعيل مربع الاختيار؛ وعند التعطيل، يتم إيقافها تلقائيًا. تعمل بنفس طريقة التبديل التلقائي لخدمات السحابة (Google، OpenRouter إلخ.) بناءً على وجود مفاتيح API.

> 💡 **إذا كانت الخدمة المحلية تعمل على كمبيوتر آخر**: أدخل عنوان IP لذلك الكمبيوتر في حقل URL الخدمة. مثال: `http://192.168.1.20:11434` (Ollama)، `http://192.168.1.20:1234` (LM Studio). إذا كانت الخدمة مرتبطة بـ `127.0.0.1` فقط وليس `0.0.0.0`، فلن يمكن الوصول إليها بعنوان IP الخارجي — تحقق من عنوان الربط في إعدادات الخدمة.

### إدخال رمز المسؤول

عند محاولة استخدام وظائف مهمة في لوحة التحكم مثل إضافة/حذف المفاتيح، تظهر نافذة منبثقة لإدخال رمز المسؤول. أدخل الرمز الذي تم إعداده في معالج الإعداد. بمجرد إدخاله، يظل ساريًا حتى إغلاق المتصفح.

> ⚠️ **إذا تجاوزت محاولات المصادقة الفاشلة 10 مرات في 15 دقيقة، سيتم حظر عنوان IP مؤقتًا.** إذا نسيت الرمز، تحقق من عنصر `admin_token` في ملف `wall-vault.yaml`.

## الوضع الموزع (بوتات متعددة)

عند تشغيل OpenClaw على عدة أجهزة في وقت واحد، هذا هو التكوين لـ **مشاركة خزنة مفاتيح واحدة**. مريح لأن إدارة المفاتيح تتم في مكان واحد.

### مثال على التكوين

```
[خادم خزنة المفاتيح]
  wall-vault vault    (خزنة المفاتيح :56243، لوحة التحكم)

[WSL ألفا]            [Raspberry Pi غاما]    [Mac Mini محلي]
  wall-vault proxy      wall-vault proxy        wall-vault proxy
  openclaw TUI          openclaw TUI            openclaw TUI
  ↕ مزامنة SSE          ↕ مزامنة SSE            ↕ مزامنة SSE
```

جميع البوتات تشير إلى خادم الخزنة المركزي، لذا عند تغيير النموذج أو إضافة المفاتيح في الخزنة، ينعكس التغيير فورًا على جميع البوتات.

### الخطوة 1: بدء خادم خزنة المفاتيح

نفذ على الكمبيوتر الذي سيستخدم كخادم خزنة:

```bash
wall-vault vault
```

### الخطوة 2: تسجيل كل بوت (عميل)

سجل مسبقًا معلومات كل بوت سيتصل بخادم الخزنة:

```bash
curl -X POST http://localhost:56243/admin/clients \
  -H "Authorization: Bearer رمز-المسؤول" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "botA",
    "name": "بوت أ",
    "token": "bota-secret",
    "default_service": "google",
    "default_model": "gemini-2.5-flash"
  }'
```

### الخطوة 3: بدء البروكسي على كل كمبيوتر بوت

على كل كمبيوتر مثبت عليه البوت، نفذ البروكسي مع تحديد عنوان خادم الخزنة والرمز:

```bash
WV_VAULT_URL=http://192.168.x.x:56243 \
WV_VAULT_TOKEN=bota-secret \
WV_VAULT_CLIENT_ID=botA \
wall-vault proxy
```

> 💡 **`192.168.x.x`** يجب استبداله بعنوان IP الداخلي الفعلي لكمبيوتر خادم الخزنة. يمكنك التحقق منه من إعدادات الراوتر أو بأمر `ip addr`.

## إعداد التشغيل التلقائي

إذا كان تشغيل wall-vault يدويًا في كل مرة تعيد فيها تشغيل الكمبيوتر أمرًا مزعجًا، سجله كخدمة نظام. بمجرد تسجيله، يبدأ تلقائيًا عند الإقلاع.

### Linux — systemd (معظم توزيعات Linux)

systemd هو النظام الذي يبدأ ويدير البرامج تلقائيًا في Linux:

```bash
wall-vault doctor deploy
systemctl --user daemon-reload
systemctl --user enable --now wall-vault
```

عرض السجلات:

```bash
journalctl --user -u wall-vault -f
```

### macOS — launchd

النظام المسؤول عن التشغيل التلقائي للبرامج في macOS:

```bash
wall-vault doctor deploy launchd
launchctl load ~/Library/LaunchAgents/com.wall-vault.plist
```

### Windows — NSSM

1. حمل NSSM من [nssm.cc](https://nssm.cc/download) وأضفه إلى PATH.
2. في PowerShell كمسؤول:

```powershell
wall-vault doctor deploy windows
```

## Doctor (التشخيص)

أمر `doctor` هو أداة **تشخص وتصلح تلقائيًا** ما إذا كان wall-vault مُعدًا بشكل صحيح.

```bash
wall-vault doctor check   # تشخيص الحالة الحالية (قراءة فقط، لا يغير شيئًا)
wall-vault doctor fix     # إصلاح المشكلات تلقائيًا
wall-vault doctor all     # تشخيص + إصلاح تلقائي دفعة واحدة
```

> 💡 إذا بدا شيء غريبًا، نفذ `wall-vault doctor all` أولاً. يكتشف ويصلح العديد من المشكلات تلقائيًا.

## RTK توفير الرموز

*(v0.1.24+)*

**RTK (أداة توفير الرموز)** تضغط تلقائيًا مخرجات أوامر الصدفة التي ينفذها وكلاء البرمجة بالذكاء الاصطناعي (مثل Claude Code)، مما يقلل استهلاك الرموز. على سبيل المثال، يتم تقليص مخرجات `git status` من 15 سطرًا إلى ملخص من سطرين.

### الاستخدام الأساسي

```bash
# غلف الأمر بـ wall-vault rtk وسيتم تصفية المخرجات تلقائيًا
wall-vault rtk git status          # يعرض فقط قائمة الملفات المتغيرة
wall-vault rtk git diff HEAD~1     # فقط الأسطر المتغيرة + سياق أدنى
wall-vault rtk git log -10         # الهاش + رسالة سطر واحد
wall-vault rtk go test ./...       # يعرض فقط الاختبارات الفاشلة
wall-vault rtk ls -la              # الأوامر غير المدعومة تُقطع تلقائيًا
```

### الأوامر المدعومة ونسب التوفير

| الأمر | طريقة التصفية | التوفير |
|------|----------|--------|
| `git status` | ملخص الملفات المتغيرة فقط | ~87% |
| `git diff` | الأسطر المتغيرة + 3 أسطر سياق | ~60-94% |
| `git log` | الهاش + السطر الأول من الرسالة | ~90% |
| `git push/pull/fetch` | إزالة التقدم، الملخص فقط | ~80% |
| `go test` | عرض الفشل فقط، عد النجاح | ~88-99% |
| `go build/vet` | عرض الأخطاء فقط | ~90% |
| جميع الأوامر الأخرى | أول 50 سطرًا + آخر 50 سطرًا، بحد أقصى 32KB | متغير |

### خط أنابيب التصفية من 3 مراحل

1. **مرشح هيكلي لكل أمر** — يفهم تنسيق مخرجات git، go إلخ. ويستخرج الأجزاء ذات المعنى فقط
2. **معالجة لاحقة بالتعبيرات النمطية** — إزالة رموز ألوان ANSI، تقليل الأسطر الفارغة، تجميع الأسطر المكررة
3. **تمرير + قطع** — الأوامر غير المدعومة تحتفظ بأول/آخر 50 سطرًا فقط

### التكامل مع Claude Code

يمكنك إعداد خطاف `PreToolUse` في Claude Code ليمر كل أمر صدفة تلقائيًا عبر RTK.

```bash
# تثبيت الخطاف (يُضاف تلقائيًا إلى settings.json في Claude Code)
wall-vault rtk hook install
```

أو أضف يدويًا إلى `~/.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "command": "wall-vault rtk rewrite"
    }]
  }
}
```

> 💡 **الحفاظ على رمز الخروج**: يعيد RTK رمز الخروج الأصلي للأمر. إذا فشل الأمر (رمز الخروج ≠ 0)، يكتشف الذكاء الاصطناعي أيضًا الفشل بشكل صحيح.

> 💡 **إجبار المخرجات بالإنجليزية**: ينفذ RTK الأوامر بـ `LC_ALL=C` لإنتاج مخرجات إنجليزية دائمًا بغض النظر عن إعدادات لغة النظام. هذا ضروري لعمل المرشحات بشكل صحيح.

## مرجع متغيرات البيئة

متغيرات البيئة هي طريقة لتمرير قيم الإعدادات للبرنامج. أدخلها في الطرفية بتنسيق `export اسم_المتغير=قيمة` أو ضعها في ملف خدمة التشغيل التلقائي للتطبيق الدائم.

| المتغير | الوصف | قيمة مثال |
|------|------|---------|
| `WV_LANG` | لغة لوحة التحكم | `ko`, `en`, `ja` |
| `WV_THEME` | سمة لوحة التحكم | `light`, `dark`, `gold` |
| `WV_KEY_GOOGLE` | مفتاح Google API (متعددة بفاصلة) | `AIza...,AIza...` |
| `WV_KEY_OPENROUTER` | مفتاح OpenRouter API | `sk-or-v1-...` |
| `WV_VAULT_URL` | عنوان خادم الخزنة في الوضع الموزع | `http://192.168.x.x:56243` |
| `WV_VAULT_TOKEN` | رمز مصادقة العميل (البوت) | `my-secret-token` |
| `WV_ADMIN_TOKEN` | رمز المسؤول | `admin-token-here` |
| `WV_MASTER_PASS` | كلمة مرور تشفير مفاتيح API | `my-password` |
| `WV_AVATAR` | مسار ملف صورة الأفاتار (مسار نسبي من `~/.openclaw/`) | `workspace/avatars/avatar.png` |
| `OLLAMA_URL` | عنوان خادم Ollama المحلي | `http://192.168.x.x:11434` |

## استكشاف الأخطاء وإصلاحها

### عندما لا يبدأ البروكسي

عادة يكون المنفذ مستخدمًا بالفعل من قبل برنامج آخر.

```bash
ss -tlnp | grep 56244   # تحقق من يستخدم المنفذ 56244
wall-vault proxy --port 8080   # ابدأ برقم منفذ آخر
```

### عند حدوث أخطاء مفاتيح API (429, 402, 401, 403, 582)

| رمز الخطأ | المعنى | الإجراء |
|----------|------|----------|
| **429** | طلبات كثيرة جدًا (تجاوز حد الاستخدام) | انتظر قليلاً أو أضف مفتاحًا آخر |
| **402** | الدفع مطلوب أو رصيد غير كافٍ | اشحن الرصيد في الخدمة المعنية |
| **401 / 403** | مفتاح خاطئ أو بدون إذن | تحقق من قيمة المفتاح مرة أخرى وأعد تسجيله |
| **582** | حمل زائد على البوابة (تهدئة 5 دقائق) | يُلغى تلقائيًا بعد 5 دقائق |

```bash
# تحقق من قائمة وحالة المفاتيح المسجلة
curl -H "Authorization: Bearer رمز-المسؤول" http://localhost:56243/admin/keys

# إعادة تعيين عدادات استخدام المفاتيح
curl -X POST -H "Authorization: Bearer رمز-المسؤول" http://localhost:56243/admin/keys/reset
```

### عندما يظهر الوكيل كـ "غير متصل"

"غير متصل" يعني أن عملية البروكسي لا ترسل إشارات (heartbeat) للخزنة. **لا يعني أن الإعدادات لم يتم حفظها.** يجب أن يكون البروكسي قيد التشغيل مع عنوان ورمز خادم الخزنة للانتقال إلى حالة الاتصال.

```bash
# بدء البروكسي مع تحديد عنوان خادم الخزنة والرمز ومعرف العميل
WV_VAULT_URL=http://عنوان-خادم-الخزنة:56243 \
WV_VAULT_TOKEN=رمز-العميل \
WV_VAULT_CLIENT_ID=معرف-العميل \
wall-vault proxy
```

إذا نجح الاتصال، ستتغير الحالة إلى 🟢 قيد التشغيل في لوحة التحكم خلال حوالي 20 ثانية.

### عندما لا يعمل الاتصال بـ Ollama

Ollama هو برنامج يشغل الذكاء الاصطناعي مباشرة على جهازك. أولاً، تحقق من أن Ollama نشط.

```bash
curl http://localhost:11434/api/tags   # إذا ظهرت قائمة النماذج، فهو طبيعي
export OLLAMA_URL=http://192.168.x.x:11434   # إذا كان يعمل على كمبيوتر آخر
```

> ⚠️ إذا لم يستجب Ollama، ابدأه أولاً بأمر `ollama serve`.

> ⚠️ **النماذج الكبيرة بطيئة في الاستجابة**: النماذج الكبيرة مثل `qwen3.5:35b`، `deepseek-r1` قد تستغرق عدة دقائق لتوليد استجابة. حتى لو بدا أنه لا توجد استجابة، قد تكون المعالجة جارية بشكل طبيعي — انتظر.

## التغييرات الأخيرة (v0.1.16 ~ v0.1.24)

### v0.1.24 (2026-04-06)

### v0.1.23 (2026-04-06)
- **إصلاح تغيير نموذج Ollama**: تم إصلاح المشكلة حيث لم يكن تغيير نموذج Ollama في لوحة تحكم الخزنة ينعكس على البروكسي. سابقًا كان يستخدم فقط متغير البيئة (`OLLAMA_MODEL`)، الآن تُعطى الأولوية لإعدادات الخزنة.
- **إشارة مرور تلقائية للخدمة المحلية**: يتم تفعيل Ollama وLM Studio وvLLM تلقائيًا عند إمكانية الاتصال وتعطيلها عند الانقطاع. تعمل بنفس طريقة التبديل التلقائي القائم على المفاتيح لخدمات السحابة.

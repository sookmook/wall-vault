## wall-vault คืออะไร?

**wall-vault = AI proxy + ตู้นิรภัย API key สำหรับ OpenClaw**

การใช้บริการ AI ต้องมี **API key** ซึ่ง API key เปรียบเหมือน **บัตรผ่านดิจิทัล** ที่พิสูจน์ว่า "บุคคลนี้มีสิทธิ์ใช้บริการนี้" แต่บัตรผ่านเหล่านี้มีจำนวนครั้งใช้งานจำกัดต่อวัน และอาจรั่วไหลได้หากจัดการไม่ดี

wall-vault เก็บบัตรผ่านเหล่านี้ในตู้นิรภัยที่ปลอดภัย และทำหน้าที่เป็น **proxy (ตัวกลาง)** ระหว่าง OpenClaw กับบริการ AI พูดง่ายๆ คือ OpenClaw เพียงเชื่อมต่อกับ wall-vault เท่านั้น ส่วนเรื่องซับซ้อนอื่นๆ wall-vault จัดการให้ทั้งหมด

ปัญหาที่ wall-vault แก้ไขให้:

- **หมุนเวียน API key อัตโนมัติ**: เมื่อ key หนึ่งใช้ครบโควตาหรือถูกบล็อกชั่วคราว (cooldown) จะเปลี่ยนไปใช้ key ถัดไปอย่างเงียบๆ OpenClaw ทำงานต่อได้โดยไม่หยุดชะงัก
- **เปลี่ยนบริการอัตโนมัติ (fallback)**: หาก Google ไม่ตอบ จะสลับไป OpenRouter โดยอัตโนมัติ หากนั่นก็ไม่ได้ จะสลับไป Ollama/LM Studio/vLLM (AI ในเครื่อง) ที่ติดตั้งในคอมพิวเตอร์ของคุณ เซสชันไม่ขาดตอน เมื่อบริการเดิมกลับมาใช้ได้ จะกลับไปใช้บริการเดิมอัตโนมัติตั้งแต่คำขอถัดไป (v0.1.18+, LM Studio/vLLM: v0.1.21+)
- **ซิงค์แบบเรียลไทม์ (SSE)**: เมื่อคุณเปลี่ยนโมเดลในแดชบอร์ดตู้นิรภัย การเปลี่ยนแปลงจะสะท้อนบนหน้าจอ OpenClaw ภายใน 1-3 วินาที SSE (Server-Sent Events) คือเทคโนโลยีที่เซิร์ฟเวอร์ส่งอัปเดตแบบเรียลไทม์ไปยังไคลเอนต์
- **การแจ้งเตือนแบบเรียลไทม์**: เหตุการณ์เช่น key หมดหรือบริการล่ม จะแสดงทันทีที่ด้านล่างของ TUI (หน้าจอเทอร์มินัล) ของ OpenClaw

> 💡 **Claude Code, Cursor, VS Code** ก็เชื่อมต่อได้เช่นกัน แต่จุดประสงค์หลักของ wall-vault คือใช้คู่กับ OpenClaw

```
OpenClaw (หน้าจอ TUI เทอร์มินัล)
        │
        ▼
  wall-vault proxy (:56244)   ← จัดการ key, routing, fallback, event
        │
        ├─ Google Gemini API
        ├─ OpenRouter API (340+ โมเดล)
        ├─ Ollama / LM Studio / vLLM (คอมฯ ในเครื่อง, ทางเลือกสุดท้าย)
        └─ OpenAI / Anthropic API
```

## การติดตั้ง

### Linux / macOS

เปิดเทอร์มินัลแล้ววางคำสั่งด้านล่าง

```bash
# Linux (PC ทั่วไป, เซิร์ฟเวอร์ — amd64)
curl -L https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-linux-amd64 \
  -o ~/.local/bin/wall-vault && chmod +x ~/.local/bin/wall-vault

# macOS Apple Silicon (M1/M2/M3 Mac)
curl -L https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-darwin-arm64 \
  -o /usr/local/bin/wall-vault && chmod +x /usr/local/bin/wall-vault
```

- `curl -L ...` — ดาวน์โหลดไฟล์จากอินเทอร์เน็ต
- `chmod +x` — ทำให้ไฟล์ที่ดาวน์โหลด "สามารถรันได้" ถ้าข้ามขั้นตอนนี้จะเกิด error "ไม่มีสิทธิ์"

### Windows

เปิด PowerShell (ในฐานะผู้ดูแลระบบ) แล้วรันคำสั่งด้านล่าง

```powershell
# ดาวน์โหลด
Invoke-WebRequest -Uri `
  "https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-windows-amd64.exe" `
  -OutFile "$env:LOCALAPPDATA\Programs\wall-vault\wall-vault.exe"

# เพิ่มลง PATH (มีผลหลังรีสตาร์ท PowerShell)
$env:PATH += ";$env:LOCALAPPDATA\Programs\wall-vault"
```

> 💡 **PATH คืออะไร?** คือรายการโฟลเดอร์ที่คอมพิวเตอร์ค้นหาคำสั่ง เมื่อเพิ่มลง PATH แล้ว คุณสามารถพิมพ์ `wall-vault` รันได้จากโฟลเดอร์ใดก็ได้

### Build จากซอร์สโค้ด (สำหรับนักพัฒนา)

ใช้ได้เฉพาะเมื่อติดตั้งสภาพแวดล้อมพัฒนาภาษา Go แล้ว

```bash
git clone https://github.com/sookmook/wall-vault
cd wall-vault
make build       # bin/wall-vault (เวอร์ชัน: v0.1.25.YYYYMMDD.HHmmss)
make install     # ~/.local/bin/wall-vault
```

> 💡 **เวอร์ชันพร้อม timestamp ของ build**: เมื่อ build ด้วย `make build` เวอร์ชันจะถูกสร้างอัตโนมัติในรูปแบบ เช่น `v0.1.25.20260408.022325` ที่รวมวันที่และเวลา ถ้า build ตรงๆ ด้วย `go build ./...` เวอร์ชันจะแสดงเป็น `"dev"` เท่านั้น

## เริ่มต้นใช้งานครั้งแรก

### รันวิซาร์ดตั้งค่า

หลังติดตั้ง จำเป็นต้องรัน **วิซาร์ดตั้งค่า** ด้วยคำสั่งด้านล่างในครั้งแรก วิซาร์ดจะแนะนำคุณโดยถามรายการที่จำเป็นทีละรายการ

```bash
wall-vault setup
```

ขั้นตอนที่วิซาร์ดดำเนินการ:

```
1. เลือกภาษา (10 ภาษารวมถึงภาษาเกาหลี)
2. เลือกธีม (light / dark / gold / cherry / ocean)
3. โหมดทำงาน — ใช้คนเดียว (standalone) หรือใช้ร่วมกันหลายเครื่อง (distributed)
4. ชื่อบอท — ชื่อที่จะแสดงบนแดชบอร์ด
5. ตั้งค่าพอร์ต — ค่าเริ่มต้น: proxy 56244, ตู้นิรภัย 56243 (กด Enter ถ้าไม่ต้องเปลี่ยน)
6. เลือกบริการ AI — Google / OpenRouter / Ollama / LM Studio / vLLM
7. ตั้งค่าตัวกรองเครื่องมือความปลอดภัย
8. ตั้งค่าโทเค็นผู้ดูแล — รหัสผ่านสำหรับล็อกฟังก์ชันจัดการแดชบอร์ด สร้างอัตโนมัติได้
9. ตั้งค่ารหัสผ่านเข้ารหัส API key — เพื่อเก็บ key ได้ปลอดภัยยิ่งขึ้น (ไม่บังคับ)
10. ตำแหน่งบันทึกไฟล์การตั้งค่า
```

> ⚠️ **โปรดจดจำโทเค็นผู้ดูแล** จำเป็นต้องใช้ภายหลังเมื่อเพิ่ม key หรือเปลี่ยนการตั้งค่าในแดชบอร์ด ถ้าลืม จะต้องแก้ไขไฟล์การตั้งค่าด้วยตนเอง

เมื่อวิซาร์ดเสร็จสมบูรณ์ ไฟล์การตั้งค่า `wall-vault.yaml` จะถูกสร้างอัตโนมัติ

### การรัน

```bash
wall-vault start
```

เซิร์ฟเวอร์สองตัวเริ่มพร้อมกัน:

- **Proxy** (`http://localhost:56244`) — ตัวกลางที่เชื่อม OpenClaw กับบริการ AI
- **ตู้นิรภัยกุญแจ** (`http://localhost:56243`) — จัดการ API key และแดชบอร์ดเว็บ

เปิด `http://localhost:56243` ในเบราว์เซอร์เพื่อดูแดชบอร์ดได้ทันที

## การลงทะเบียน API key

มี 4 วิธีลงทะเบียน API key **สำหรับผู้เริ่มต้น แนะนำวิธีที่ 1 (environment variable)**

### วิธีที่ 1: Environment variable (แนะนำ — ง่ายที่สุด)

Environment variable คือ **ค่าที่ตั้งไว้ล่วงหน้า** ที่โปรแกรมอ่านเมื่อเริ่มทำงาน พิมพ์ในเทอร์มินัลตามด้านล่าง

```bash
# ลงทะเบียน key Google Gemini
export WV_KEY_GOOGLE=AIzaSy...

# ลงทะเบียน key OpenRouter
export WV_KEY_OPENROUTER=sk-or-v1-...

# รันหลังลงทะเบียน
wall-vault start
```

ถ้ามีหลาย key ให้เชื่อมด้วยเครื่องหมายจุลภาค(,) wall-vault จะใช้สลับกันอัตโนมัติ (round robin):

```bash
export WV_KEY_GOOGLE=AIzaSy...,AIzaSy...,AIzaSy...
```

> 💡 **เคล็ดลับ**: คำสั่ง `export` ใช้ได้เฉพาะเซสชันเทอร์มินัลปัจจุบัน เพื่อให้คงอยู่หลังรีสตาร์ทคอมพิวเตอร์ ให้เพิ่มบรรทัดด้านบนลงในไฟล์ `~/.bashrc` หรือ `~/.zshrc`

### วิธีที่ 2: UI แดชบอร์ด (คลิกเมาส์)

1. เข้า `http://localhost:56243` ในเบราว์เซอร์
2. คลิกปุ่ม `[+ เพิ่ม]` ในการ์ด **🔑 API key** ด้านบน
3. กรอกประเภทบริการ, ค่า key, ป้ายกำกับ (ชื่ออ้างอิง) และโควตารายวัน แล้วบันทึก

### วิธีที่ 3: REST API (สำหรับ automation/script)

REST API คือวิธีแลกเปลี่ยนข้อมูลระหว่างโปรแกรมผ่าน HTTP เหมาะสำหรับลงทะเบียนอัตโนมัติผ่าน script

```bash
curl -X POST http://localhost:56243/admin/keys \
  -H "Authorization: Bearer โทเค็นผู้ดูแล" \
  -H "Content-Type: application/json" \
  -d '{
    "service": "google",
    "key": "AIzaSy...",
    "label": "key หลัก",
    "daily_limit": 1000
  }'
```

### วิธีที่ 4: proxy flag (สำหรับทดสอบเร็ว)

ใช้สำหรับทดสอบชั่วคราวโดยใส่ key โดยไม่ลงทะเบียนอย่างเป็นทางการ หายไปเมื่อปิดโปรแกรม

```bash
wall-vault proxy --key-google=AIzaSy... --key-openrouter=sk-or-...
```

## วิธีใช้งาน proxy

### ใช้กับ OpenClaw (จุดประสงค์หลัก)

วิธีตั้งค่า OpenClaw ให้เชื่อมต่อกับบริการ AI ผ่าน wall-vault

เปิดไฟล์ `~/.openclaw/openclaw.json` แล้วเพิ่มเนื้อหาด้านล่าง:

```json5
// ~/.openclaw/openclaw.json
{
  models: {
    providers: {
      "wall-vault": {
        baseUrl: "http://localhost:56244/v1",
        apiKey: "your-agent-token",   // โทเค็นเอเจนต์ของ vault
        api: "openai-completions",
        models: [
          { id: "wall-vault/gemini-2.5-flash" },
          { id: "wall-vault/gemini-2.5-pro" },
          { id: "wall-vault/hunter-alpha" },    // context 1M ฟรี
          { id: "wall-vault/claude-opus-4-6" }
        ]
      }
    }
  }
}
```

> 💡 **วิธีง่ายกว่า**: คลิกปุ่ม **🦞 คัดลอกการตั้งค่า OpenClaw** ในการ์ดเอเจนต์บนแดชบอร์ด snippet ที่มีโทเค็นและที่อยู่กรอกแล้วจะถูกคัดลอกไปยังคลิปบอร์ด แค่วางก็เสร็จ

**`wall-vault/` หน้าชื่อโมเดลชี้ไปที่ไหน?**

wall-vault กำหนดอัตโนมัติว่าจะส่งคำขอไปยังบริการ AI ใดตามชื่อโมเดล:

| รูปแบบโมเดล | บริการที่เชื่อมต่อ |
|----------|--------------|
| `wall-vault/gemini-*` | เชื่อมต่อตรงกับ Google Gemini |
| `wall-vault/gpt-*`, `wall-vault/o3`, `wall-vault/o4*` | เชื่อมต่อตรงกับ OpenAI |
| `wall-vault/claude-*` | เชื่อมต่อกับ Anthropic ผ่าน OpenRouter |
| `wall-vault/hunter-alpha`, `wall-vault/healer-alpha` | OpenRouter (context 1 ล้านโทเค็นฟรี) |
| `wall-vault/kimi-*`, `wall-vault/glm-*`, `wall-vault/deepseek-*` | เชื่อมต่อกับ OpenRouter |
| `google/ชื่อโมเดล`, `openai/ชื่อโมเดล`, `anthropic/ชื่อโมเดล` ฯลฯ | เชื่อมต่อตรงกับบริการนั้นๆ |
| `custom/google/ชื่อโมเดล`, `custom/openai/ชื่อโมเดล` ฯลฯ | ลบส่วน `custom/` แล้ว route ใหม่ |
| `ชื่อโมเดล:cloud` | ลบส่วน `:cloud` แล้วเชื่อมต่อผ่าน OpenRouter |

> 💡 **Context คืออะไร?** คือปริมาณบทสนทนาที่ AI จำได้ในครั้งเดียว 1M (หนึ่งล้านโทเค็น) หมายความว่าสามารถประมวลผลบทสนทนาที่ยาวมากหรือเอกสารขนาดใหญ่ได้ในครั้งเดียว

### เชื่อมต่อตรงในรูปแบบ Gemini API (ใช้งานร่วมกับเครื่องมือที่มีอยู่)

ถ้าคุณมีเครื่องมือที่ใช้ Google Gemini API โดยตรง แค่เปลี่ยนที่อยู่เป็น wall-vault:

```bash
export ANTHROPIC_BASE_URL=http://localhost:56244/google
```

หรือถ้าเครื่องมือระบุ URL โดยตรงได้:

```
http://localhost:56244/google/v1beta/models/gemini-2.5-flash:generateContent
```

### ใช้กับ OpenAI SDK (Python)

คุณยังสามารถเชื่อมต่อ wall-vault ในโค้ด Python ที่ใช้ AI ได้ แค่เปลี่ยน `base_url`:

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:56244/v1",
    api_key="not-needed"  # API key จัดการโดย wall-vault
)

response = client.chat.completions.create(
    model="google/gemini-2.5-flash",   # รูปแบบ provider/model
    messages=[{"role": "user", "content": "สวัสดี"}]
)
```

### เปลี่ยนโมเดลระหว่างรัน

เพื่อเปลี่ยนโมเดล AI ขณะ wall-vault กำลังทำงาน:

```bash
# เปลี่ยนโมเดลโดยส่งคำขอตรงไปที่ proxy
curl -X PUT http://localhost:56244/api/config/model \
  -H "Content-Type: application/json" \
  -d '{"service": "openrouter", "model": "anthropic/claude-3.5-sonnet"}'

# ในโหมดกระจาย (มัลติบอท) เปลี่ยนที่เซิร์ฟเวอร์ตู้นิรภัย → สะท้อนทันทีผ่าน SSE
curl -X PUT http://localhost:56243/admin/clients/id-บอท-ของฉัน \
  -H "Authorization: Bearer โทเค็นผู้ดูแล" \
  -H "Content-Type: application/json" \
  -d '{"default_service": "google", "default_model": "gemini-2.5-pro"}'
```

### ตรวจสอบรายการโมเดลที่ใช้ได้

```bash
# ดูรายการทั้งหมด
curl http://localhost:56244/api/models | python3 -m json.tool

# ดูเฉพาะโมเดล Google
curl "http://localhost:56244/api/models?service=google"

# ค้นหาตามชื่อ (เช่น: โมเดลที่มี "claude")
curl "http://localhost:56244/api/models?q=claude"
```

**สรุปโมเดลหลักตามบริการ:**

| บริการ | โมเดลหลัก |

## แดชบอร์ดตู้นิรภัยกุญแจ

เข้า `http://localhost:56243` ในเบราว์เซอร์เพื่อดูแดชบอร์ด

**เลย์เอาต์หน้าจอ:**
- **แถบด้านบนคงที่ (topbar)**: โลโก้, ตัวเลือกภาษา/ธีม, ตัวแสดงสถานะการเชื่อมต่อ SSE
- **กริดการ์ด**: การ์ดเอเจนต์ บริการ และ API key จัดเรียงในรูปแบบไทล์

### การ์ด API key

การ์ดสำหรับจัดการ API key ที่ลงทะเบียนไว้ได้ในมุมมองเดียว

- แสดงรายการ key จำแนกตามบริการ
- `today_usage`: จำนวนโทเค็น (ตัวอักษรที่ AI อ่านและเขียน) ที่ประมวลผลสำเร็จวันนี้
- `today_attempts`: จำนวนการเรียกทั้งหมดวันนี้ (รวมสำเร็จ + ล้มเหลว)
- ลงทะเบียน key ใหม่ด้วยปุ่ม `[+ เพิ่ม]` และลบด้วย `✕`

> 💡 **โทเค็น (token) คืออะไร?** คือหน่วยที่ AI ใช้ประมวลผลข้อความ โดยประมาณเท่ากับคำภาษาอังกฤษหนึ่งคำ หรือตัวอักษรเกาหลี 1-2 ตัว ค่าบริการ API มักคำนวณจากจำนวนโทเค็น

### การ์ดเอเจนต์

การ์ดที่แสดงสถานะของบอท (เอเจนต์) ที่เชื่อมต่อกับ wall-vault proxy

**สถานะการเชื่อมต่อแสดง 4 ระดับ:**

| ตัวแสดง | สถานะ | ความหมาย |
|------|------|------|
| 🟢 | ทำงานอยู่ | Proxy ทำงานปกติ |
| 🟡 | หน่วง | ตอบสนองแต่ช้า |
| 🔴 | ออฟไลน์ | Proxy ไม่ตอบสนอง |
| ⚫ | ไม่เชื่อมต่อ/ไม่ใช้งาน | Proxy ไม่เคยเชื่อมต่อตู้นิรภัยหรือถูกปิดใช้งาน |

**คู่มือปุ่มด้านล่างการ์ดเอเจนต์:**

เมื่อลงทะเบียนเอเจนต์ ระบุ **ประเภทเอเจนต์** และปุ่มอำนวยความสะดวกที่เกี่ยวข้องจะปรากฏอัตโนมัติ

#### 🔘 ปุ่มคัดลอกการตั้งค่า — สร้างการตั้งค่าการเชื่อมต่ออัตโนมัติ

เมื่อคลิกปุ่ม snippet การตั้งค่าที่มีโทเค็นเอเจนต์ ที่อยู่ proxy และข้อมูลโมเดลกรอกแล้วจะถูกคัดลอกไปยังคลิปบอร์ด แค่วางเนื้อหาที่คัดลอกในตำแหน่งที่ระบุในตารางด้านล่างเพื่อเสร็จสิ้นการตั้งค่าการเชื่อมต่อ

| ปุ่ม | ประเภทเอเจนต์ | ที่วาง |

> ⚠️ **Continue เวอร์ชันล่าสุดใช้ `config.yaml`** ถ้ามี `config.yaml` อยู่ `config.json` จะถูกละเว้นทั้งหมด ต้องวางใน `config.yaml` เท่านั้น

> ⚠️ **เมื่อคัดลอกไปยังคลิปบอร์ดไม่ทำงาน**: นโยบายความปลอดภัยของเบราว์เซอร์อาจบล็อกการคัดลอก ถ้ามีกล่องข้อความ popup ให้เลือกทั้งหมดด้วย Ctrl+A แล้วคัดลอกด้วย Ctrl+C

#### 🔀 ลากและวางการ์ดเพื่อจัดเรียงใหม่ (v0.1.17, ปรับปรุง v0.1.25)

คุณสามารถ **ลาก** การ์ดเอเจนต์ในแดชบอร์ดเพื่อจัดเรียงตามลำดับที่ต้องการ

1. คลิกค้างที่บริเวณ **สัญญาณไฟ (●)** มุมซ้ายบนของการ์ดแล้วลาก
2. ปล่อยบนการ์ดในตำแหน่งที่ต้องการ ลำดับจะเปลี่ยน

> 💡 ส่วนเนื้อหาของการ์ด (ช่องกรอก, ปุ่ม ฯลฯ) ลากไม่ได้ จับได้เฉพาะจากบริเวณสัญญาณไฟเท่านั้น

#### 🟠 การตรวจจับกระบวนการเอเจนต์ (v0.1.25)

เมื่อ proxy ทำงานปกติแต่กระบวนการเอเจนต์ในเครื่อง (NanoClaw, OpenClaw) หยุดทำงาน สัญญาณไฟของการ์ดจะเปลี่ยนเป็น **สีส้ม (กะพริบ)** และแสดงข้อความ "กระบวนการเอเจนต์หยุด"

- 🟢 เขียว: Proxy + เอเจนต์ปกติ
- 🟠 ส้ม (กะพริบ): Proxy ปกติ, เอเจนต์หยุด
- 🔴 แดง: Proxy ออฟไลน์
3. ลำดับที่เปลี่ยนจะ **บันทึกทันทีที่เซิร์ฟเวอร์** และคงอยู่แม้รีเฟรชหน้า

> 💡 อุปกรณ์ระบบสัมผัส (มือถือ/แท็บเล็ต) ยังไม่รองรับ ใช้เบราว์เซอร์เดสก์ท็อป

#### 🔄 ซิงค์โมเดลสองทาง (v0.1.16)

เมื่อคุณเปลี่ยนโมเดลของเอเจนต์ในแดชบอร์ดตู้นิรภัย การตั้งค่าในเครื่องของเอเจนต์จะอัปเดตอัตโนมัติ

**สำหรับ Cline:**
- เปลี่ยนโมเดลในตู้นิรภัย → SSE event → proxy อัปเดต field โมเดลใน `globalState.json`
- Field ที่อัปเดต: `actModeOpenAiModelId`, `planModeOpenAiModelId`, `openAiModelId`
- ไม่เปลี่ยน `openAiBaseUrl` และ API key
- **ต้อง reload VS Code (`Ctrl+Alt+R` หรือ `Ctrl+Shift+P` → `Developer: Reload Window`)**
  - Cline ไม่อ่านไฟล์การตั้งค่าใหม่ระหว่างทำงาน

**สำหรับ Claude Code:**
- เปลี่ยนโมเดลในตู้นิรภัย → SSE event → proxy อัปเดต field `model` ใน `settings.json`
- ค้นหาอัตโนมัติทั้ง path WSL และ Windows (`~/.claude/`, `/mnt/c/Users/*/.claude/`)

**ทิศทางกลับ (เอเจนต์ → ตู้นิรภัย):**
- เมื่อเอเจนต์ (Cline, Claude Code ฯลฯ) ส่งคำขอไปยัง proxy, proxy จะรวมข้อมูลบริการ/โมเดลของไคลเอนต์นั้นใน heartbeat
- บริการ/โมเดลที่ใช้อยู่ปัจจุบันจะแสดงแบบเรียลไทม์ในการ์ดเอเจนต์บนแดชบอร์ดตู้นิรภัย

> 💡 **ประเด็นสำคัญ**: Proxy ระบุเอเจนต์จากโทเค็น Authorization ในคำขอ และ route อัตโนมัติไปยังบริการ/โมเดลที่ตั้งค่าในตู้นิรภัย แม้ Cline หรือ Claude Code จะส่งชื่อโมเดลที่แตกต่าง proxy จะ override ด้วยการตั้งค่าตู้นิรภัย

### ใช้ Cline ใน VS Code — คู่มือละเอียด

#### ขั้นตอน 1: ติดตั้ง Cline

ติดตั้ง **Cline** (ID: `saoudrizwan.claude-dev`) จาก marketplace ส่วนขยาย VS Code

#### ขั้นตอน 2: ลงทะเบียนเอเจนต์ในตู้นิรภัย

1. เปิดแดชบอร์ดตู้นิรภัย (`http://IP-ตู้นิรภัย:56243`)
2. ในส่วน **เอเจนต์** คลิก **+ เพิ่ม**
3. กรอกดังนี้:

| ฟิลด์ | ค่า | คำอธิบาย |
|------|----|------|
| ID | `my_cline` | ตัวระบุไม่ซ้ำกัน (ภาษาอังกฤษ, ไม่มีเว้นวรรค) |
| ชื่อ | `My Cline` | ชื่อที่แสดงบนแดชบอร์ด |
| ประเภทเอเจนต์ | `cline` | ← ต้องเลือก `cline` |
| บริการ | เลือกบริการที่ต้องการ (เช่น: `google`) | |
| โมเดล | กรอกโมเดลที่ต้องการ (เช่น: `gemini-2.5-flash`) | |

4. คลิก **บันทึก** และโทเค็นจะถูกสร้างอัตโนมัติ

#### ขั้นตอน 3: เชื่อมต่อกับ Cline

**วิธี A — นำไปใช้อัตโนมัติ (แนะนำ)**

1. ตรวจสอบว่า **proxy** wall-vault ทำงานอยู่บนเครื่องนั้น (`localhost:56244`)
2. คลิกปุ่ม **⚡ นำไปใช้การตั้งค่า Cline** ในการ์ดเอเจนต์บนแดชบอร์ด
3. ถ้าแจ้งเตือน "นำไปใช้การตั้งค่าสำเร็จ!" ปรากฏ แสดงว่าสำเร็จ
4. Reload VS Code (`Ctrl+Alt+R`)

**วิธี B — ตั้งค่าเอง**

เปิดการตั้งค่า (⚙️) ใน sidebar ของ Cline:
- **API Provider**: `OpenAI Compatible`
- **Base URL**: `http://ที่อยู่-proxy:56244/v1`
  - ถ้าเครื่องเดียวกัน: `http://localhost:56244/v1`
  - ถ้าเครื่องอื่น (เช่น เซิร์ฟเวอร์ Mini): `http://192.168.1.20:56244/v1`
- **API Key**: โทเค็นที่ออกจากตู้นิรภัย (คัดลอกจากการ์ดเอเจนต์)
- **Model ID**: โมเดลที่ตั้งค่าในตู้นิรภัย (เช่น: `gemini-2.5-flash`)

#### ขั้นตอน 4: ตรวจสอบ

ส่งข้อความใดๆ ในแชท Cline ถ้าปกติ:
- **จุดเขียว (● ทำงานอยู่)** จะปรากฏในการ์ดเอเจนต์ที่เกี่ยวข้องบนแดชบอร์ดตู้นิรภัย
- บริการ/โมเดลปัจจุบันจะแสดงบนการ์ด (เช่น: `google / gemini-2.5-flash`)

#### เปลี่ยนโมเดล

เมื่อต้องการเปลี่ยนโมเดลของ Cline ให้เปลี่ยนที่ **แดชบอร์ดตู้นิรภัย**:

1. เปลี่ยน dropdown บริการ/โมเดลในการ์ดเอเจนต์
2. คลิก **นำไปใช้**
3. Reload VS Code (`Ctrl+Alt+R`) — ชื่อโมเดลใน footer ของ Cline จะอัปเดต
4. โมเดลใหม่จะถูกใช้ตั้งแต่คำขอถัดไป

> 💡 ในความเป็นจริง proxy ระบุคำขอของ Cline ด้วยโทเค็นและ route ไปยังโมเดลที่ตั้งค่าในตู้นิรภัย แม้ไม่ reload VS Code **โมเดลที่ใช้จริงจะเปลี่ยนทันที** — reload เพียงเพื่ออัปเดตการแสดงโมเดลใน UI ของ Cline

#### การตรวจจับการตัดการเชื่อมต่อ

เมื่อปิด VS Code การ์ดเอเจนต์ในแดชบอร์ดตู้นิรภัยจะเปลี่ยนเป็นสีเหลือง (หน่วง) หลังจากประมาณ **90 วินาที** และเป็นสีแดง (ออฟไลน์) หลังจาก **3 นาที** (ตั้งแต่ v0.1.18 การตรวจจับออฟไลน์เร็วขึ้นด้วยการตรวจสอบสถานะทุก 15 วินาที)

#### การแก้ไขปัญหา

| อาการ | สาเหตุ | วิธีแก้ |
|------|------|------|
| Error "เชื่อมต่อล้มเหลว" ใน Cline | Proxy ไม่ทำงานหรือที่อยู่ผิด | ตรวจสอบ proxy ด้วย `curl http://localhost:56244/health` |

2. ลงทะเบียนบริการ systemd user อัตโนมัติ
3. เริ่มบริการและเชื่อมต่อตู้นิรภัยอัตโนมัติ

> 💡 สคริปต์มีโทเค็นของเอเจนต์นี้และที่อยู่เซิร์ฟเวอร์ตู้นิรภัยกรอกอยู่แล้ว สามารถรันได้ทันทีหลังวางโดยไม่ต้องแก้ไข

### การ์ดบริการ

การ์ดสำหรับเปิด/ปิดหรือตั้งค่าบริการ AI ที่จะใช้

- สวิตช์เปิด/ปิดสำหรับแต่ละบริการ
- กรอกที่อยู่เซิร์ฟเวอร์ AI ในเครื่อง (Ollama, LM Studio, vLLM ฯลฯ ที่ทำงานในคอมพิวเตอร์ของคุณ) แล้วโมเดลที่ใช้ได้จะถูกตรวจจับอัตโนมัติ
- **ตัวแสดงสถานะการเชื่อมต่อบริการในเครื่อง**: จุด ● ข้างชื่อบริการเป็น **เขียว** ถ้าเชื่อมต่อ และ **เทา** ถ้าไม่เชื่อมต่อ
- **สัญญาณไฟอัตโนมัติสำหรับบริการในเครื่อง** (v0.1.23+): บริการในเครื่อง (Ollama, LM Studio, vLLM) เปิด/ปิดอัตโนมัติตามความสามารถในการเชื่อมต่อ เมื่อเปิดบริการ ภายใน 15 วินาทีจุด ● จะเขียวและ checkbox จะเปิด เมื่อปิดจะปิดอัตโนมัติ ทำงานเหมือนกับ toggle อัตโนมัติของบริการคลาวด์ (Google, OpenRouter ฯลฯ) ตาม API key

> 💡 **ถ้าบริการในเครื่องทำงานบนคอมพิวเตอร์อื่น**: กรอก IP ของคอมพิวเตอร์นั้นในช่อง URL บริการ เช่น: `http://192.168.1.20:11434` (Ollama), `http://192.168.1.20:1234` (LM Studio) ถ้าบริการ bind เฉพาะ `127.0.0.1` ไม่ใช่ `0.0.0.0` การเข้าถึงผ่าน IP ภายนอกจะไม่ทำงาน ให้ตรวจสอบที่อยู่ binding ในการตั้งค่าบริการ

### การกรอกโทเค็นผู้ดูแล

เมื่อพยายามใช้ฟังก์ชันสำคัญในแดชบอร์ด เช่น เพิ่ม/ลบ key จะมี popup ให้กรอกโทเค็นผู้ดูแล กรอกโทเค็นที่ตั้งค่าในวิซาร์ดตั้งค่า เมื่อกรอกแล้ว จะใช้ได้จนกว่าจะปิดเบราว์เซอร์

> ⚠️ **ถ้าการยืนยันตัวตนล้มเหลวเกิน 10 ครั้งใน 15 นาที IP นั้นจะถูกบล็อกชั่วคราว** ถ้าลืมโทเค็น ให้ตรวจสอบรายการ `admin_token` ในไฟล์ `wall-vault.yaml`

## โหมดกระจาย (มัลติบอท)

การตั้งค่าที่ **แชร์ตู้นิรภัยกุญแจเดียว** เมื่อเปิดใช้งาน OpenClaw บนหลายคอมพิวเตอร์พร้อมกัน สะดวกเพราะจัดการ key ที่เดียว

### ตัวอย่างการตั้งค่า

```
[เซิร์ฟเวอร์ตู้นิรภัยกุญแจ]
  wall-vault vault    (ตู้นิรภัยกุญแจ :56243, แดชบอร์ด)

[WSL อัลฟ่า]            [Raspberry Pi แกมม่า]    [Mac Mini ในเครื่อง]
  wall-vault proxy      wall-vault proxy        wall-vault proxy
  openclaw TUI          openclaw TUI            openclaw TUI
  ↕ ซิงค์ SSE            ↕ ซิงค์ SSE              ↕ ซิงค์ SSE
```

บอททุกตัวชี้ไปยังเซิร์ฟเวอร์ตู้นิรภัยกลาง เมื่อเปลี่ยนโมเดลหรือเพิ่ม key ในตู้นิรภัย จะสะท้อนทันทีไปยังบอททุกตัว

### ขั้นตอน 1: เริ่มเซิร์ฟเวอร์ตู้นิรภัยกุญแจ

รันบนคอมพิวเตอร์ที่จะใช้เป็นเซิร์ฟเวอร์ตู้นิรภัย:

```bash
wall-vault vault
```

### ขั้นตอน 2: ลงทะเบียนแต่ละบอท (ไคลเอนต์)

ลงทะเบียนข้อมูลแต่ละบอทที่จะเชื่อมต่อกับเซิร์ฟเวอร์ตู้นิรภัยล่วงหน้า:

```bash
curl -X POST http://localhost:56243/admin/clients \
  -H "Authorization: Bearer โทเค็นผู้ดูแล" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "botA",
    "name": "บอทA",
    "token": "bota-secret",
    "default_service": "google",
    "default_model": "gemini-2.5-flash"
  }'
```

### ขั้นตอน 3: เริ่ม proxy บนคอมพิวเตอร์แต่ละบอท

รัน proxy บนแต่ละคอมพิวเตอร์ที่ติดตั้งบอทพร้อมระบุที่อยู่เซิร์ฟเวอร์ตู้นิรภัยและโทเค็น:

```bash
WV_VAULT_URL=http://192.168.x.x:56243 \
WV_VAULT_TOKEN=bota-secret \
WV_VAULT_CLIENT_ID=botA \
wall-vault proxy
```

> 💡 **แทนที่ `192.168.x.x`** ด้วย IP ภายในจริงของคอมพิวเตอร์เซิร์ฟเวอร์ตู้นิรภัย ตรวจสอบได้จากการตั้งค่าเราเตอร์หรือคำสั่ง `ip addr`

## การตั้งค่าเริ่มอัตโนมัติ

ถ้ารู้สึกยุ่งยากที่ต้องเริ่ม wall-vault เองทุกครั้งที่รีสตาร์ทคอมพิวเตอร์ ให้ลงทะเบียนเป็นบริการระบบ เมื่อลงทะเบียนแล้วจะเริ่มอัตโนมัติเมื่อบูต

### Linux — systemd (Linux ส่วนใหญ่)

systemd คือระบบสำหรับเริ่มและจัดการโปรแกรมอัตโนมัติใน Linux:

```bash
wall-vault doctor deploy
systemctl --user daemon-reload
systemctl --user enable --now wall-vault
```

ดูล็อก:

```bash
journalctl --user -u wall-vault -f
```

### macOS — launchd

ระบบที่รับผิดชอบการรันโปรแกรมอัตโนมัติใน macOS:

```bash
wall-vault doctor deploy launchd
launchctl load ~/Library/LaunchAgents/com.wall-vault.plist
```

### Windows — NSSM

1. ดาวน์โหลด NSSM จาก [nssm.cc](https://nssm.cc/download) แล้วเพิ่มลง PATH
2. ใน PowerShell ในฐานะผู้ดูแลระบบ:

```powershell
wall-vault doctor deploy windows
```

## RTK ประหยัดโทเค็น

*(v0.1.24+)*

**RTK (เครื่องมือประหยัดโทเค็น)** บีบอัดอัตโนมัติ output ของคำสั่ง shell ที่รันโดยเอเจนต์ AI coding (เช่น Claude Code) ลดการใช้โทเค็น ตัวอย่างเช่น output 15 บรรทัดของ `git status` ถูกย่อเป็น 2 บรรทัด

### การใช้งานพื้นฐาน

```bash
# ครอบคำสั่งด้วย wall-vault rtk เพื่อกรอง output อัตโนมัติ
wall-vault rtk git status          # แสดงเฉพาะรายการไฟล์ที่เปลี่ยน
wall-vault rtk git diff HEAD~1     # บรรทัดที่เปลี่ยน + context ขั้นต่ำ
wall-vault rtk git log -10         # บรรทัดเดียวต่อ commit: hash + ข้อความ
wall-vault rtk go test ./...       # แสดงเฉพาะ test ที่ล้มเหลว
wall-vault rtk ls -la              # คำสั่งที่ไม่รองรับจะถูกตัดอัตโนมัติ
```

### คำสั่งที่รองรับและการประหยัด

| คำสั่ง | วิธีกรอง | การประหยัด |
|------|----------|--------|
| `git status` | สรุปไฟล์ที่เปลี่ยนเท่านั้น | ~87% |
| `git diff` | บรรทัดที่เปลี่ยน + 3 บรรทัด context | ~60-94% |
| `git log` | Hash + บรรทัดแรกของข้อความ | ~90% |
| `git push/pull/fetch` | ลบ progress, สรุปเท่านั้น | ~80% |
| `go test` | แสดงเฉพาะที่ล้มเหลว, นับที่ผ่าน | ~88-99% |
| `go build/vet` | แสดงเฉพาะ error | ~90% |
| คำสั่งอื่นทั้งหมด | 50 บรรทัดแรก + 50 บรรทัดสุดท้าย, สูงสุด 32KB | แปรผัน |

### Pipeline กรอง 3 ขั้นตอน

1. **ตัวกรองโครงสร้างตามคำสั่ง** — เข้าใจรูปแบบ output ของ git, go ฯลฯ และดึงเฉพาะส่วนที่มีความหมาย
2. **Post-processing ด้วย regex** — ลบรหัสสี ANSI, ลดบรรทัดว่าง, รวมบรรทัดซ้ำ
3. **Passthrough + ตัด** — คำสั่งที่ไม่รองรับเก็บเฉพาะ 50 บรรทัดแรก/สุดท้าย

### การผสานกับ Claude Code

สามารถตั้งค่าให้คำสั่ง shell ทั้งหมดผ่าน RTK อัตโนมัติโดยใช้ hook `PreToolUse` ของ Claude Code

```bash
# ติดตั้ง hook (เพิ่มอัตโนมัติใน settings.json ของ Claude Code)
wall-vault rtk hook install
```

หรือเพิ่มเองใน `~/.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "command": "wall-vault rtk rewrite"
    }]
  }
}
```

> 💡 **รักษา exit code**: RTK ส่งคืน exit code เดิมของคำสั่ง ถ้าคำสั่งล้มเหลว (exit code ≠ 0) AI ก็ตรวจจับความล้มเหลวได้ถูกต้อง

> 💡 **บังคับ output ภาษาอังกฤษ**: RTK รันคำสั่งด้วย `LC_ALL=C` เพื่อสร้าง output ภาษาอังกฤษเสมอโดยไม่ขึ้นกับการตั้งค่าภาษาระบบ ทำให้ตัวกรองทำงานถูกต้อง

## อ้างอิง environment variable

Environment variable คือวิธีส่งค่าการตั้งค่าให้โปรแกรม กรอกในเทอร์มินัลในรูปแบบ `export ชื่อตัวแปร=ค่า` หรือใส่ในไฟล์บริการเริ่มอัตโนมัติเพื่อใช้ตลอด

| ตัวแปร | คำอธิบาย | ค่าตัวอย่าง |
|------|------|---------|
| `WV_LANG` | ภาษาแดชบอร์ด | `ko`, `en`, `ja` |
| `WV_THEME` | ธีมแดชบอร์ด | `light`, `dark`, `gold` |
| `WV_KEY_GOOGLE` | Google API key (หลายตัวคั่นด้วยจุลภาค) | `AIza...,AIza...` |
| `WV_KEY_OPENROUTER` | OpenRouter API key | `sk-or-v1-...` |
| `WV_VAULT_URL` | ที่อยู่เซิร์ฟเวอร์ตู้นิรภัยในโหมดกระจาย | `http://192.168.x.x:56243` |
| `WV_VAULT_TOKEN` | โทเค็นยืนยันตัวตนไคลเอนต์ (บอท) | `my-secret-token` |
| `WV_ADMIN_TOKEN` | โทเค็นผู้ดูแล | `admin-token-here` |
| `WV_MASTER_PASS` | รหัสผ่านเข้ารหัส API key | `my-password` |
| `WV_AVATAR` | path ไฟล์อวาตาร์ (สัมพัทธ์กับ `~/.openclaw/`) | `workspace/avatars/avatar.png` |
| `OLLAMA_URL` | ที่อยู่เซิร์ฟเวอร์ Ollama ในเครื่อง | `http://192.168.x.x:11434` |

## การแก้ไขปัญหา

### เมื่อ proxy เริ่มไม่ได้

บ่อยครั้งที่พอร์ตถูกใช้โดยโปรแกรมอื่นอยู่แล้ว

```bash
ss -tlnp | grep 56244   # ตรวจสอบว่าใครใช้พอร์ต 56244
wall-vault proxy --port 8080   # เริ่มด้วยหมายเลขพอร์ตอื่น
```

### Error ของ API key (429, 402, 401, 403, 582)

| รหัส error | ความหมาย | การดำเนินการ |
|----------|------|----------|
| **429** | คำขอมากเกินไป (เกินโควตาใช้งาน) | รอสักครู่หรือเพิ่ม key อื่น |
| **402** | ต้องชำระเงินหรือเครดิตไม่พอ | เติมเครดิตในบริการที่เกี่ยวข้อง |
| **401 / 403** | Key ผิดหรือไม่มีสิทธิ์ | ตรวจสอบค่า key แล้วลงทะเบียนใหม่ |
| **582** | Gateway โหลดหนัก (cooldown 5 นาที) | จะปลดล็อกอัตโนมัติหลัง 5 นาที |

```bash
# ตรวจสอบรายการ key ที่ลงทะเบียนและสถานะ
curl -H "Authorization: Bearer โทเค็นผู้ดูแล" http://localhost:56243/admin/keys

# รีเซ็ตตัวนับการใช้งาน key
curl -X POST -H "Authorization: Bearer โทเค็นผู้ดูแล" http://localhost:56243/admin/keys/reset
```

### เมื่อเอเจนต์แสดง "ไม่เชื่อมต่อ"

"ไม่เชื่อมต่อ" หมายความว่ากระบวนการ proxy ไม่ส่งสัญญาณ (heartbeat) ไปยังตู้นิรภัย **ไม่ได้หมายความว่าการตั้งค่าไม่ได้บันทึก** Proxy ต้องทำงานพร้อมที่อยู่เซิร์ฟเวอร์ตู้นิรภัยและโทเค็นเพื่อให้สถานะการเชื่อมต่อเปลี่ยน

```bash
# เริ่ม proxy พร้อมระบุที่อยู่เซิร์ฟเวอร์ตู้นิรภัย, โทเค็น และ ID ไคลเอนต์
WV_VAULT_URL=http://ที่อยู่-เซิร์ฟเวอร์-ตู้นิรภัย:56243 \
WV_VAULT_TOKEN=โทเค็น-ไคลเอนต์ \
WV_VAULT_CLIENT_ID=ID-ไคลเอนต์ \
wall-vault proxy
```

ถ้าเชื่อมต่อสำเร็จ สถานะจะเปลี่ยนเป็น 🟢 ทำงานอยู่ในแดชบอร์ดภายในประมาณ 20 วินาที

### เมื่อเชื่อมต่อ Ollama ไม่ได้

Ollama คือโปรแกรมที่รัน AI บนคอมพิวเตอร์ของคุณโดยตรง ก่อนอื่นตรวจสอบว่า Ollama เปิดอยู่

```bash
curl http://localhost:11434/api/tags   # ถ้ารายการโมเดลแสดง แสดงว่าปกติ
export OLLAMA_URL=http://192.168.x.x:11434   # ถ้าทำงานบนคอมพิวเตอร์อื่น
```

> ⚠️ ถ้า Ollama ไม่ตอบสนอง ให้เริ่มก่อนด้วยคำสั่ง `ollama serve`

> ⚠️ **โมเดลใหญ่จะช้า**: โมเดลใหญ่เช่น `qwen3.5:35b` และ `deepseek-r1` อาจใช้เวลาหลายนาทีในการสร้างคำตอบ แม้ดูเหมือนไม่มีคำตอบ อาจกำลังประมวลผลปกติ กรุณารอ

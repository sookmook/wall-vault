## wall-vault के हो?

**wall-vault = OpenClaw को लागि AI एजेन्ट (प्रोक्सी) + API कुञ्जी तिजोरी**

AI सेवाहरू प्रयोग गर्न **API कुञ्जी** चाहिन्छ। API कुञ्जी भनेको "यो व्यक्तिले यो सेवा प्रयोग गर्ने अधिकार छ" भनेर प्रमाणित गर्ने **डिजिटल प्रवेश कार्ड** जस्तो हो। तर यो प्रवेश कार्डको दैनिक प्रयोगको सीमा हुन्छ, र राम्रोसँग व्यवस्थापन नगरे उजागर हुने जोखिम पनि छ।

wall-vault ले यी प्रवेश कार्डहरूलाई सुरक्षित तिजोरीमा राख्छ र OpenClaw र AI सेवाहरू बीचमा **एजेन्ट (प्रोक्सी)** को भूमिका निभाउँछ। सरल भाषामा, OpenClaw लाई wall-vault मा मात्र जडान गर्नुपर्छ, बाँकी जटिल कामहरू wall-vault ले आफैं सम्हाल्छ।

wall-vault ले समाधान गर्ने समस्याहरू:

- **API कुञ्जी स्वचालित घुमाउने**: एउटा कुञ्जीको प्रयोग सीमामा पुगेमा वा अस्थायी रूपमा रोकिएमा (cooldown), चुपचाप अर्को कुञ्जीमा सर्छ। OpenClaw बिना रोकिएर काम गर्दै रहन्छ।
- **सेवा स्वचालित प्रतिस्थापन (Fallback)**: Google ले जवाफ नदिएमा OpenRouter मा सर्छ, त्यो पनि नभएमा तपाईंको कम्प्युटरमा स्थापित Ollama/LM Studio/vLLM (स्थानीय AI) मा स्वचालित रूपमा सर्छ। सत्र टुट्दैन। मूल सेवा पुनर्स्थापित हुँदा अर्को अनुरोधबाट स्वचालित रूपमा फर्किन्छ (v0.1.18+, LM Studio/vLLM: v0.1.21+)।
- **वास्तविक-समय समन्वय (SSE)**: तिजोरी ड्यासबोर्डमा मोडेल परिवर्तन गर्दा 1-3 सेकेन्डभित्र OpenClaw स्क्रिनमा प्रतिबिम्बित हुन्छ। SSE (Server-Sent Events) भनेको सर्भरले परिवर्तनहरू वास्तविक समयमा क्लाइन्टहरूलाई पठाउने प्रविधि हो।
- **वास्तविक-समय सूचना**: कुञ्जी सकिने वा सेवा असफल जस्ता घटनाहरू OpenClaw TUI (टर्मिनल स्क्रिन) को तलतिर तुरुन्तै देखिन्छ।

> 💡 **Claude Code, Cursor, VS Code** पनि जडान गर्न सकिन्छ, तर wall-vault को मूल उद्देश्य OpenClaw सँग प्रयोग गर्नु हो।

```
OpenClaw (TUI टर्मिनल स्क्रिन)
        │
        ▼
  wall-vault proxy (:56244)   ← कुञ्जी व्यवस्थापन, राउटिङ, fallback, घटनाहरू
        │
        ├─ Google Gemini API
        ├─ OpenRouter API (340+ मोडेलहरू)
        ├─ Ollama / LM Studio / vLLM (तपाईंको कम्प्युटर, अन्तिम शरण)
        └─ OpenAI / Anthropic API
```

## स्थापना

### Linux / macOS

टर्मिनल खोल्नुहोस् र तलका आदेशहरू टाँस्नुहोस्।

```bash
# Linux (सामान्य PC, सर्भर — amd64)
curl -L https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-linux-amd64 \
  -o ~/.local/bin/wall-vault && chmod +x ~/.local/bin/wall-vault

# macOS Apple Silicon (M1/M2/M3 Mac)
curl -L https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-darwin-arm64 \
  -o /usr/local/bin/wall-vault && chmod +x /usr/local/bin/wall-vault
```

- `curl -L ...` — इन्टरनेटबाट फाइल डाउनलोड गर्छ।
- `chmod +x` — डाउनलोड गरिएको फाइललाई "चलाउन सकिने" बनाउँछ। यो चरण छोड्नुभयो भने "अनुमति छैन" त्रुटि आउँछ।

### Windows

PowerShell (प्रशासक अधिकार) खोल्नुहोस् र निम्न आदेशहरू चलाउनुहोस्।

```powershell
# डाउनलोड
Invoke-WebRequest -Uri `
  "https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-windows-amd64.exe" `
  -OutFile "$env:LOCALAPPDATA\Programs\wall-vault\wall-vault.exe"

# PATH मा थप्नुहोस् (PowerShell पुनः सुरु गरेपछि लागू हुन्छ)
$env:PATH += ";$env:LOCALAPPDATA\Programs\wall-vault"
```

> 💡 **PATH भनेको के हो?** कम्प्युटरले आदेशहरू खोज्ने फोल्डरहरूको सूची हो। PATH मा थप्नुपर्छ ताकि कुनै पनि फोल्डरबाट `wall-vault` टाइप गरेर चलाउन सकिन्छ।

### स्रोत कोडबाट निर्माण (विकासकर्ताहरूका लागि)

Go भाषाको विकास वातावरण स्थापित भएको अवस्थामा मात्र लागू हुन्छ।

```bash
git clone https://github.com/sookmook/wall-vault
cd wall-vault
make build       # bin/wall-vault (संस्करण: v0.1.25.YYYYMMDD.HHmmss)
make install     # ~/.local/bin/wall-vault
```

> 💡 **निर्माण समय-छाप संस्करण**: `make build` ले निर्माण गर्दा संस्करण `v0.1.25.20260408.022325` जस्तो मिति र समय सहितको ढाँचामा स्वचालित रूपमा उत्पन्न हुन्छ। `go build ./...` ले सिधा निर्माण गर्दा संस्करण `"dev"` मात्र देखिन्छ।

## पहिलो पटक सुरु गर्नुहोस्

### Setup जादुगर चलाउनुहोस्

स्थापना पछि, पहिलो पटक अवश्य तलको आदेशले **सेटअप जादुगर** चलाउनुहोस्। जादुगरले आवश्यक कुराहरू एक एक गरेर सोध्दै मार्गदर्शन गर्छ।

```bash
wall-vault setup
```

जादुगरले अगाडि बढाउने चरणहरू यस्ता छन्:

```
1. भाषा छान्नुहोस् (नेपाली सहित 10 भाषाहरू)
2. थिम छान्नुहोस् (light / dark / gold / cherry / ocean)
3. सञ्चालन मोड — एक्लै प्रयोग (standalone) वा धेरै मेसिनमा (distributed) छान्नुहोस्
4. बट नाम प्रविष्ट गर्नुहोस् — ड्यासबोर्डमा देखिने नाम
5. पोर्ट सेटिङ — पूर्वनिर्धारित: proxy 56244, तिजोरी 56243 (परिवर्तन आवश्यक नभए Enter थिच्नुहोस्)
6. AI सेवा छान्नुहोस् — Google / OpenRouter / Ollama / LM Studio / vLLM
7. उपकरण सुरक्षा फिल्टर सेटिङ
8. प्रशासक टोकन सेट गर्नुहोस् — ड्यासबोर्ड व्यवस्थापन सुविधाहरू लक गर्ने पासवर्ड। स्वचालित उत्पादन पनि सम्भव छ
9. API कुञ्जी इन्क्रिप्सन पासवर्ड सेट गर्नुहोस् — कुञ्जीहरू अझ सुरक्षित रूपमा भण्डारण गर्न चाहनुहुन्छ भने (वैकल्पिक)
10. सेटिङ फाइल बचत मार्ग
```

> ⚠️ **प्रशासक टोकन अवश्य सम्झनुहोस्।** पछि ड्यासबोर्डमा कुञ्जी थप्दा वा सेटिङ परिवर्तन गर्दा आवश्यक पर्छ। बिर्सनुभयो भने सेटिङ फाइल सिधा सम्पादन गर्नुपर्छ।

जादुगर पूरा भएपछि `wall-vault.yaml` सेटिङ फाइल स्वचालित रूपमा सिर्जना हुन्छ।

### चलाउनुहोस्

```bash
wall-vault start
```

दुई सर्भरहरू एकै साथ सुरु हुन्छन्:

- **प्रोक्सी** (`http://localhost:56244`) — OpenClaw र AI सेवाहरू बीच जडान गर्ने एजेन्ट
- **कुञ्जी तिजोरी** (`http://localhost:56243`) — API कुञ्जी व्यवस्थापन र वेब ड्यासबोर्ड

ब्राउजरमा `http://localhost:56243` खोल्नुहोस् ड्यासबोर्ड तुरुन्तै हेर्न।

## API कुञ्जी दर्ता

API कुञ्जी दर्ता गर्ने चार विधिहरू छन्। **सुरु गर्दै हुनुहुन्छ भने विधि 1 (वातावरण चर) सिफारिस गरिन्छ**।

### विधि 1: वातावरण चर (सिफारिस — सबैभन्दा सरल)

वातावरण चर भनेको प्रोग्राम सुरु हुँदा पढ्ने **पूर्व-निर्धारित मान** हो। टर्मिनलमा यसरी टाइप गर्नुहोस्।

```bash
# Google Gemini कुञ्जी दर्ता
export WV_KEY_GOOGLE=AIzaSy...

# OpenRouter कुञ्जी दर्ता
export WV_KEY_OPENROUTER=sk-or-v1-...

# दर्ता पछि चलाउनुहोस्
wall-vault start
```

धेरै कुञ्जीहरू भएमा अल्पविराम (,) ले जोड्नुहोस्। wall-vault ले कुञ्जीहरू बारीबारी स्वचालित रूपमा प्रयोग गर्छ (round robin):

```bash
export WV_KEY_GOOGLE=AIzaSy...,AIzaSy...,AIzaSy...
```

> 💡 **सुझाव**: `export` आदेश हालको टर्मिनल सत्रमा मात्र लागू हुन्छ। कम्प्युटर पुनः सुरु गरेपछि पनि कायम रहोस् भने `~/.bashrc` वा `~/.zshrc` फाइलमा माथिको पंक्ति थप्नुहोस्।

### विधि 2: ड्यासबोर्ड UI (माउसले क्लिक गर्नुहोस्)

1. ब्राउजरमा `http://localhost:56243` मा जानुहोस्
2. माथिको **🔑 API कुञ्जी** कार्डमा `[+ थप्नुहोस्]` बटन क्लिक गर्नुहोस्
3. सेवा प्रकार, कुञ्जी मान, लेबल (सम्झनाको नाम), र दैनिक सीमा प्रविष्ट गरेर बचत गर्नुहोस्

### विधि 3: REST API (स्वचालित/स्क्रिप्टको लागि)

REST API भनेको प्रोग्रामहरू बीच HTTP मार्फत डेटा आदानप्रदान गर्ने तरिका हो। स्क्रिप्टबाट स्वचालित दर्ता गर्दा उपयोगी।

```bash
curl -X POST http://localhost:56243/admin/keys \
  -H "Authorization: Bearer प्रशासक-टोकन" \
  -H "Content-Type: application/json" \
  -d '{
    "service": "google",
    "key": "AIzaSy...",
    "label": "मुख्य कुञ्जी",
    "daily_limit": 1000
  }'
```

### विधि 4: Proxy फ्ल्याग (छोटो परीक्षणको लागि)

औपचारिक दर्ता बिना अस्थायी रूपमा कुञ्जी हालेर परीक्षण गर्दा प्रयोग गरिन्छ। प्रोग्राम बन्द गर्दा हराउँछ।

```bash
wall-vault proxy --key-google=AIzaSy... --key-openrouter=sk-or-...
```

## प्रोक्सी प्रयोग विधि

### OpenClaw मा प्रयोग (मुख्य उद्देश्य)

OpenClaw लाई wall-vault मार्फत AI सेवाहरूमा जडान हुने गरी सेट गर्ने तरिका।

`~/.openclaw/openclaw.json` फाइल खोल्नुहोस् र तलको सामग्री थप्नुहोस्:

```json5
// ~/.openclaw/openclaw.json
{
  models: {
    providers: {
      "wall-vault": {
        baseUrl: "http://localhost:56244/v1",
        apiKey: "your-agent-token",   // vault एजेन्ट टोकन
        api: "openai-completions",
        models: [
          { id: "wall-vault/gemini-2.5-flash" },
          { id: "wall-vault/gemini-2.5-pro" },
          { id: "wall-vault/hunter-alpha" },    // निःशुल्क 1M context
          { id: "wall-vault/claude-opus-4-6" }
        ]
      }
    }
  }
}
```

> 💡 **अझ सजिलो तरिका**: ड्यासबोर्ड एजेन्ट कार्डको **🦞 OpenClaw सेटिङ कपि गर्नुहोस्** बटन थिच्नुहोस् र टोकन र ठेगाना पहिल्यै भरिएको स्निपेट क्लिपबोर्डमा कपि हुन्छ। टाँस्नुहोस् मात्र।

**मोडेल नामको अगाडि `wall-vault/` कहाँ जडान हुन्छ?**

wall-vault ले मोडेल नामको आधारमा कुन AI सेवामा अनुरोध पठाउने स्वचालित रूपमा निर्णय गर्छ:

| मोडेल ढाँचा | जडान हुने सेवा |
|----------|--------------|
| `wall-vault/gemini-*` | Google Gemini सँग सिधा जडान |
| `wall-vault/gpt-*`, `wall-vault/o3`, `wall-vault/o4*` | OpenAI सँग सिधा जडान |
| `wall-vault/claude-*` | OpenRouter मार्फत Anthropic जडान |
| `wall-vault/hunter-alpha`, `wall-vault/healer-alpha` | OpenRouter (निःशुल्क 10 लाख टोकन context) |
| `wall-vault/kimi-*`, `wall-vault/glm-*`, `wall-vault/deepseek-*` | OpenRouter जडान |
| `google/मोडेल-नाम`, `openai/मोडेल-नाम`, `anthropic/मोडेल-नाम` आदि | सम्बन्धित सेवासँग सिधा जडान |
| `custom/google/मोडेल-नाम`, `custom/openai/मोडेल-नाम` आदि | `custom/` भाग हटाएर पुन: राउट गरिन्छ |
| `मोडेल-नाम:cloud` | `:cloud` भाग हटाएर OpenRouter मा जडान |

> 💡 **Context भनेको के हो?** AI ले एक पटकमा सम्झन सक्ने कुराकानीको मात्रा हो। 1M (10 लाख टोकन) भनेको धेरै लामो कुराकानी वा लामो कागजात पनि एक पटकमा प्रशोधन गर्न सकिन्छ।

### Gemini API ढाँचामा सिधा जडान (विद्यमान उपकरण अनुकूलता)

Google Gemini API सिधा प्रयोग गरिरहेका उपकरणहरू छन् भने ठेगाना मात्र wall-vault मा बदल्नुहोस्:

```bash
export ANTHROPIC_BASE_URL=http://localhost:56244/google
```

वा URL सिधा निर्दिष्ट गर्ने उपकरणहरूका लागि:

```
http://localhost:56244/google/v1beta/models/gemini-2.5-flash:generateContent
```

### OpenAI SDK (Python) मा प्रयोग

Python मा AI प्रयोग गर्ने कोडमा पनि wall-vault जडान गर्न सकिन्छ। `base_url` मात्र बदल्नुहोस्:

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:56244/v1",
    api_key="not-needed"  # API कुञ्जी wall-vault ले आफैं व्यवस्थापन गर्छ
)

response = client.chat.completions.create(
    model="google/gemini-2.5-flash",   # प्रदायक/मोडेल ढाँचा
    messages=[{"role": "user", "content": "नमस्ते"}]
)
```

### चलिरहेको बेलामा मोडेल परिवर्तन गर्ने

wall-vault पहिल्यै चलिरहेको अवस्थामा AI मोडेल बदल्न:

```bash
# प्रोक्सीमा सिधा अनुरोध गरेर मोडेल परिवर्तन
```

### API कुञ्जी कार्ड

दर्ता गरिएका API कुञ्जीहरू एक नजरमा व्यवस्थापन गर्न सक्ने कार्ड।

- सेवा अनुसार विभाजित गरेर कुञ्जी सूची देखाउँछ।
- `today_usage`: आज सफलतापूर्वक प्रशोधन गरिएको टोकन (AI ले पढेको र लेखेको अक्षर) संख्या
- `today_attempts`: आजको कुल कल संख्या (सफल + असफल सहित)
- `[+ थप्नुहोस्]` बटनले नयाँ कुञ्जी दर्ता, र `✕` ले कुञ्जी मेटाउँछ।

> 💡 **टोकन भनेको के हो?** AI ले पाठ प्रशोधन गर्दा प्रयोग गर्ने एकाइ हो। लगभग अंग्रेजीको एक शब्द, वा कोरियाली 1-2 अक्षर बराबर हुन्छ। API शुल्क सामान्यतया यो टोकन संख्याको आधारमा गणना गरिन्छ।

तिजोरी ड्यासबोर्डमा एजेन्टको मोडेल बदल्दा, त्यो एजेन्टको स्थानीय सेटिङ स्वचालित रूपमा अद्यावधिक हुन्छ।

- `openAiBaseUrl` र API कुञ्जी छुँदैन
- **VS Code पुनः लोड (`Ctrl+Alt+R` वा `Ctrl+Shift+P` → `Developer: Reload Window`) आवश्यक छ**
  - किनभने Cline चलिरहेको बेलामा सेटिङ फाइल पुनः पढ्दैन

**Claude Code को लागि:**
- तिजोरीमा मोडेल परिवर्तन गर्दा → SSE घटना → प्रोक्सीले `settings.json` को `model` फिल्ड अद्यावधिक गर्छ
- WSL र Windows दुवै मार्ग स्वचालित रूपमा खोज्छ (`~/.claude/`, `/mnt/c/Users/*/.claude/`)

**विपरीत दिशा (एजेन्ट → तिजोरी):**
- एजेन्ट (Cline, Claude Code आदि) ले प्रोक्सीमा अनुरोध पठाउँदा, प्रोक्सीले heartbeat मा त्यो क्लाइन्टको सेवा/मोडेल जानकारी समावेश गर्छ
- तिजोरी ड्यासबोर्डको एजेन्ट कार्डमा हाल प्रयोग भइरहेको सेवा/मोडेल वास्तविक समयमा देखिन्छ

> 💡 **मुख्य कुरा**: प्रोक्सीले अनुरोधको Authorization टोकनबाट एजेन्ट पहिचान गर्छ, र तिजोरीमा सेट गरिएको सेवा/मोडेलमा स्वचालित रूपमा राउट गर्छ। Cline वा Claude Code ले फरक मोडेल नाम पठाए पनि, प्रोक्सीले तिजोरी सेटिङबाट ओभरराइड गर्छ।

### VS Code मा Cline प्रयोग — विस्तृत मार्गदर्शन

#### चरण 1: Cline स्थापना

#### चरण 2: तिजोरीमा एजेन्ट दर्ता

1. तिजोरी ड्यासबोर्ड (`http://तिजोरी-IP:56243`) खोल्नुहोस्
2. **एजेन्ट** खण्डमा **+ थप्नुहोस्** क्लिक गर्नुहोस्
3. निम्नानुसार प्रविष्ट गर्नुहोस्:

| फिल्ड | मान | विवरण |
|------|----|------|

4. **बचत** थिच्नुहोस् र टोकन स्वचालित रूपमा सिर्जना हुन्छ

#### चरण 3: Cline मा जडान

**विधि A — स्वचालित लागू (सिफारिस)**

1. त्यो मेसिनमा wall-vault **प्रोक्सी** चलिरहेको छ भनी सुनिश्चित गर्नुहोस् (`localhost:56244`)
2. ड्यासबोर्डको एजेन्ट कार्डमा **⚡ Cline सेटिङ लागू** बटन क्लिक गर्नुहोस्
3. "सेटिङ लागू भयो!" सूचना देखिए सफल
4. VS Code पुनः लोड गर्नुहोस् (`Ctrl+Alt+R`)

**विधि B — म्यानुअल सेटिङ**

Cline साइडबारमा सेटिङ (⚙️) खोल्नुहोस् र:
- **API Provider**: `OpenAI Compatible`
- **Base URL**: `http://प्रोक्सी-ठेगाना:56244/v1`
  - उही मेसिनमा: `http://localhost:56244/v1`
  - अर्को मेसिनमा जस्तै Mini सर्भर: `http://192.168.0.6:56244/v1`
- **API Key**: तिजोरीबाट जारी गरिएको टोकन (एजेन्ट कार्डबाट कपि)
- **Model ID**: तिजोरीमा सेट गरिएको मोडेल (उदा: `gemini-2.5-flash`)

#### चरण 4: पुष्टि

Cline च्याटमा कुनै पनि सन्देश पठाउनुहोस्। सामान्य भए:
- तिजोरी ड्यासबोर्डको सम्बन्धित एजेन्ट कार्डमा **हरियो बिन्दु (● चलिरहेको)** देखिन्छ
- कार्डमा हालको सेवा/मोडेल देखिन्छ (उदा: `google / gemini-2.5-flash`)

#### मोडेल परिवर्तन

Cline को मोडेल बदल्न चाहनुहुन्छ भने **तिजोरी ड्यासबोर्ड** बाट परिवर्तन गर्नुहोस्:

1. एजेन्ट कार्डको सेवा/मोडेल ड्रपडाउन परिवर्तन गर्नुहोस्
2. **लागू गर्नुहोस्** क्लिक गर्नुहोस्
3. VS Code पुनः लोड (`Ctrl+Alt+R`) — Cline फुटरको मोडेल नाम अद्यावधिक हुन्छ
4. अर्को अनुरोधबाट नयाँ मोडेल प्रयोग हुन्छ

> 💡 वास्तवमा, प्रोक्सीले Cline को अनुरोध टोकनबाट पहिचान गरेर तिजोरी सेटिङको मोडेलमा राउट गर्छ। VS Code पुनः लोड नगरे पनि **वास्तवमा प्रयोग हुने मोडेल तुरुन्तै बदलिन्छ** — पुनः लोड Cline UI मा मोडेल प्रदर्शन अद्यावधिक गर्नको लागि हो।

#### जडान विच्छेद पत्ता लगाउने

VS Code बन्द गर्दा तिजोरी ड्यासबोर्डमा एजेन्ट कार्ड लगभग **90 सेकेन्ड** पछि पहेँलो (ढिलो), र **3 मिनेट** पछि रातो (अफलाइन) मा बदलिन्छ। (v0.1.18 देखि 15 सेकेन्ड अन्तरालको स्थिति जाँचले अफलाइन पत्ता लगाउने चाँडो भएको छ।)

|------|------|------|
| Cline मा "जडान असफल" त्रुटि | प्रोक्सी चलिरहेको छैन वा ठेगाना गलत | `curl http://localhost:56244/health` ले प्रोक्सी पुष्टि गर्नुहोस् |

नयाँ कम्प्युटरमा wall-vault प्रोक्सी पहिलो पटक स्थापना गरेर तिजोरीमा जडान गर्दा प्रयोग गरिन्छ। बटन क्लिक गर्दा स्थापना स्क्रिप्ट पूरै कपि हुन्छ। नयाँ कम्प्युटरको टर्मिनलमा टाँसेर चलाउनुहोस्, र निम्न कामहरू एकै पटक हुन्छन्:

1. wall-vault बाइनरी स्थापना (पहिल्यै स्थापित भए छोड्छ)
2. systemd प्रयोगकर्ता सेवा स्वचालित दर्ता
3. सेवा सुरु र तिजोरीमा स्वचालित जडान

> 💡 स्क्रिप्टमा यो एजेन्टको टोकन र तिजोरी सर्भर ठेगाना पहिल्यै भरिएको हुन्छ, त्यसैले टाँसेपछि कुनै परिवर्तन बिना सिधा चलाउन सकिन्छ।

### सेवा कार्ड

प्रयोग गर्ने AI सेवाहरू अन/अफ गर्ने वा सेट गर्ने कार्ड।

- सेवा अनुसार सक्रिय/निष्क्रिय टगल स्विच
- स्थानीय AI सर्भर (तपाईंको कम्प्युटरमा चलिरहेको Ollama, LM Studio, vLLM आदि) को ठेगाना प्रविष्ट गर्दा उपलब्ध मोडेलहरू स्वचालित रूपमा पत्ता लगाइन्छ।
- **स्थानीय सेवा जडान स्थिति प्रदर्शन**: सेवा नामको छेउमा ● बिन्दु **हरियो** भए जडान, **खैरो** भए जडान छैन
- **स्थानीय सेवा स्वचालित सिग्नल बत्ती** (v0.1.23+): स्थानीय सेवाहरू (Ollama, LM Studio, vLLM) जडान सम्भव हुँदा स्वचालित रूपमा सक्रिय हुन्छन्, विच्छेद हुँदा स्वचालित रूपमा निष्क्रिय हुन्छन्। सेवा सक्रिय गर्दा 15 सेकेन्डभित्र ● हरियोमा बदलिन्छ र चेकबक्स अन हुन्छ, र बन्द गर्दा स्वचालित रूपमा बन्द हुन्छ। क्लाउड सेवाहरू (Google, OpenRouter आदि) ले API कुञ्जीको आधारमा स्वचालित टगल गर्ने जस्तै काम गर्छ।

> 💡 **स्थानीय सेवा अर्को कम्प्युटरमा चलिरहेको छ भने**: सेवा URL इनपुटमा त्यो कम्प्युटरको IP प्रविष्ट गर्नुहोस्। उदा: `http://192.168.0.6:11434` (Ollama), `http://192.168.0.6:1234` (LM Studio)। सेवा `0.0.0.0` होइन `127.0.0.1` मा मात्र बाइन्ड भएको छ भने बाहिरी IP बाट पहुँच हुँदैन, त्यसैले सेवा सेटिङमा बाइन्ड ठेगाना जाँच्नुहोस्।

### प्रशासक टोकन प्रविष्टि

ड्यासबोर्डमा कुञ्जी थप्ने/मेटाउने जस्ता महत्त्वपूर्ण कार्यहरू प्रयोग गर्न खोज्दा प्रशासक टोकन प्रविष्टि पपअप देखिन्छ। Setup जादुगरमा सेट गरेको टोकन प्रविष्ट गर्नुहोस्। एक पटक प्रविष्ट गरेपछि ब्राउजर बन्द नगरेसम्म कायम रहन्छ।

> ⚠️ **15 मिनेटभित्र प्रमाणीकरण 10 पटक भन्दा बढी असफल भएमा त्यो IP अस्थायी रूपमा ब्लक हुन्छ।** टोकन बिर्सनुभयो भने `wall-vault.yaml` फाइलमा `admin_token` खण्ड जाँच्नुहोस्।

## वितरित मोड (बहु-बट)

धेरै कम्प्युटरहरूमा एकै साथ OpenClaw सञ्चालन गर्दा **एउटा कुञ्जी तिजोरी साझा** गर्ने संरचना। कुञ्जी व्यवस्थापन एक ठाउँमा मात्र गर्नुपर्ने भएकोले सुविधाजनक।

### संरचना उदाहरण

```
[कुञ्जी तिजोरी सर्भर]
  wall-vault vault    (कुञ्जी तिजोरी :56243, ड्यासबोर्ड)

[WSL अल्फा]            [Raspberry Pi गामा]    [Mac Mini स्थानीय]
  wall-vault proxy      wall-vault proxy        wall-vault proxy
  openclaw TUI          openclaw TUI            openclaw TUI
  ↕ SSE समन्वय          ↕ SSE समन्वय            ↕ SSE समन्वय
```

सबै बटहरू बीचको तिजोरी सर्भरतर्फ हेरिरहेका हुन्छन्, त्यसैले तिजोरीमा मोडेल बदल्दा वा कुञ्जी थप्दा सबै बटमा तुरुन्तै प्रतिबिम्बित हुन्छ।

### चरण 1: कुञ्जी तिजोरी सर्भर सुरु

तिजोरी सर्भरको रूपमा प्रयोग गर्ने कम्प्युटरमा चलाउनुहोस्:

```bash
wall-vault vault
```

### चरण 2: प्रत्येक बट (क्लाइन्ट) दर्ता

तिजोरी सर्भरमा जडान हुने प्रत्येक बटको जानकारी पहिल्यै दर्ता गर्नुहोस्:

```bash
curl -X POST http://localhost:56243/admin/clients \
  -H "Authorization: Bearer प्रशासक-टोकन" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "botA",
    "name": "बटA",
    "token": "bota-secret",
    "default_service": "google",
    "default_model": "gemini-2.5-flash"
  }'
```

### चरण 3: प्रत्येक बट कम्प्युटरमा प्रोक्सी सुरु

बट स्थापित भएको प्रत्येक कम्प्युटरमा तिजोरी सर्भर ठेगाना र टोकन निर्दिष्ट गरेर प्रोक्सी चलाउनुहोस्:

```bash
WV_VAULT_URL=http://192.168.x.x:56243 \
WV_VAULT_TOKEN=bota-secret \
WV_VAULT_CLIENT_ID=botA \
wall-vault proxy
```

> 💡 **`192.168.x.x`** भागलाई तिजोरी सर्भर कम्प्युटरको वास्तविक आन्तरिक IP ठेगानामा बदल्नुहोस्। राउटर सेटिङ वा `ip addr` आदेशले पुष्टि गर्न सकिन्छ।

## स्वचालित सुरुआत सेटअप

कम्प्युटर पुनः सुरु गर्दा हरेक पटक म्यानुअली wall-vault सुरु गर्नु झन्झटिलो छ भने, प्रणाली सेवाको रूपमा दर्ता गर्नुहोस्। एक पटक दर्ता भएपछि बुट हुँदा स्वचालित रूपमा सुरु हुन्छ।

### Linux — systemd (अधिकांश Linux)

systemd भनेको Linux मा प्रोग्रामहरू स्वचालित रूपमा सुरु र व्यवस्थापन गर्ने प्रणाली हो:

```bash
wall-vault doctor deploy
systemctl --user daemon-reload
systemctl --user enable --now wall-vault
```

लग हेर्नुहोस्:

```bash
journalctl --user -u wall-vault -f
```

### macOS — launchd

macOS मा प्रोग्राम स्वचालित सुरुआत व्यवस्थापन गर्ने प्रणाली:

```bash
wall-vault doctor deploy launchd
launchctl load ~/Library/LaunchAgents/com.wall-vault.plist
```

### Windows — NSSM

1. [nssm.cc](https://nssm.cc/download) बाट NSSM डाउनलोड गरेर PATH मा थप्नुहोस्।
2. प्रशासक PowerShell मा:

```powershell
wall-vault doctor deploy windows
```

## RTK टोकन बचत

*(v0.1.24+)*

**RTK (टोकन बचत उपकरण)** ले AI कोडिङ एजेन्ट (Claude Code आदि) ले चलाउने शेल आदेशको आउटपुट स्वचालित रूपमा संकुचित गरेर टोकन प्रयोग घटाउँछ। उदाहरणका लागि, `git status` को 15 पंक्ति आउटपुट 2 पंक्ति सारांशमा घट्छ।

### आधारभूत प्रयोग

```bash
# आदेशलाई wall-vault rtk ले बेर्नुहोस् र आउटपुट स्वचालित रूपमा फिल्टर हुन्छ
wall-vault rtk git status          # परिवर्तित फाइल सूची मात्र
wall-vault rtk git diff HEAD~1     # परिवर्तित पंक्ति + न्यूनतम context
wall-vault rtk git log -10         # Hash + एक पंक्ति सन्देश
wall-vault rtk go test ./...       # असफल परीक्षण मात्र
wall-vault rtk ls -la              # असमर्थित आदेश स्वचालित काटिन्छ
```

### समर्थित आदेश र बचत प्रभाव

| आदेश | फिल्टर विधि | बचत दर |
|------|----------|--------|
| `git status` | परिवर्तित फाइल सारांश मात्र | ~87% |
| `git diff` | परिवर्तित पंक्ति + 3 पंक्ति context | ~60-94% |
| `git log` | Hash + पहिलो पंक्ति सन्देश | ~90% |
| `git push/pull/fetch` | प्रगति हटाउने, सारांश मात्र | ~80% |
| `go test` | असफल मात्र देखाउने, उत्तीर्ण गणना | ~88-99% |
| `go build/vet` | त्रुटि मात्र देखाउने | ~90% |
| अन्य सबै आदेश | पहिलो 50 पंक्ति + अन्तिम 50 पंक्ति, अधिकतम 32KB | परिवर्तनशील |

### 3-चरण फिल्टर पाइपलाइन

1. **आदेश अनुसार संरचना फिल्टर** — git, go आदिको आउटपुट ढाँचा बुझेर अर्थपूर्ण भागहरू मात्र निकाल्छ
2. **Regex पश्च-प्रशोधन** — ANSI रंग कोड हटाउने, खाली पंक्ति संकुचित, दोहोरिएका पंक्ति एकत्रित
3. **Passthrough + काटछाँट** — असमर्थित आदेशले पहिलो/अन्तिम 50 पंक्ति मात्र राख्छ

### Claude Code सँग एकीकरण

Claude Code को `PreToolUse` हुकबाट सबै शेल आदेशहरू स्वचालित रूपमा RTK मार्फत पठाउन सेट गर्न सकिन्छ।

```bash
# हुक स्थापना (Claude Code settings.json मा स्वचालित रूपमा थपिन्छ)
wall-vault rtk hook install
```

वा म्यानुअली `~/.claude/settings.json` मा थप्नुहोस्:

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "command": "wall-vault rtk rewrite"
    }]
  }
}
```

> 💡 **Exit code संरक्षण**: RTK ले मूल आदेशको निकास कोड जस्ताको तस्तै फर्काउँछ। आदेश असफल भएमा (exit code ≠ 0) AI ले पनि असफलता सही रूपमा पत्ता लगाउँछ।

> 💡 **अंग्रेजी बाध्यता**: RTK ले `LC_ALL=C` मा आदेश चलाउँछ ताकि प्रणाली भाषा सेटिङको पर्वाह नगरी सधैं अंग्रेजी आउटपुट उत्पन्न गर्छ। यसले फिल्टर सही रूपमा काम गर्न सुनिश्चित गर्छ।

## वातावरण चर सन्दर्भ

वातावरण चर भनेको प्रोग्राममा सेटिङ मान पठाउने तरिका हो। `export चर-नाम=मान` ढाँचामा टर्मिनलमा प्रविष्ट गर्नुहोस्, वा स्वचालित सुरुआत सेवा फाइलमा राख्नुहोस् ताकि सधैं लागू रहोस्।

| चर | विवरण | उदाहरण मान |
|------|------|---------|
| `WV_LANG` | ड्यासबोर्ड भाषा | `ko`, `en`, `ja` |
| `WV_THEME` | ड्यासबोर्ड थिम | `light`, `dark`, `gold` |
| `WV_KEY_GOOGLE` | Google API कुञ्जी (अल्पविरामले धेरै) | `AIza...,AIza...` |
| `WV_KEY_OPENROUTER` | OpenRouter API कुञ्जी | `sk-or-v1-...` |
| `WV_VAULT_URL` | वितरित मोडमा तिजोरी सर्भर ठेगाना | `http://192.168.x.x:56243` |
| `WV_VAULT_TOKEN` | क्लाइन्ट (बट) प्रमाणीकरण टोकन | `my-secret-token` |
| `WV_ADMIN_TOKEN` | प्रशासक टोकन | `admin-token-here` |
| `WV_MASTER_PASS` | API कुञ्जी इन्क्रिप्सन पासवर्ड | `my-password` |
| `WV_AVATAR` | अवतार छवि फाइल मार्ग (`~/.openclaw/` बाट सापेक्ष मार्ग) | `workspace/avatars/avatar.png` |
| `OLLAMA_URL` | Ollama स्थानीय सर्भर ठेगाना | `http://192.168.x.x:11434` |

### API कुञ्जी त्रुटि आउँदा (429, 402, 401, 403, 582)

| त्रुटि कोड | अर्थ | समाधान |
|----------|------|----------|
| **429** | धेरै अनुरोधहरू (प्रयोग सीमा नाघ्यो) | केही बेर पर्खनुहोस् वा अर्को कुञ्जी थप्नुहोस् |
| **402** | भुक्तानी आवश्यक वा क्रेडिट अपर्याप्त | सम्बन्धित सेवामा क्रेडिट चार्ज गर्नुहोस् |
| **401 / 403** | कुञ्जी गलत वा अनुमति छैन | कुञ्जी मान पुनः पुष्टि गरेर दर्ता गर्नुहोस् |
| **582** | Gateway ओभरलोड (5 मिनेट cooldown) | 5 मिनेट पछि स्वचालित रूपमा खुल्छ |

```bash
# दर्ता भएका कुञ्जी सूची र स्थिति जाँच
curl -H "Authorization: Bearer प्रशासक-टोकन" http://localhost:56243/admin/keys

# कुञ्जी प्रयोग काउन्टर रिसेट
curl -X POST -H "Authorization: Bearer प्रशासक-टोकन" http://localhost:56243/admin/keys/reset
```

### एजेन्ट "नजडान" मा देखिँदा

"नजडान" भनेको प्रोक्सी प्रक्रियाले तिजोरीमा सिग्नल (heartbeat) पठाइरहेको छैन। **सेटिङ बचत भएको छैन भन्ने अर्थ होइन।** प्रोक्सीले तिजोरी सर्भर ठेगाना र टोकन जानेर चलिरहेको हुनुपर्छ ताकि जडान स्थिति बदलिन्छ।

```bash
# तिजोरी सर्भर ठेगाना, टोकन, र क्लाइन्ट ID निर्दिष्ट गरेर प्रोक्सी सुरु
WV_VAULT_URL=http://तिजोरी-सर्भर-ठेगाना:56243 \
WV_VAULT_TOKEN=क्लाइन्ट-टोकन \
WV_VAULT_CLIENT_ID=क्लाइन्ट-ID \
wall-vault proxy
```

जडान सफल भएमा लगभग 20 सेकेन्डभित्र ड्यासबोर्डमा 🟢 चलिरहेको मा बदलिन्छ।

### Ollama जडान नहुँदा

Ollama भनेको तपाईंको कम्प्युटरमा सिधा AI चलाउने प्रोग्राम हो। पहिले Ollama सुरु भएको छ कि छैन सुनिश्चित गर्नुहोस्।

```bash
curl http://localhost:11434/api/tags   # मोडेल सूची देखिए सामान्य हो
export OLLAMA_URL=http://192.168.x.x:11434   # अर्को कम्प्युटरमा चलिरहेको भए
```

> ⚠️ Ollama ले जवाफ नदिएमा पहिले `ollama serve` आदेशले Ollama सुरु गर्नुहोस्।

> ⚠️ **ठूला मोडेलहरू ढिला हुन्छन्**: `qwen3.5:35b`, `deepseek-r1` जस्ता ठूला मोडेलले जवाफ उत्पन्न गर्न केही मिनेट लाग्न सक्छ। जवाफ नआएको जस्तो लागे पनि सामान्य रूपमा प्रशोधन भइरहेको हुन सक्छ, त्यसैले पर्खनुहोस्।

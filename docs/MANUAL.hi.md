## wall-vault क्या है?

**wall-vault = OpenClaw के लिए AI प्रॉक्सी + API कुंजी तिजोरी**

AI सेवाओं का उपयोग करने के लिए आपको **API कुंजियों** की आवश्यकता होती है। API कुंजी एक **डिजिटल एक्सेस पास** जैसी है जो यह साबित करती है कि "इस व्यक्ति के पास इस सेवा का उपयोग करने की अनुमति है"। लेकिन इन एक्सेस पास की प्रतिदिन उपयोग सीमा होती है, और खराब प्रबंधन से ये उजागर हो सकते हैं।

wall-vault इन एक्सेस पास को एक सुरक्षित तिजोरी में रखता है और OpenClaw तथा AI सेवाओं के बीच **प्रॉक्सी (मध्यस्थ)** की भूमिका निभाता है। सीधे शब्दों में, OpenClaw को केवल wall-vault से जुड़ना होता है, और बाकी जटिल काम wall-vault संभालता है।

wall-vault जो समस्याएं हल करता है:

- **API कुंजियों का स्वचालित रोटेशन**: जब किसी कुंजी की उपयोग सीमा पूरी हो जाती है या अस्थायी रूप से ब्लॉक हो जाती है (cooldown), तो चुपचाप अगली कुंजी पर स्विच हो जाती है। OpenClaw बिना रुकावट के काम करता रहता है।
- **स्वचालित सेवा स्विचिंग (fallback)**: अगर Google जवाब नहीं देता, तो स्वचालित रूप से OpenRouter पर स्विच हो जाता है; वह भी काम न करे तो आपके कंप्यूटर पर इंस्टॉल Ollama/LM Studio/vLLM (लोकल AI) पर स्विच हो जाता है। सेशन नहीं टूटता। जब मूल सेवा ठीक हो जाती है, अगले रिक्वेस्ट से स्वचालित रूप से वापस आ जाता है (v0.1.18+, LM Studio/vLLM: v0.1.21+)।
- **रियल-टाइम सिंक्रोनाइज़ेशन (SSE)**: जब आप तिजोरी डैशबोर्ड में मॉडल बदलते हैं, 1-3 सेकंड में OpenClaw की स्क्रीन पर प्रतिबिंबित होता है। SSE (Server-Sent Events) एक तकनीक है जिसमें सर्वर क्लाइंट को रियल-टाइम में अपडेट भेजता है।
- **रियल-टाइम सूचनाएं**: कुंजी समाप्ति या सेवा विफलता जैसी घटनाएं OpenClaw TUI (टर्मिनल स्क्रीन) के निचले भाग में तुरंत प्रदर्शित होती हैं।

> 💡 **Claude Code, Cursor, VS Code** भी कनेक्ट किए जा सकते हैं, लेकिन wall-vault का मूल उद्देश्य OpenClaw के साथ उपयोग करना है।

```
OpenClaw (TUI टर्मिनल स्क्रीन)
        │
        ▼
  wall-vault प्रॉक्सी (:56244)   ← कुंजी प्रबंधन, रूटिंग, फॉलबैक, इवेंट्स
        │
        ├─ Google Gemini API
        ├─ OpenRouter API (340+ मॉडल)
        ├─ Ollama / LM Studio / vLLM (लोकल कंप्यूटर, अंतिम उपाय)
        └─ OpenAI / Anthropic API
```

## इंस्टॉलेशन

### Linux / macOS

टर्मिनल खोलें और नीचे दिए गए कमांड पेस्ट करें।

```bash
# Linux (सामान्य PC, सर्वर — amd64)
curl -L https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-linux-amd64 \
  -o ~/.local/bin/wall-vault && chmod +x ~/.local/bin/wall-vault

# macOS Apple Silicon (M1/M2/M3 Mac)
curl -L https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-darwin-arm64 \
  -o /usr/local/bin/wall-vault && chmod +x /usr/local/bin/wall-vault
```

- `curl -L ...` — इंटरनेट से फ़ाइल डाउनलोड करता है।
- `chmod +x` — डाउनलोड की गई फ़ाइल को "एक्ज़ीक्यूटेबल" बनाता है। इस चरण को छोड़ने पर "अनुमति अस्वीकृत" त्रुटि आएगी।

### Windows

PowerShell (एडमिनिस्ट्रेटर अधिकारों के साथ) खोलें और नीचे दिए गए कमांड चलाएं।

```powershell
# डाउनलोड
Invoke-WebRequest -Uri `
  "https://github.com/sookmook/wall-vault/releases/latest/download/wall-vault-windows-amd64.exe" `
  -OutFile "$env:LOCALAPPDATA\Programs\wall-vault\wall-vault.exe"

# PATH में जोड़ें (PowerShell रीस्टार्ट के बाद लागू)
$env:PATH += ";$env:LOCALAPPDATA\Programs\wall-vault"
```

> 💡 **PATH क्या है?** यह उन फ़ोल्डरों की सूची है जहां कंप्यूटर कमांड खोजता है। PATH में जोड़ने से आप किसी भी फ़ोल्डर से `wall-vault` टाइप करके चला सकते हैं।

### सोर्स से बिल्ड (डेवलपर्स के लिए)

केवल तभी लागू जब Go भाषा का डेवलपमेंट एनवायरनमेंट इंस्टॉल हो।

```bash
git clone https://github.com/sookmook/wall-vault
cd wall-vault
make build       # bin/wall-vault (वर्ज़न: v0.1.25.YYYYMMDD.HHmmss)
make install     # ~/.local/bin/wall-vault
```

> 💡 **बिल्ड टाइमस्टैम्प वर्ज़न**: `make build` से बिल्ड करने पर वर्ज़न `v0.1.25.20260408.022325` जैसे दिनांक और समय सहित फ़ॉर्मैट में स्वचालित रूप से बनता है। `go build ./...` से सीधे बिल्ड करने पर वर्ज़न केवल `"dev"` दिखाई देता है।

## पहली शुरुआत

### setup विज़ार्ड चलाएं

इंस्टॉलेशन के बाद, पहली बार नीचे दिए गए कमांड से **सेटअप विज़ार्ड** चलाना अनिवार्य है। विज़ार्ड आपको एक-एक करके आवश्यक आइटम पूछते हुए मार्गदर्शन करेगा।

```bash
wall-vault setup
```

विज़ार्ड के चरण इस प्रकार हैं:

```
1. भाषा चयन (कोरियन सहित 10 भाषाएं)
2. थीम चयन (light / dark / gold / cherry / ocean)
3. ऑपरेशन मोड — अकेले उपयोग (standalone) या कई कंप्यूटरों पर (distributed)
4. बॉट का नाम — डैशबोर्ड पर दिखने वाला नाम
5. पोर्ट सेटिंग — डिफ़ॉल्ट: प्रॉक्सी 56244, तिजोरी 56243 (बदलने की ज़रूरत नहीं तो Enter दबाएं)
6. AI सेवा चयन — Google / OpenRouter / Ollama / LM Studio / vLLM
7. सुरक्षा टूल फ़िल्टर सेटिंग
8. एडमिन टोकन सेटिंग — डैशबोर्ड प्रबंधन कार्यों को लॉक करने के लिए पासवर्ड। ऑटो-जनरेट भी संभव
9. API कुंजी एन्क्रिप्शन पासवर्ड सेटिंग — कुंजियों को अधिक सुरक्षित रूप से स्टोर करने के लिए (वैकल्पिक)
10. कॉन्फ़िगरेशन फ़ाइल सेव पथ
```

> ⚠️ **एडमिन टोकन ज़रूर याद रखें।** बाद में डैशबोर्ड में कुंजियां जोड़ने या सेटिंग बदलने के लिए इसकी आवश्यकता होगी। भूल जाने पर कॉन्फ़िगरेशन फ़ाइल को मैनुअली संपादित करना होगा।

विज़ार्ड पूरा होने पर `wall-vault.yaml` कॉन्फ़िगरेशन फ़ाइल स्वचालित रूप से बनती है।

### चलाएं

```bash
wall-vault start
```

दो सर्वर एक साथ शुरू होते हैं:

- **प्रॉक्सी** (`http://localhost:56244`) — OpenClaw और AI सेवाओं को जोड़ने वाला मध्यस्थ
- **कुंजी तिजोरी** (`http://localhost:56243`) — API कुंजी प्रबंधन और वेब डैशबोर्ड

ब्राउज़र में `http://localhost:56243` खोलें और डैशबोर्ड तुरंत देखें।

## API कुंजी पंजीकरण

API कुंजी पंजीकृत करने के चार तरीके हैं। **शुरुआत करने वालों के लिए विधि 1 (एनवायरनमेंट वेरिएबल) की सिफारिश है**।

### विधि 1: एनवायरनमेंट वेरिएबल (अनुशंसित — सबसे सरल)

एनवायरनमेंट वेरिएबल **पहले से सेट किए गए मान** हैं जो प्रोग्राम शुरू होने पर पढ़ता है। टर्मिनल में नीचे दिए अनुसार टाइप करें।

```bash
# Google Gemini कुंजी पंजीकरण
export WV_KEY_GOOGLE=AIzaSy...

# OpenRouter कुंजी पंजीकरण
export WV_KEY_OPENROUTER=sk-or-v1-...

# पंजीकरण के बाद चलाएं
wall-vault start
```

अगर आपके पास कई कुंजियां हैं, तो कॉमा(,) से जोड़ें। wall-vault उन्हें बारी-बारी से स्वचालित रूप से उपयोग करेगा (round robin):

```bash
export WV_KEY_GOOGLE=AIzaSy...,AIzaSy...,AIzaSy...
```

> 💡 **टिप**: `export` कमांड केवल वर्तमान टर्मिनल सेशन पर लागू होता है। कंप्यूटर रीस्टार्ट के बाद भी बनाए रखने के लिए, ऊपर की पंक्तियों को `~/.bashrc` या `~/.zshrc` फ़ाइल में जोड़ें।

### विधि 2: डैशबोर्ड UI (माउस से क्लिक)

1. ब्राउज़र में `http://localhost:56243` खोलें
2. शीर्ष पर **🔑 API कुंजी** कार्ड में `[+ जोड़ें]` बटन क्लिक करें
3. सेवा प्रकार, कुंजी मान, लेबल (संदर्भ नाम), और दैनिक सीमा दर्ज करें, फिर सेव करें

### विधि 3: REST API (ऑटोमेशन/स्क्रिप्ट के लिए)

REST API प्रोग्रामों के बीच HTTP के माध्यम से डेटा का आदान-प्रदान करने का तरीका है। स्क्रिप्ट के माध्यम से ऑटो-रजिस्ट्रेशन के लिए उपयोगी।

```bash
curl -X POST http://localhost:56243/admin/keys \
  -H "Authorization: Bearer एडमिन_टोकन" \
  -H "Content-Type: application/json" \
  -d '{
    "service": "google",
    "key": "AIzaSy...",
    "label": "मुख्य कुंजी",
    "daily_limit": 1000
  }'
```

### विधि 4: proxy फ़्लैग (त्वरित परीक्षण के लिए)

बिना औपचारिक पंजीकरण के अस्थायी रूप से कुंजी डालकर परीक्षण करने के लिए। प्रोग्राम बंद करने पर गायब हो जाती है।

```bash
wall-vault proxy --key-google=AIzaSy... --key-openrouter=sk-or-...
```

## प्रॉक्सी का उपयोग

### OpenClaw के साथ उपयोग (मुख्य उद्देश्य)

OpenClaw को wall-vault के माध्यम से AI सेवाओं से कनेक्ट करने का तरीका यहां बताया गया है।

`~/.openclaw/openclaw.json` फ़ाइल खोलें और नीचे दी गई सामग्री जोड़ें:

```json5
// ~/.openclaw/openclaw.json
{
  models: {
    providers: {
      "wall-vault": {
        baseUrl: "http://localhost:56244/v1",
        apiKey: "your-agent-token",   // vault एजेंट टोकन
        api: "openai-completions",
        models: [
          { id: "wall-vault/gemini-2.5-flash" },
          { id: "wall-vault/gemini-2.5-pro" },
          { id: "wall-vault/hunter-alpha" },    // मुफ़्त 1M context
          { id: "wall-vault/claude-opus-4-6" }
        ]
      }
    }
  }
}
```

> 💡 **आसान तरीका**: डैशबोर्ड में एजेंट कार्ड पर **🦞 OpenClaw कॉन्फ़िग कॉपी** बटन दबाएं। टोकन और एड्रेस पहले से भरा हुआ स्निपेट क्लिपबोर्ड पर कॉपी हो जाएगा। बस पेस्ट करें।

**मॉडल नाम के पहले `wall-vault/` कहां पॉइंट करता है?**

wall-vault मॉडल नाम के आधार पर स्वचालित रूप से तय करता है कि किस AI सेवा को रिक्वेस्ट भेजना है:

| मॉडल फ़ॉर्मैट | कनेक्ट होने वाली सेवा |
|----------|--------------|
| `wall-vault/gemini-*` | Google Gemini से सीधा कनेक्शन |
| `wall-vault/gpt-*`, `wall-vault/o3`, `wall-vault/o4*` | OpenAI से सीधा कनेक्शन |
| `wall-vault/claude-*` | OpenRouter के माध्यम से Anthropic कनेक्शन |
| `wall-vault/hunter-alpha`, `wall-vault/healer-alpha` | OpenRouter (10 लाख टोकन context मुफ़्त) |
| `wall-vault/kimi-*`, `wall-vault/glm-*`, `wall-vault/deepseek-*` | OpenRouter कनेक्शन |
| `google/मॉडल_नाम`, `openai/मॉडल_नाम`, `anthropic/मॉडल_नाम` आदि | संबंधित सेवा से सीधा कनेक्शन |
| `custom/google/मॉडल_नाम`, `custom/openai/मॉडल_नाम` आदि | `custom/` भाग हटाकर री-रूट |
| `मॉडल_नाम:cloud` | `:cloud` भाग हटाकर OpenRouter कनेक्शन |

> 💡 **Context क्या है?** AI एक बार में कितनी बातचीत याद रख सकता है, उसकी मात्रा। 1M (दस लाख टोकन) का मतलब है बहुत लंबी बातचीत या बड़े दस्तावेज़ एक बार में प्रोसेस किए जा सकते हैं।

### Gemini API फ़ॉर्मैट में सीधा कनेक्शन (मौजूदा टूल्स के साथ संगतता)

अगर आपके पास ऐसे टूल हैं जो Google Gemini API सीधे उपयोग करते हैं, बस एड्रेस wall-vault में बदलें:

```bash
export ANTHROPIC_BASE_URL=http://localhost:56244/google
```

या अगर टूल सीधे URL निर्दिष्ट करने की अनुमति देता है:

```
http://localhost:56244/google/v1beta/models/gemini-2.5-flash:generateContent
```

### OpenAI SDK (Python) में उपयोग

Python में AI उपयोग करने वाले कोड में भी wall-vault कनेक्ट किया जा सकता है। बस `base_url` बदलें:

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:56244/v1",
    api_key="not-needed"  # API कुंजियां wall-vault प्रबंधित करता है
)

response = client.chat.completions.create(
    model="google/gemini-2.5-flash",   # provider/model फ़ॉर्मैट
    messages=[{"role": "user", "content": "नमस्ते"}]
)
```

### चलते समय मॉडल बदलना

wall-vault चल रहा होने पर AI मॉडल बदलने के लिए:

```bash
# प्रॉक्सी पर सीधे रिक्वेस्ट करके मॉडल बदलें
curl -X PUT http://localhost:56244/api/config/model \
  -H "Content-Type: application/json" \
  -d '{"service": "openrouter", "model": "anthropic/claude-3.5-sonnet"}'

# वितरित मोड (मल्टी-बॉट) में, तिजोरी सर्वर से बदलें → SSE के माध्यम से तुरंत प्रतिबिंबित
curl -X PUT http://localhost:56243/admin/clients/मेरा-बॉट-id \
  -H "Authorization: Bearer एडमिन_टोकन" \
  -H "Content-Type: application/json" \
  -d '{"default_service": "google", "default_model": "gemini-2.5-pro"}'
```

- **कार्ड ग्रिड**: एजेंट, सेवाएं और API कुंजी कार्ड टाइल फ़ॉर्मैट में व्यवस्थित

### API कुंजी कार्ड

पंजीकृत API कुंजियों को एक नज़र में प्रबंधित करने का कार्ड।

- सेवा के अनुसार कुंजी सूची दिखाता है।
- `today_usage`: आज सफलतापूर्वक प्रोसेस किए गए टोकन (AI द्वारा पढ़े और लिखे गए अक्षर) की संख्या
- `today_attempts`: आज कुल कॉल संख्या (सफल + विफल सहित)
- `[+ जोड़ें]` बटन से नई कुंजियां पंजीकृत करें, `✕` से हटाएं।

> 💡 **टोकन क्या है?** AI जब टेक्स्ट प्रोसेस करता है तब उपयोग की जाने वाली इकाई। लगभग अंग्रेज़ी का एक शब्द, या कोरियन के 1-2 अक्षर के बराबर। API शुल्क आमतौर पर इस टोकन संख्या के आधार पर गणना होती है।

### एजेंट कार्ड

wall-vault प्रॉक्सी से जुड़े बॉट (एजेंट) की स्थिति दिखाने वाला कार्ड।

**कनेक्शन स्थिति 4 स्तरों में प्रदर्शित होती है:**

| संकेतक | स्थिति | अर्थ |
|------|------|------|
| 🟢 | चल रहा | प्रॉक्सी सामान्य रूप से काम कर रहा है |
| 🟡 | विलंबित | जवाब आ रहा है लेकिन धीमा |
| 🔴 | ऑफ़लाइन | प्रॉक्सी जवाब नहीं दे रहा |
| ⚫ | असंबद्ध/निष्क्रिय | प्रॉक्सी तिजोरी से कभी नहीं जुड़ा या निष्क्रिय |

**एजेंट कार्ड के निचले बटन गाइड:**

एजेंट रजिस्टर करते समय **एजेंट प्रकार** निर्दिष्ट करें, और उस प्रकार के अनुसार सुविधा बटन स्वचालित रूप से दिखाई देंगे।

बटन क्लिक करने पर, एजेंट का टोकन, प्रॉक्सी एड्रेस और मॉडल जानकारी पहले से भरी हुई कॉन्फ़िग स्निपेट क्लिपबोर्ड पर कॉपी होती है। कॉपी की गई सामग्री नीचे तालिका में दिए गए स्थान पर पेस्ट करने से कनेक्शन सेटअप पूरा होता है।

| बटन | एजेंट प्रकार | पेस्ट करने का स्थान |
|------|-------------|-------------|
| 🦞 OpenClaw कॉन्फ़िग कॉपी | `openclaw` | `~/.openclaw/openclaw.json` |
| 🦀 NanoClaw कॉन्फ़िग कॉपी | `nanoclaw` | `~/.openclaw/openclaw.json` |
| 🟠 Claude Code कॉन्फ़िग कॉपी | `claude-code` | `~/.claude/settings.json` |

**उदाहरण — Claude Code प्रकार के लिए यह सामग्री कॉपी होगी:**


> ⚠️ **Continue का नवीनतम संस्करण `config.yaml` उपयोग करता है।** यदि `config.yaml` मौजूद है, तो `config.json` पूरी तरह से अनदेखा किया जाता है। `config.yaml` में ही पेस्ट करें।

### VS Code में Cline उपयोग — विस्तृत गाइड

#### चरण 1: Cline इंस्टॉल करें

VS Code एक्सटेंशन मार्केटप्लेस से **Cline** (ID: `saoudrizwan.claude-dev`) इंस्टॉल करें।

#### चरण 2: तिजोरी में एजेंट रजिस्टर करें

1. तिजोरी डैशबोर्ड (`http://तिजोरी_IP:56243`) खोलें
2. **एजेंट** सेक्शन में **+ जोड़ें** क्लिक करें
3. निम्नानुसार भरें:

| फ़ील्ड | मान | विवरण |
|------|----|------|
| ID | `my_cline` | अद्वितीय पहचानकर्ता (अंग्रेज़ी, बिना स्पेस) |
| नाम | `My Cline` | डैशबोर्ड पर दिखने वाला नाम |
| एजेंट प्रकार | `cline` | ← अवश्य `cline` चुनें |
| सेवा | वांछित सेवा चुनें (उदा: `google`) | |
| मॉडल | वांछित मॉडल दर्ज करें (उदा: `gemini-2.5-flash`) | |

4. **सेव** पर क्लिक करने पर टोकन स्वचालित रूप से बनता है

#### चरण 3: Cline से कनेक्ट करें

**विधि A — ऑटो-अप्लाई (अनुशंसित)**

1. उस मशीन पर wall-vault **प्रॉक्सी** चल रहा है, यह सुनिश्चित करें (`localhost:56244`)
2. डैशबोर्ड के एजेंट कार्ड में **⚡ Cline सेटिंग अप्लाई** बटन क्लिक करें
3. "सेटिंग अप्लाई सफल!" सूचना आने पर सफल
4. VS Code रीलोड करें (`Ctrl+Alt+R`)

**विधि B — मैनुअल सेटअप**

Cline साइडबार में सेटिंग्स (⚙️) खोलें:
- **API Provider**: `OpenAI Compatible`
- **Base URL**: `http://प्रॉक्सी_एड्रेस:56244/v1`
  - यदि एक ही मशीन: `http://localhost:56244/v1`
  - यदि दूसरी मशीन (जैसे Mini सर्वर): `http://192.168.1.20:56244/v1`
- **API Key**: तिजोरी से जारी टोकन (एजेंट कार्ड से कॉपी)
- **Model ID**: तिजोरी में सेट मॉडल (उदा: `gemini-2.5-flash`)

#### चरण 4: सत्यापन

Cline चैट में कोई भी मैसेज भेजें। सामान्य होने पर:
- तिजोरी डैशबोर्ड में उस एजेंट कार्ड पर **हरा बिंदु (● चल रहा)** दिखेगा
- कार्ड पर वर्तमान सेवा/मॉडल दिखेगा (उदा: `google / gemini-2.5-flash`)

#### मॉडल बदलना

Cline का मॉडल बदलना चाहते हैं तो **तिजोरी डैशबोर्ड** से बदलें:

1. एजेंट कार्ड का सेवा/मॉडल ड्रॉपडाउन बदलें
2. **अप्लाई** क्लिक करें
3. VS Code रीलोड करें (`Ctrl+Alt+R`) — Cline फ़ुटर का मॉडल नाम अपडेट होगा
4. अगले रिक्वेस्ट से नया मॉडल उपयोग होगा

> 💡 वास्तव में, प्रॉक्सी Cline के रिक्वेस्ट को टोकन से पहचानता है और तिजोरी सेटिंग के मॉडल पर रूट करता है। VS Code रीलोड न करने पर भी **वास्तव में उपयोग होने वाला मॉडल तुरंत बदल जाता है** — रीलोड केवल Cline UI में मॉडल डिस्प्ले अपडेट करने के लिए है।

| Cline में "कनेक्शन विफल" त्रुटि | प्रॉक्सी नहीं चल रहा या एड्रेस गलत | `curl http://localhost:56244/health` से प्रॉक्सी जांचें |
| तिजोरी में हरा बिंदु नहीं दिखता | API कुंजी (टोकन) सेट नहीं | **⚡ Cline सेटिंग अप्लाई** बटन फिर से क्लिक करें |
| Cline फ़ुटर मॉडल नहीं बदलता | Cline सेटिंग्स कैश करता है | VS Code रीलोड (`Ctrl+Alt+R`) |
| गलत मॉडल नाम दिखता है | पुराना बग (v0.1.16 में ठीक किया गया) | प्रॉक्सी को v0.1.16 या ऊपर अपडेट करें |

#### 🟣 डिप्लॉय कमांड कॉपी बटन — नई मशीन पर इंस्टॉल करने के लिए

नए कंप्यूटर पर wall-vault प्रॉक्सी पहली बार इंस्टॉल करने और तिजोरी से कनेक्ट करने के लिए। बटन क्लिक करने पर पूरी इंस्टॉलेशन स्क्रिप्ट कॉपी होती है। नए कंप्यूटर के टर्मिनल में पेस्ट करके चलाने पर यह एक बार में हो जाता है:

1. wall-vault बाइनरी इंस्टॉलेशन (पहले से इंस्टॉल होने पर छोड़ देता है)
2. systemd यूज़र सर्विस ऑटो-रजिस्ट्रेशन
3. सेवा शुरू और तिजोरी से ऑटो-कनेक्ट

> 💡 स्क्रिप्ट में इस एजेंट का टोकन और तिजोरी सर्वर एड्रेस पहले से भरा होता है, इसलिए पेस्ट के बाद बिना किसी संशोधन के सीधे चलाया जा सकता है।

### सेवा कार्ड

उपयोग करने योग्य AI सेवाओं को चालू/बंद करने या कॉन्फ़िगर करने का कार्ड।

- प्रत्येक सेवा के लिए सक्रिय/निष्क्रिय टॉगल स्विच
- लोकल AI सर्वर (आपके कंप्यूटर पर चलने वाले Ollama, LM Studio, vLLM आदि) का एड्रेस दर्ज करें और उपलब्ध मॉडल स्वचालित रूप से पहचाने जाएंगे।
- **लोकल सेवा कनेक्शन स्थिति संकेतक**: सेवा नाम के बगल में ● बिंदु **हरा** होने पर कनेक्ट, **ग्रे** होने पर डिस्कनेक्ट
- **लोकल सेवा ऑटो ट्रैफ़िक लाइट** (v0.1.23+): लोकल सेवाएं (Ollama, LM Studio, vLLM) कनेक्ट होने पर स्वचालित सक्रिय और डिस्कनेक्ट होने पर स्वचालित निष्क्रिय होती हैं। सेवा चालू करने पर 15 सेकंड में ● हरा होता है और चेकबॉक्स ऑन होता है; बंद करने पर स्वचालित ऑफ। क्लाउड सेवाओं (Google, OpenRouter आदि) के API कुंजी-आधारित ऑटो टॉगल जैसा ही काम करता है।

> 💡 **यदि लोकल सेवा दूसरे कंप्यूटर पर चल रही है**: सेवा URL फ़ील्ड में उस कंप्यूटर का IP दर्ज करें। उदा: `http://192.168.1.20:11434` (Ollama), `http://192.168.1.20:1234` (LM Studio)। यदि सेवा `0.0.0.0` के बजाय केवल `127.0.0.1` पर बाइंड है, तो बाहरी IP से एक्सेस नहीं होगा, सेवा सेटिंग्स में बाइंडिंग एड्रेस जांचें।

### एडमिन टोकन इनपुट

डैशबोर्ड में कुंजी जोड़ने/हटाने जैसे महत्वपूर्ण कार्य करने पर एडमिन टोकन इनपुट पॉपअप दिखता है। setup विज़ार्ड में सेट किया गया टोकन दर्ज करें। एक बार दर्ज करने पर ब्राउज़र बंद होने तक मान्य रहता है।

> ⚠️ **15 मिनट में 10 से अधिक प्रमाणीकरण विफलताओं पर वह IP अस्थायी रूप से ब्लॉक हो जाता है।** यदि टोकन भूल गए हैं, तो `wall-vault.yaml` फ़ाइल में `admin_token` आइटम जांचें।

## वितरित मोड (मल्टी-बॉट)

कई कंप्यूटरों पर एक साथ OpenClaw चलाते समय, **एक कुंजी तिजोरी साझा** करने का कॉन्फ़िगरेशन। कुंजी प्रबंधन एक ही स्थान से करने की सुविधा।

### कॉन्फ़िगरेशन उदाहरण

```
[कुंजी तिजोरी सर्वर]
  wall-vault vault    (कुंजी तिजोरी :56243, डैशबोर्ड)

[WSL अल्फ़ा]            [Raspberry Pi गामा]    [Mac Mini लोकल]
  wall-vault proxy      wall-vault proxy        wall-vault proxy
  openclaw TUI          openclaw TUI            openclaw TUI
  ↕ SSE सिंक             ↕ SSE सिंक              ↕ SSE सिंक
```

सभी बॉट केंद्रीय तिजोरी सर्वर की ओर देखते हैं, इसलिए तिजोरी में मॉडल बदलने या कुंजी जोड़ने पर सभी बॉट पर तुरंत प्रतिबिंबित होता है।

### चरण 1: कुंजी तिजोरी सर्वर शुरू करें

तिजोरी सर्वर के रूप में उपयोग होने वाले कंप्यूटर पर चलाएं:

```bash
wall-vault vault
```

### चरण 2: प्रत्येक बॉट (क्लाइंट) रजिस्टर करें

तिजोरी सर्वर से कनेक्ट होने वाले प्रत्येक बॉट की जानकारी पहले से रजिस्टर करें:

```bash
curl -X POST http://localhost:56243/admin/clients \
  -H "Authorization: Bearer एडमिन_टोकन" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "botA",
    "name": "बॉटA",
    "token": "bota-secret",
    "default_service": "google",
    "default_model": "gemini-2.5-flash"
  }'
```

### चरण 3: प्रत्येक बॉट कंप्यूटर पर प्रॉक्सी शुरू करें

प्रत्येक बॉट इंस्टॉल कंप्यूटर पर तिजोरी सर्वर एड्रेस और टोकन निर्दिष्ट करके प्रॉक्सी चलाएं:

```bash
WV_VAULT_URL=http://192.168.x.x:56243 \
WV_VAULT_TOKEN=bota-secret \
WV_VAULT_CLIENT_ID=botA \
wall-vault proxy
```

> 💡 **`192.168.x.x`** को तिजोरी सर्वर कंप्यूटर के वास्तविक आंतरिक IP एड्रेस से बदलें। राउटर सेटिंग्स या `ip addr` कमांड से जांच सकते हैं।

## स्वचालित स्टार्ट सेटअप

कंप्यूटर रीस्टार्ट होने पर हर बार मैनुअली wall-vault शुरू करना असुविधाजनक लगे तो, सिस्टम सेवा के रूप में रजिस्टर करें। एक बार रजिस्टर करने पर बूट पर स्वचालित शुरू होगा।

### Linux — systemd (अधिकांश Linux)

systemd Linux में प्रोग्रामों को स्वचालित रूप से शुरू और प्रबंधित करने की प्रणाली है:

```bash
wall-vault doctor deploy
systemctl --user daemon-reload
systemctl --user enable --now wall-vault
```

लॉग देखें:

```bash
journalctl --user -u wall-vault -f
```

### macOS — launchd

macOS में प्रोग्रामों के स्वचालित निष्पादन की ज़िम्मेदार प्रणाली:

```bash
wall-vault doctor deploy launchd
launchctl load ~/Library/LaunchAgents/com.wall-vault.plist
```

### Windows — NSSM

1. [nssm.cc](https://nssm.cc/download) से NSSM डाउनलोड करें और PATH में जोड़ें।
2. एडमिनिस्ट्रेटर PowerShell में:

```powershell
wall-vault doctor deploy windows
```

## Doctor (डॉक्टर)

`doctor` कमांड wall-vault सही ढंग से सेट है या नहीं, यह **स्वयं निदान और मरम्मत करने वाला टूल** है।

```bash
wall-vault doctor check   # वर्तमान स्थिति निदान (केवल पढ़ना, कुछ नहीं बदलता)
wall-vault doctor fix     # समस्याओं को स्वचालित मरम्मत
wall-vault doctor all     # निदान + स्वचालित मरम्मत एक बार में
```

> 💡 कुछ गलत लगे तो पहले `wall-vault doctor all` चलाएं। कई समस्याएं स्वचालित रूप से ठीक हो जाती हैं।

## RTK टोकन बचत

*(v0.1.24+)*

**RTK (टोकन बचत टूल)** AI कोडिंग एजेंट (जैसे Claude Code) द्वारा चलाए गए शेल कमांड के आउटपुट को स्वचालित रूप से संपीड़ित करता है, जिससे टोकन उपयोग कम होता है। उदाहरण के लिए, `git status` का 15-पंक्ति आउटपुट 2-पंक्ति सारांश में सिमट जाता है।

### बुनियादी उपयोग

```bash
# कमांड को wall-vault rtk से लपेटें और आउटपुट स्वचालित फ़िल्टर होगा
wall-vault rtk git status          # केवल बदली हुई फ़ाइलों की सूची दिखाता है
wall-vault rtk git diff HEAD~1     # बदली हुई पंक्तियां + न्यूनतम context
wall-vault rtk git log -10         # प्रति commit एक पंक्ति: हैश + मैसेज
wall-vault rtk go test ./...       # केवल विफल टेस्ट दिखाता है
wall-vault rtk ls -la              # असमर्थित कमांड स्वचालित कटे
```

### समर्थित कमांड और बचत

| कमांड | फ़िल्टर विधि | बचत |
|------|----------|--------|
| `git status` | केवल बदली फ़ाइलों का सारांश | ~87% |
| `git diff` | बदली पंक्तियां + 3 पंक्तियां context | ~60-94% |
| `git log` | हैश + पहली पंक्ति मैसेज | ~90% |
| `git push/pull/fetch` | प्रगति हटाई, केवल सारांश | ~80% |
| `go test` | केवल विफल दिखाए, पास गिने | ~88-99% |
| `go build/vet` | केवल एरर दिखाए | ~90% |
| अन्य सभी कमांड | पहली 50 + अंतिम 50 पंक्तियां, अधिकतम 32KB | परिवर्तनशील |

### 3-चरण फ़िल्टर पाइपलाइन

1. **कमांड-विशिष्ट स्ट्रक्चर फ़िल्टर** — git, go आदि के आउटपुट फ़ॉर्मैट को समझता है और केवल सार्थक भाग निकालता है
2. **रेगेक्स पोस्ट-प्रोसेसिंग** — ANSI रंग कोड हटाना, खाली पंक्तियां सिकोड़ना, डुप्लिकेट पंक्तियां एकत्रित करना
3. **पासथ्रू + कटिंग** — असमर्थित कमांड केवल पहली/अंतिम 50 पंक्तियां रखते हैं

### Claude Code इंटीग्रेशन

Claude Code के `PreToolUse` हुक से सभी शेल कमांड स्वचालित रूप से RTK से गुज़रने के लिए सेट किया जा सकता है।

```bash
# हुक इंस्टॉल (Claude Code settings.json में स्वचालित जुड़ता है)
wall-vault rtk hook install
```

या मैनुअली `~/.claude/settings.json` में जोड़ें:

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "command": "wall-vault rtk rewrite"
    }]
  }
}
```

> 💡 **Exit code संरक्षण**: RTK मूल कमांड का exit code वैसा ही लौटाता है। कमांड विफल होने पर (exit code ≠ 0) AI भी विफलता सटीक रूप से पहचानता है।

> 💡 **अंग्रेज़ी आउटपुट अनिवार्य**: RTK `LC_ALL=C` के साथ कमांड चलाता है ताकि सिस्टम भाषा सेटिंग से स्वतंत्र हमेशा अंग्रेज़ी आउटपुट बने। इससे फ़िल्टर सही ढंग से काम करते हैं।

| `WV_THEME` | डैशबोर्ड थीम | `light`, `dark`, `gold` |
| `WV_KEY_GOOGLE` | Google API कुंजी (कॉमा से कई) | `AIza...,AIza...` |
| `WV_KEY_OPENROUTER` | OpenRouter API कुंजी | `sk-or-v1-...` |
| `WV_VAULT_URL` | वितरित मोड में तिजोरी सर्वर एड्रेस | `http://192.168.x.x:56243` |
| `WV_VAULT_TOKEN` | क्लाइंट (बॉट) प्रमाणीकरण टोकन | `my-secret-token` |
| `WV_ADMIN_TOKEN` | एडमिन टोकन | `admin-token-here` |
| `WV_MASTER_PASS` | API कुंजी एन्क्रिप्शन पासवर्ड | `my-password` |
| `WV_AVATAR` | अवतार इमेज फ़ाइल पथ (`~/.openclaw/` से सापेक्ष) | `workspace/avatars/avatar.png` |
| `OLLAMA_URL` | Ollama लोकल सर्वर एड्रेस | `http://192.168.x.x:11434` |

## समस्या निवारण

### जब प्रॉक्सी शुरू नहीं होता

अक्सर पोर्ट पहले से किसी अन्य प्रोग्राम द्वारा उपयोग में होता है।

```bash
ss -tlnp | grep 56244   # पोर्ट 56244 कौन उपयोग कर रहा है जांचें
wall-vault proxy --port 8080   # दूसरे पोर्ट नंबर से शुरू करें
```

### API कुंजी त्रुटियां (429, 402, 401, 403, 582)

| एरर कोड | अर्थ | उपाय |
|----------|------|----------|
| **429** | बहुत अधिक रिक्वेस्ट (उपयोग सीमा पार) | थोड़ा रुकें या दूसरी कुंजी जोड़ें |
| **402** | भुगतान आवश्यक या क्रेडिट अपर्याप्त | संबंधित सेवा में क्रेडिट रिचार्ज करें |
| **401 / 403** | कुंजी गलत या अनुमति नहीं | कुंजी मान पुनः जांचें और फिर से रजिस्टर करें |
| **582** | गेटवे ओवरलोड (5 मिनट cooldown) | 5 मिनट बाद स्वचालित रिलीज़ |

```bash
# पंजीकृत कुंजी सूची और स्थिति जांचें
curl -H "Authorization: Bearer एडमिन_टोकन" http://localhost:56243/admin/keys

# कुंजी उपयोग काउंटर रीसेट
curl -X POST -H "Authorization: Bearer एडमिन_टोकन" http://localhost:56243/admin/keys/reset
```

### जब एजेंट "असंबद्ध" दिखाता है

"असंबद्ध" का मतलब है कि प्रॉक्सी प्रक्रिया तिजोरी को सिग्नल (heartbeat) नहीं भेज रही। **इसका मतलब सेटिंग्स सेव नहीं हुईं, ऐसा नहीं है।** प्रॉक्सी को तिजोरी सर्वर एड्रेस और टोकन के साथ चल रहा होना चाहिए ताकि कनेक्शन स्थिति बदले।

```bash
# तिजोरी सर्वर एड्रेस, टोकन और क्लाइंट ID निर्दिष्ट करके प्रॉक्सी शुरू करें
WV_VAULT_URL=http://तिजोरी_सर्वर_एड्रेस:56243 \
WV_VAULT_TOKEN=क्लाइंट_टोकन \
WV_VAULT_CLIENT_ID=क्लाइंट_ID \
wall-vault proxy
```

कनेक्शन सफल होने पर लगभग 20 सेकंड में डैशबोर्ड में 🟢 चल रहा दिखेगा।

### जब Ollama कनेक्शन नहीं होता

Ollama आपके कंप्यूटर पर सीधे AI चलाने वाला प्रोग्राम है। पहले जांचें कि Ollama चालू है।

```bash
curl http://localhost:11434/api/tags   # मॉडल सूची दिखे तो सामान्य
export OLLAMA_URL=http://192.168.x.x:11434   # दूसरे कंप्यूटर पर चल रहा हो तो
```

> ⚠️ Ollama जवाब न दे तो पहले `ollama serve` कमांड से Ollama शुरू करें।

> ⚠️ **बड़े मॉडल धीमे होते हैं**: `qwen3.5:35b`, `deepseek-r1` जैसे बड़े मॉडल में जवाब बनने में कई मिनट लग सकते हैं। भले ही जवाब न दिखे, सामान्य प्रोसेसिंग हो रही हो सकती है, कृपया प्रतीक्षा करें।

## हाल के बदलाव (v0.1.16 ~ v0.1.25)

### v0.1.25 (2026-04-08)
- **एजेंट प्रक्रिया पहचान**: प्रॉक्सी लोकल एजेंट (NanoClaw/OpenClaw) जीवित है या नहीं पहचानता है और डैशबोर्ड में नारंगी ट्रैफ़िक लाइट दिखाता है।
- **ड्रैग हैंडल सुधार**: कार्ड सॉर्ट करते समय केवल ट्रैफ़िक लाइट (●) क्षेत्र से ही पकड़ सकते हैं। इनपुट फ़ील्ड या बटन से गलती से ड्रैग नहीं होता।

### v0.1.24 (2026-04-06)
- **RTK टोकन बचत सबकमांड**: `wall-vault rtk <command>` शेल कमांड आउटपुट को स्वचालित फ़िल्टर करता है, AI एजेंट का टोकन उपयोग 60-90% कम करता है। git, go जैसे मुख्य कमांड के लिए समर्पित फ़िल्टर, असमर्थित कमांड स्वचालित कटते हैं। Claude Code `PreToolUse` हुक से पारदर्शी इंटीग्रेशन।

### v0.1.23 (2026-04-06)
- **Ollama मॉडल बदलाव सुधार**: तिजोरी डैशबोर्ड में Ollama मॉडल बदलने पर वास्तविक प्रॉक्सी में प्रतिबिंबित न होने की समस्या ठीक। पहले केवल एनवायरनमेंट वेरिएबल (`OLLAMA_MODEL`) उपयोग होता था, अब तिजोरी सेटिंग प्राथमिकता।
- **लोकल सेवा ऑटो ट्रैफ़िक लाइट**: Ollama/LM Studio/vLLM कनेक्ट होने पर स्वचालित सक्रिय, डिस्कनेक्ट होने पर स्वचालित निष्क्रिय। क्लाउड सेवाओं के कुंजी-आधारित ऑटो टॉगल जैसा।

### v0.1.22 (2026-04-05)
- **खाली content फ़ील्ड गायब सुधार**: thinking मॉडल (gemini-3.1-pro, o1, claude thinking आदि) जब max_tokens सीमा reasoning में खर्च कर वास्तविक जवाब न बना पाएं, प्रॉक्सी `omitempty` से जवाब JSON के `content`/`text` फ़ील्ड गायब करता था, जिससे OpenAI/Anthropic SDK क्लाइंट `Cannot read properties of undefined (reading 'trim')` एरर से क्रैश होते थे। आधिकारिक API स्पेक अनुसार फ़ील्ड हमेशा शामिल करने के लिए सुधार।
